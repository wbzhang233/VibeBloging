# Claude Code 精密拆解系列 — 前言与阅读指南

> 本系列基于 Claude Code 开源/逆向注释仓库（zhangbo2008/claude_code_annotated、oboard/claude-code-rev、shareAI-lab/learn-claude-code）进行深度研究，力求还原这台"AI 编程机器"的真实运作方式。

---

## 这个系列是什么

Claude Code 是 Anthropic 推出的 AI 编程 CLI 工具，表面上看是一个"在终端里问问题"的工具，但它的内部是一套精心设计的 **Harness 工程体系**——将 Claude API 的推理能力转化为对真实世界（文件系统、Shell、IDE、MCP服务）的操作能力。

本系列从源码层面拆解这套体系，共 **9 篇研究笔记 + 8 张交互可视化图**。

目标：让读者不仅知道 Claude Code"能做什么"，更理解它"为什么这么做"——每一个设计决策背后的工程权衡。

---

## 系列结构

本系列分为两个互补的部分：

### `notes/` — 研究笔记（文字版）

源码级深度分析，适合细读。每篇笔记包含：关键源码片段、数据结构定义、执行流程伪代码、设计决策分析。

### `figures/` — 交互可视化（图像版）

与笔记对应的 HTML 交互图，适合快速建立宏观认知。用浏览器打开即可，无需任何依赖。详见下方"可视化图表指南"。

### 图表清单与阅读要点

#### 00b — 启动与初始化序列

**这张图展示什么：** `claude` 命令执行到进入 QueryLoop 之间的 7 个启动阶段。

**阅读要点：**

- 图中每个 Phase 卡片可点击展开——先浏览 Phase 概览条，再逐一展开感兴趣的阶段
- 重点关注 **Phase 2（CLAUDE.md 链式加载）**：这是 Claude Code 能"感知项目"的关键
- **Phase 3 的 Skill 懒加载**：解释了为什么启动很快——内容不是一次性全部加载的
- **Phase 6 的 Prompt Cache 预热**：理解为什么第二次运行比第一次便宜

#### 00 — 机器全景图

**这张图展示什么：** Claude Code 的完整四层架构鸟瞰图。

**阅读要点：**

- 从外到内：L4 扩展层 → L3 控制层 → L2 上下文层 → L1 执行层
- 注意每层的职责边界：**L1 是唯一真正执行的层**，其余三层都是"规则与约束"
- Claude API 处于图中心，但注意它是**无状态的**——所有状态由 Harness（外壳）持有

#### 01 — 请求生命周期

**这张图展示什么：** 从用户按下回车到 Claude 响应，中间发生的完整垂直流程。

**阅读要点：**

- 注意流程中的**三路并行**：systemPrompt、userContext、systemContext 同时获取
- QueryLoop 是核心循环：`API调用 → 工具执行 → 再调用`，直到 `stop_reason = end_turn`
- 找到"上下文压缩触发点"——它不是主动发生的，而是检查到 70% 使用量才触发

#### 02 — 权限决策引擎

**这张图展示什么：** 每次工具调用前，Claude Code 如何决定"允许还是拒绝"。

**阅读要点：**

- 这是 **5 种外部权限模式 + 2 种内部模式**的全貌图
- 重点看决策树的优先级：Deny 规则 → Allow 规则 → 用户确认
- 注意 `bypassPermissions` 模式的位置——它是最高权限，跳过所有检查
- `plan` 模式（只读）和 `default` 模式的分支有明显区别

#### 03 — 工具执行引擎

**这张图展示什么：** 45+ 工具如何被并发/串行执行，以及并发安全边界如何划定。

**阅读要点：**

- 看"并发安全"与"并发不安全"工具的分组——只读操作可以并行，写操作必须串行
- `TrackedTool` 数据结构是核心：id + isConcurrencySafe + AbortController
- 找到工具输出持久化的触发点（50KB / 30KB 阈值）

#### 04 — Coordinator/Worker 架构

**这张图展示什么：** 多 Agent 协作模式——一个 Coordinator 如何调度多个 Worker 并行完成复杂任务。

**阅读要点：**

- 注意图中**两种模式的对比**：单 Agent 模式 vs Coordinator/Worker 模式
- Worker 之间通过 `<task-notification>` XML 接收任务，这是隔离边界
- Coordinator 拥有 600+ 行专用系统提示——这是它"知道如何分配任务"的关键

#### 05 — 上下文压缩系统

**这张图展示什么：** 在 200K Token 窗口内，Claude Code 如何用 6 种策略压缩/管理上下文。

**阅读要点：**

- 6 种策略按"触发时机"排列：工具执行时 → 响应接收时 → 上下文满时
- 重点理解 `persistedOutput`（磁盘落地）和 `autoCompact`（AI 摘要）的触发条件差异
- 注意 70% 使用量才触发 autoCompact——留有缓冲而不是到 100% 才压缩

#### 06 — 记忆与提示组装

**这张图展示什么：** 每次 API 调用前，最终发出的 Prompt 是如何从多个来源拼装而成的。

**阅读要点：**

- 找到四层记忆体系的位置：CLAUDE.md（固定）→ 对话历史（动态）→ Memory 文件（跨会话）→ Prompt Cache（缓存）
- 重点关注"Prompt Cache"图层——命中率 40-95%，节省 40-60% Token，这是成本优化的核心
- MEMORY.md 的 200 行 / 25KB 截断是硬限制，图中有体现

#### 07 — Hooks 与遥测远控

**这张图展示什么：** 两套扩展机制——Hooks（事件驱动的 Shell 脚本扩展）和 MCP（外部能力协议接入）。

**阅读要点：**

- Hooks 部分：注意 `PreToolUse` 是唯一能**阻止**工具执行的 Hook（exit 1 返回 block JSON）
- MCP 部分：注意 6 种传输协议（stdio/SSE/HTTP/WebSocket 等）对应不同的部署场景
- 找到 Hooks 和 MCP 的本质区别：Hooks 是**事件响应**（被动），MCP 是**能力注入**（主动）

### 图表 vs. 笔记

| 图表 | 笔记 |
| ---- | ---- |
| 建立空间感、理解组件关系 | 理解细节、读懂设计决策 |

| 抽象化处理，突出核心流程 | 包含源码片段和精确数据 |

推荐使用方式：先用图建立认知框架（10分钟），再用对应笔记填充细节（30–60分钟/章）。
