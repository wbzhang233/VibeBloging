配置加载 — 三层覆盖模型
settings.json
▾
优先级（低 → 高）
~/.claude/settings.json        ← 全局配置，所有项目共享
    ↑ 被覆盖
.claude/settings.local.json    ← 项目本地，不提交 git
    ↑ 被覆盖
CLI flags + 环境变量            ← 本次运行，最高优先级
检测关键维度
维度检测方式影响
交互模式process.stdin.isTTY是否显示 TUI / 等待输入

settings.json 核心字段
interface ClaudeSettings {
  model?:       string          // 默认模型
  permissions:  {
    allow: string[]           // 工具允许规则（glob）
    deny:  string[]           // 工具拒绝规则（glob）
  }
  hooks?:       Record<string, HookConfig[]>
  mcpServers?:  Record<string, MCPServerConfig>
  env?:         Record<string, string>  // 注入子进程的环境变量
}

2
项目上下文发现
CLAUDE.md + Memory
▾
Git Root 检测
git rev-parse --show-toplevel
  ├── 成功 → 使用 git root 作为有效 cwd  ← 项目感知上下文
  └── 失败 → 使用当前目录
CLAUDE.md 链式加载（逐级向上）
/project/src/components/   CLAUDE.md?  → 加载，继续向上
/project/src/              CLAUDE.md?  → 加载，继续向上
/project/                  CLAUDE.md?  → 加载（项目根）
~/.claude/                 CLAUDE.md   → 加载（全局 fallback）

子目录规则优先，内容按"子 → 父"顺序合并
@file 引用在加载时递归展开
Memory 文件加载
  • ~/.claude/memory/MEMORY.md（全局）
  • .claude/memory/MEMORY.md（项目级）
  • 嵌套 @attachment 递归加载
  • 硬限制：≤ 200行 / 25KB
@file 内联导入
  • CLAUDE.md 可引用任意文件
  • 启动时递归展开
  • 支持相对路径
  • 常用于拆分大型项目文档

3
扩展初始化
Skills · MCP · Hooks
▾
Skill 元数据预缓存（懒加载设计）
loadAllPluginsCacheOnly()
  // 仅建立元数据索引，不加载 Skill 内容
  ├── ~/.claude/skills/          ← 全局用户 Skills
  ├── .claude/skills/            ← 项目级 Skills
  └── 内置 Skills                ← 打包进二进制

  → 结果：Map<skillName, SkillMetadata>
  → Skill 内容在匹配到任务时才懒加载（避免 Context 膨胀）
MCP 服务器连接
读取 mcpServers 配置
  │
  ├── transport: "stdio"
  │     → spawn 子进程 → stdin/stdout JSON-RPC 通道
  │
  ├── transport: "sse" / "http"
  │     → HTTP 长连接 / Server-Sent Events
  │
  └── transport: "ws"
        → WebSocket 双向通道

每台服务器建连后执行能力协商：
  GET tools/list     → 获取工具列表
  GET resources/list → 获取可读资源列表

连接失败 → warn + skip，不中断启动
关键设计：MCP 连接失败是非致命的。工具不可用不应阻断整个会话——这保证了即使外部服务宕机，Claude Code 本身仍然可用。

QueryEngine 实例化
会话级单例
▾
const config: QueryEngineConfig = {
  tools,                    // Phase 4 的 tools[]
  commands,                 // 斜杠命令列表
  mcpClients,               // Phase 3 的 MCP 连接池
  canUseTool,               // 权限决策函数（见笔记 02）
  taskBudget: new BudgetTracker({
    maxUsd:    flags.maxBudget,    // --max-budget
    maxTokens: 200_000,
  }),
  thinkingConfig: resolveThinkingMode(flags.thinking),
  // ...20+ 字段
}

const engine = new QueryEngine(config)
// 进程级单例：整个会话只有一个 QueryEngine
单例设计：QueryEngine 是整个进程的状态中枢。单例保证状态一致性，避免并发竞争——这是 Claude Code 有状态会话的基础。

6
初始系统提示组装
3路并行 + Cache预热
▾
buildSystemInitMessage() — 会话元数据
首条 system 消息写入：
  - 当前日期时间
  - OS + shell 信息
  - 有效 cwd（git root）
  - git 状态摘要（分支 + uncommitted changes）
  - 使用的模型名称
fetchSystemPromptParts() — 三路并行
Promise.all([
  buildSystemPrompt(),   ← 核心人格 + 工具描述 + 安全规则（静态）
  buildUserContext(),    ← CLAUDE.md + Memory + 用户偏好（半静态）
  buildSystemContext(),  ← 环境信息 + Hooks 指引 + 可用命令（动态）
])
→ 三路合并 → 最终 system prompt blocks[]
Prompt Cache 预热
cache_control: { type: "ephemeral" } 标记稳定块：
  ├── 系统提示核心    命中率 60-95%
  └── CLAUDE.md 内容  命中率 80%+

首次请求建立缓存 → 后续命中 → Token 成本降低 40-60%

7
会话启动
Interactive / Non-Interactive
▾
Interactive Mode
  • 打印欢迎信息（模型名、cwd）
  • 加载对话历史（--continue）
  • 显示输入提示符，等待用户
  • Enter → UserPromptSubmit Hook → QueryLoop
Non-Interactive Mode
  • 检测：!stdin.isTTY 或 --print
  • 读取 stdin / --print 参数值
  • 跳过欢迎屏，直接进入 QueryLoop
  • 输出完毕后 exit（适合脚本/CI）
→ 进入 QueryLoop（见图 01）
关键设计决策
决策实现理由
CLAUDE.md 链式加载逐级向上合并，子优先支持 monorepo 多级目录上下文
Skill 懒加载元数据启动预缓存，内容按需展开避免 Context 在启动时就被撑满
MCP 失败非致命warn + skip工具不可用不阻断整个会话
QueryEngine 单例进程级单实例状态一致性，避免并发竞争
Prompt Cache 预热首次请求标记 ephemeral同 cwd 的后续请求显著降低成本
