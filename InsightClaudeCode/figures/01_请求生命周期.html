请求生命周期
从用户输入到最终输出的完整垂直流

基于 s00b-one-request-lifecycle · QueryEngine.ts · query.ts 深度解析

STAGE 0
用户输入进入
QueryEngine.processUserInput()
用户消息首先经过预处理。斜杠命令（/commit, /review等）在此拦截直接处理，不进入主API循环。普通消息转换为 SDKMessage 进入 QueryEngine。
// 初始化 QueryState
messages: [{"role":"user","content":"..."}]
turn_count: 1
transition: None
continuation_count: 0
has_attempted_compact: False
↓SDKMessage async generator 开始
STAGE 1
提示组装
fetchSystemPromptParts() · src/utils/queryContext.ts
3路并行调用组装完整 API Payload。注意：自定义提示会完全替换默认系统提示（不合并）。
// 并行执行（Promise.all）
systemPrompt ← getSystemPrompt() // 核心指令 ~5K tokens
userContext ← getUserContext() // Memory Files ~5K tokens
systemContext ← getSystemContext() // 环境/CLAUDE.md ~8K tokens

// 最终 API Payload
system blocks + normalized messages + memory attachments
+ reminder injections + tool definitions
↓完整 API Payload → Anthropic API
STAGE 2
模型推理
claude_api(messages) · Anthropic SDK
Claude 模型接收完整上下文进行推理。输出只有两种类型：普通文本答案（请求可能在此结束）或工具调用意图（进入工具控制平面）。
→ 普通文本 (text)
→ 工具调用意图 (tool_use)
// doesMostRecentAssistantMessageExceed200k()?
if output > 200K: ESCALATED_MAX_TOKENS 恢复流程
MAX_OUTPUT_TOKENS_RECOVERY_LIMIT = 3 // 最多重试3次
↓tool_use → 进入工具控制平面
STAGE 3
工具路由
runTools() · src/query.ts
根据工具名前缀路由到不同执行目标。工具名决定了执行路径。
if tool_name.startswith("mcp__") → MCP Server (JSON-RPC 2.0)
elif tool_name == "Agent" → Sub-Agent (独立上下文)
elif tool_name.startswith("task_") → Task System (TaskRecord)
elif tool_name == "SendMessage" → Teammate (JSONL inbox)
else → Native Tool (本地执行)
↓路由确定后 → 权限检查
STAGE 4
权限决策
useCanUseTool.tsx · 5阶段处理链
5个权限处理程序按序执行，第一个产生明确决策的处理程序获胜。Deny rules 永远优先，无法绕过。
Handler 1: Config Rules → settings.json allow/deny 第一匹配胜出
Handler 2: BASH_CLASSIFIER → ML分类器（DANGEROUS_BASH_PATTERNS）并发竞态
Handler 3: Coordinator → 仅 Coordinator 模式（更高权限）
Handler 4: SwarmWorker → 仅 Worker 模式（受限权限）
Handler 5: Interactive → 弹出用户确认框（队列化防并发）
allow → 继续
deny → 返回错误
ask → 用户确认
permissionDenials[] 追踪拒绝记录 · 连续3次 → 断路器 → 建议切换 plan mode
↓权限通过 → PreToolUse Hooks
STAGE 5
PreToolUse Hooks
src/services/tools/toolHooks.ts
执行所有匹配的 PreToolUse 钩子脚本。支持3种退出码协议，以及 JSON 格式的高级阻止决策。
exit 0 → 继续执行工具（静默通过）
exit 1 → 阻止工具执行，stderr 作为错误原因
exit 2 → stderr 注入对话，工具仍然执行
echo '{"decision":"block","reason":"..."}' → JSON高级阻止
↓Hooks 通过 → 工具执行
STAGE 6
工具执行
StreamingToolExecutor · 并发模型
StreamingToolExecutor 根据 isConcurrencySafe 标志决定并发策略。并发安全工具（Read/Grep/Glob等）并行执行，非并发工具（Bash/Write等）独占串行。
// 并发安全 → 并行执行
Read(a) ║ Read(b) ║ Glob(.) ║ Grep(*) // 同时运行

// 非并发 → 串行独占
Bash(cmd1) → Bash(cmd2) // 按序执行

siblingAbortController: Bash失败 → 终止所有兄弟子进程
结果按原始请求顺序缓冲排序后输出
↓工具执行完成 → PostToolUse Hooks
STAGE 7
结果注入与状态更新
messages.append(tool_result) · 平台状态更新
工具结果不只是追加文本。一次工具调用可能同时更新多个平台状态层。
tool_result → messages.append({
role: "tool",
tool_use_id: "...",
content: "..."
})

同时可能更新:
TaskRecord (任务板状态)
RuntimeTaskState (运行时执行槽)
MemoryEntry (记忆记录) [EXTRACT_MEMORIES gate]
WorktreeRecord (Worktree记录)
FileStateCache (文件变更追踪)
tool_result_continuation max_tokens_recovery compact_retry transport_retry
↓state.transition 写入 → turn_count++ → 下一轮推理
STAGE 8
上下文压缩（条件触发）
src/services/compact/autoCompact.ts
每轮推理前先执行 microCompact（静默），若 tokens 超过阈值则触发更大规模压缩。
// 每轮前静默执行
microCompact(messages) // 旧tool_result → "[Previous: used {name}]"

// 条件触发（按优先级）
tokens > AUTOCOMPACT_BUFFER_TOKENS(13K) + threshold → autoCompact
tokens > WARNING_THRESHOLD_BUFFER_TOKENS(20K) → 显示警告
HISTORY_SNIP gate → snipCompact（外科手术式裁剪）
REACTIVE_COMPACT gate → reactiveCompact
CONTEXT_COLLAPSE gate → contextCollapse

断路器: MAX_CONSECUTIVE_AUTOCOMPACT_FAILURES = 3
↓压缩完成 → 继续推理 or 最终输出
STAGE 9
最终输出
display_output() → Stop Hooks → SessionStorage
当模型输出纯文本（无工具调用）时请求生命周期结束。触发 Stop Hooks，SessionStorage 持久化会话记录。
display_output(response) // 流式输出到终端
run_stop_hooks() // Stop Hooks 触发
SessionStorage.recordTranscript() // 持久化对话记录
SessionStorage.flushSessionStorage() // 写入磁盘
user_input = wait_input() // 等待下一个用户输入
QueryState 控制平面状态机
// 状态 vs 内容 的严格分离

【内容状态 Content State】
messages[] 对话历史
tool_result 工具输出
memory text 记忆内容

【控制状态 Process-Control State】
turn_count 当前轮次
transition 继续原因（TransitionReason）
continuation_count 继续次数（防无限循环）
has_attempted_compact 压缩标志（防重复压缩）
max_output_tokens_override 临时token限制
三层架构（请求 = 哪一层负责？）
【查询循环层 Query Loop Layer】
拥有: 多轮次请求过程
管理: QueryState + TransitionReason

【工具控制平面 Tool Control Plane】
拥有: 路由 + 权限 + Hooks + 执行上下文
核心: ToolUseContext（共享执行环境）

【平台状态层 Platform State Layer】
拥有: 持久记录（durable records）
包含: Task / RuntimeSlot / Teammate
Worktree / MCP Config / Memory
