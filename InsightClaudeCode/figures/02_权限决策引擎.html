权限决策引擎
7种权限模式 · 完整决策流水线 · Handler Chain

基于 src/types/permissions.ts · src/utils/permissions/permissions.ts · src/hooks/useCanUseTool.tsx

External Modes（5种 · 用户可配置 · 可写入 settings.json）
External · 1
default
Default
color: text（中性）
Deny规则优先 → Allow规则匹配 → 其余请求用户确认。标准工作模式，大多数用户的默认状态。
External · 2
acceptEdits
Accept edits
color: autoAccept（绿）· ⏵⏵
工作目录内文件 Write / Edit 自动放行，无需提示。safetyCheck 路径（.git/.claude 等）仍需确认。
External · 3
plan
Plan Mode
color: planMode（蓝灰）· ⏸
阻止所有写操作，只读自动通过。规划、探索阶段的安全模式。未接入 UI 循环时为 dontAsk 注释。
External · 4
bypassPermissions
Bypass Permissions
color: error（红）· ⏵⏵
跳过几乎所有权限检查，最宽松模式。但 bypass-immune 检查仍不可绕过（steps 1e/1f/1g）。
External · 5
dontAsk
Don't Ask
color: error（红）· ⏵⏵ · 未接入 UI 循环
所有 ask → deny（静默拒绝），绝不弹出确认框。适用于无人值守 CI / CD 全自动场景。
Internal Modes（2种 · Feature Gate / 代码限定）
Internal · 6 · 需要 TRANSCRIPT_CLASSIFIER Feature Gate · 对 ant（内部）用户可访问
auto
Auto mode
color: warning（黄）· ⏵⏵ · external 映射 → 'default'
AI 分类器（YoloClassifier）自主决策，替代人工确认。三级优化：acceptEdits 快速路径 → 安全工具白名单 → YoloClassifier 2阶段 API（fast + thinking）。断路器防止连续拒绝失控。
Internal · 7 · 父上下文专用 · 外部映射 → 'default'
bubble
（无 UI 标题 / 无 UI 循环入口）
color: — · external 映射 → 'default'
向父上下文冒泡权限请求，由父 Agent 决策。子 Agent 在嵌套架构中使用，外部展示统一映射为 'default'。
// src/types/permissions.ts — 权限模式权威类型定义
export const EXTERNAL_PERMISSION_MODES = [
  'acceptEdits', 'bypassPermissions', 'default', 'dontAsk', 'plan',
] as const
export type ExternalPermissionMode = typeof EXTERNAL_PERMISSION_MODES[number]       // 5种外部模式
export type InternalPermissionMode = ExternalPermissionMode | 'auto' | 'bubble'    // +2种内部 = 7种
export type PermissionMode = InternalPermissionMode

hasPermissionsToUseToolInner — 完整决策流水线（第一个明确结果获胜）
1aDeny rule for entire tool?
deny rules 命中 → 立即 DENY，优先级最高，无法被任何模式绕过
→ DENY无匹配 → 继续
1bAsk rule for entire tool?
ask rule 命中 → ASK（除非 sandbox 已自动允许）
→ ASK无匹配 → 继续
1ctool.checkPermissions(input, context)
工具自身权限检查：Bash 子命令规则、路径安全检查、工具特定逻辑
→ DENY→ ASK通过 → 继续
1dtool returned deny?
1c 中工具 checkPermissions 返回明确 deny 结果
→ DENY否 → 继续
1etool.requiresUserInteraction()?bypass-immune
工具必须与用户交互 → 强制 ASK，bypassPermissions 模式下也无法绕过
→ ASK（强制，不可绕过）否 → 继续
1fcontent-specific ask rule?bypass-immune
内容特定的 ask 规则（ruleBehavior === 'ask'）→ bypassPermissions 也无法绕过
→ ASK（bypass-immune）无 → 继续
1gsafetyCheck? (.git / .claude / .vscode / shell configs)bypass-immune
.git/ · .claude/ · .vscode/ · .github/workflows/ · .bashrc · .zshrc 等 → 强制确认，所有模式均无法绕过
→ ASK（任何模式均不可绕过）安全 → 继续
2aShould bypass permissions?
mode === 'bypassPermissions' OR (mode === 'plan' && isBypassPermissionsModeAvailable) → 直接放行
→ ALLOW否 → 继续
2bAlways-allowed rule（整体工具 allow 规则）?
整体工具 allow 规则命中（settings.json allow[] 中的整工具规则）
→ ALLOW无 → 继续
3passthrough → ask（兜底转换）
所有步骤均无明确决策时，转换为 ask，传入后处理层（dontAsk / auto / headless / Handler Chain）
→ ASK（传递给后处理层）
⚠ bypass-immune — 三个不可绕过的检查点
步骤 1e · 1f · 1g 是 bypass-immune 检查点。
即使在 bypassPermissions 模式下，
这三步产生的 ASK 结果也永不会被绕过。
这是系统最后一道安全防线。

Step 1a 的 DENY 也不可绕过（最高优先级）。
bypassPermissions 模式（Step 2a）
条件: mode === 'bypassPermissions'
OR (mode === 'plan' && isBypassAvail)

在 steps 1a-1g 全部通过后 → ALLOW

仍无法绕过: 1a(deny) · 1e · 1f · 1g
适用场景: 高信任自动化脚本
dontAsk 模式（后处理层）
流水线输出 'ask'
↓ mode === 'dontAsk'
ASK → DENY（静默）
msg: DONT_ASK_REJECT_MESSAGE(tool)
reason: { type:'mode', mode:'dontAsk' }

典型: --dangerously-skip-permissions
无人值守 CI/CD 全自动场景

触发: TRANSCRIPT_CLASSIFIER gate

shouldAvoidPermissionPrompts

Handler Chain（hasPermissionsToUseTool 返回 'ask' 后触发）
Handler Chain 在 hasPermissionsToUseToolInner 流水线之后运行，是 useCanUseTool.tsx 中的第二层处理，仅当流水线最终输出 ask 时触发。
Step 4a · Coordinator Worker
coordinatorHandler
awaitAutomatedChecksBeforeDialog 标志位为 true 时，先执行来自 Coordinator 的自动授权检查，通过后放行
↓ 未处理则继续
Step 4b · Swarm Worker
swarmWorkerHandler
Worker 模式下，将权限请求通过 mailbox 转发给 Lead Agent，等待 Coordinator 授权决策
↓ 未处理则继续
Step 4c · BASH_CLASSIFIER（竞态 2s）
speculative classifier check
2秒竞态：投机性分类器与对话框并跑。高置信度匹配 → ALLOW，跳过用户确认对话框
↓ 无结论则继续
Step 4d · Interactive（最终兜底）
interactiveHandler
弹出确认对话框，请求队列化防并发。Yes → 本次允许 · No → 拒绝并记入 permissionDenials[] · Always → 允许并写入 settings.json（断路器: 连续拒绝≥3 → 建议切换 plan mode）
auto 模式 AI 分类器（三级优化路径 · 快→慢）
目标：在保持安全性的前提下尽量跳过昂贵的 API 调用。三级路径按速度从快到慢依次尝试。
1
FAST PATH · acceptEdits 快速路径
acceptEdits 模拟检查
若该工具在 acceptEdits 模式下会被允许 → 直接 ALLOW。
跳过 YoloClassifier API 调用，最快路径。
2
ALLOWLIST · 安全工具白名单
isAutoModeAllowlistedTool(tool.name)
已知安全工具直接 ALLOW，无需分类器。
覆盖 Read · Glob · Grep 等纯只读工具。
3
CLASSIFIER · YoloClassifier API（最慢）
classifyYoloAction(messages, action)
formatActionForClassifier(tool.name, input)
→ 2阶段分类: fast + thinking
shouldBlock: false → ALLOW（记 'auto-mode'）
shouldBlock: true → DENY（记 auto-mode-denial）
断路器: consecutiveDenials / totalDenials 超限
→ autoModeCircuitBroken = true → fallback interactive
tengu_iron_gate_closed: true=fail-closed(拒绝) / false=fail-open(放行)
PowerShell例外: 需 POWERSHELL_AUTO_MODE flag 才允许分类器处理
