## 完整启动流程

```
$ claude [options]
  │
  ├── argv 解析 ─────────── --model / --project / --verbose / --print / --max-budget
  │
  ▼
Phase 1: 配置加载（三层覆盖）
  │  ~/.claude/settings.json          ← 全局配置（低优先级）
  │  .claude/settings.local.json      ← 项目本地配置（中优先级）
  │  CLI flags + 环境变量             ← 运行时参数（高优先级，覆盖前两层）
  │
  ▼
Phase 2: 项目上下文发现
  │  git root 检测 → 确定有效 cwd
  │  CLAUDE.md 链式加载（逐级向上）
  │  Memory 文件加载
  │
  ▼
Phase 3: 扩展初始化
  │  ├── loadAllPluginsCacheOnly()   → Skill/Plugin 元数据索引
  │  ├── MCP 服务器连接             → 进程启动 + 能力协商
  │  └── Hooks 注册                 → 事件 → Shell 脚本映射
  │
  ▼
Phase 4: 工具注册表组装
  │  native tools (45+) + MCP tools → 最终 tools[] 列表
  │
  ▼
Phase 5: QueryEngine 实例化
  │  QueryEngineConfig 组装完毕
  │  BudgetTracker + 权限函数 + Hooks 全部就位
  │
  ▼
Phase 6: 初始系统提示组装
  │  buildSystemInitMessage() ← 会话元数据
  │  fetchSystemPromptParts() ← 3路并行
  │  Prompt Cache 首帧预热
  │
  ▼
Phase 7: 会话启动
  ├── Interactive Mode: 显示欢迎屏 → 等待用户输入
  └── Non-Interactive:  处理 stdin / --print 参数 → 作为首条消息
  │
  ▼
→ QueryLoop（见笔记 01）
```

## Phase 1：配置加载——三层覆盖模型

### 配置优先级（低 → 高）

```
~/.claude/settings.json           [全局，所有项目共享]
    ↑ 被覆盖
.claude/settings.local.json       [项目本地，不提交 git]
    ↑ 被覆盖
CLI flags + ENV vars              [本次运行，最高优先级]
```

### settings.json 核心字段

```typescript
interface ClaudeSettings {
  model?: string                        // 默认模型
  permissions: {
    allow: string[]                     // 工具允许规则（glob 模式）
    deny:  string[]                     // 工具拒绝规则（glob 模式）
  }
  hooks?: {
    [event: string]: HookConfig[]       // 事件 → Shell 脚本
  }
  mcpServers?: {
    [name: string]: MCPServerConfig     // MCP 服务器定义
  }
  env?: Record<string, string>          // 注入到子进程的环境变量
}
```

### 模式检测

启动时检测两个关键维度：

| 维度 | 检测方式 | 影响 |
|------|---------|------|
| **交互模式** | `process.stdin.isTTY` | 决定是否显示 TUI / 等待用户输入 |
| **Coordinator 模式** | `CLAUDE_CODE_COORDINATOR_MODE` 环境变量 | 激活 Worker 接收模式（见笔记 05） |

## Phase 2：项目上下文发现

### Git Root 检测

```
运行 git rev-parse --show-toplevel
  ├── 成功 → 使用 git root 作为有效 cwd（项目感知上下文）
  └── 失败 → 使用当前目录
```

这是 Claude Code 能感知"整个项目"而非"当前目录"的关键。

### CLAUDE.md 链式加载

```
从 cwd 开始逐级向上查找：

/project/src/components/   CLAUDE.md?  → 加载，继续向上
/project/src/              CLAUDE.md?  → 加载，继续向上
/project/                  CLAUDE.md?  → 加载（项目根）
/home/user/                CLAUDE.md?  → 无，继续
~/.claude/                 CLAUDE.md   → 加载（全局 fallback）

所有找到的 CLAUDE.md 内容按 "子目录 → 父目录" 顺序合并，
子目录规则优先。
```

**`@file` 内联导入**：CLAUDE.md 中可通过 `@filename` 引用其他文件，启动时递归展开。

### Memory 文件加载

```
~/.claude/memory/MEMORY.md          ← 全局用户记忆（主入口）
  └── 嵌套 @attachment 引用         ← 主题化记忆文件
      └── 递归加载，直到无 @ 引用

.claude/memory/MEMORY.md            ← 项目级记忆（如存在）

硬限制：MEMORY.md ≤ 200行 / 25KB（超出则截断，见笔记 06）
```

## Phase 3：扩展初始化

### Skill / Plugin 元数据预加载

```typescript
loadAllPluginsCacheOnly()
  // 扫描 Skill 目录（不加载 Skill 内容，只建索引）
  ├── ~/.claude/skills/
  ├── .claude/skills/            // 项目级 Skills
  └── 内置 Skills                // 打包进二进制

// 结果：skills 元数据索引 Map<skillName, SkillMetadata>
// 注意：Skill 的实际 SKILL.md 内容此时 NOT 加载（懒加载）
```

**懒加载设计**：避免在启动时将所有 Skill 内容塞入 Context，只在匹配到任务时才展开具体 Skill 内容。

### MCP 服务器连接

```
读取 settings 中的 mcpServers 配置
  │
  ├── transport: "stdio"
  │     → spawn 子进程（本地 MCP 服务器）
  │     → 通过 stdin/stdout 建立 JSON-RPC 通道
  │
  ├── transport: "sse" / "http"
  │     → 建立 HTTP 长连接 / Server-Sent Events
  │
  └── transport: "ws"
        → 建立 WebSocket 双向通道

每个服务器建连后执行能力协商（capability negotiation）：
  GET tools/list  → 获取该服务器提供的工具列表
  GET resources/list → 获取可读资源列表

连接失败：记录 warning，跳过该服务器，不中断启动
```

### Hooks 注册

```
读取 settings.hooks 配置：

{
  "PreToolUse":  [{ "command": "security-check.sh $TOOL_NAME" }],
  "PostToolUse": [{ "command": "logger.sh" }],
  "Stop":        [{ "command": "notify.sh '任务完成'" }]
}

→ 注册到 Hook 事件分发器
→ 后续每次触发对应事件时，同步执行 Shell 脚本
```

## Phase 6：初始系统提示组装

### buildSystemInitMessage() — 会话元数据块

```
插入对话首条 "system" 消息，内容包括：

- 当前日期时间
- 操作系统 + shell 信息
- 有效 cwd（git root 或当前目录）
- git 状态摘要（当前分支 + uncommitted changes 数量）
- 使用的模型名称
- 可用内存/CPU 信息（部分环境）
```

### fetchSystemPromptParts() — 三路并行组装

```
Promise.all([
  buildSystemPrompt(),    // L1: 核心人格 + 工具描述 + 安全规则（静态，缓存命中率高）
  buildUserContext(),     // L2: CLAUDE.md + Memory + 用户偏好（半静态）
  buildSystemContext(),   // L3: 环境信息 + Hooks 指引 + 可用命令（动态）
])

三路并行 → 合并为最终 system prompt blocks[]
```

### Prompt Cache 预热

```
cache_control: { type: "ephemeral" } 标记到稳定块：
  ├── 系统提示核心部分（每次启动相同，缓存命中率 60-95%）
  └── CLAUDE.md 内容（项目内不变，缓存命中率 80%+）

首次请求建立缓存 → 后续请求命中 → Token 成本降低 40-60%
```

## Phase 7：会话启动

### Interactive Mode

```
1. 打印欢迎信息：模型名、当前目录、可用命令提示
2. 加载对话历史（如有 --continue 标志）
3. 显示输入提示符，等待用户键入
4. 用户按 Enter → 触发 UserPromptSubmit Hook → 进入 QueryLoop
```

### Non-Interactive Mode（管道 / --print）

```
检测到 !process.stdin.isTTY 或 --print 标志：

1. 从 stdin 读取完整输入 / 使用 --print 参数值
2. 直接作为首条用户消息
3. 跳过欢迎屏，直接进入 QueryLoop
4. 输出完毕后退出进程（exit code = 0 / 1）
```

## 关键设计决策

| 决策 | 实现 | 理由 |
|------|------|------|
| CLAUDE.md 链式加载 | 逐级向上合并 | 支持 monorepo 的多级目录上下文 |
| Skill 懒加载 | 元数据启动预缓存，内容按需展开 | 避免 Context 在启动时就被撑满 |
| MCP 连接失败非致命 | warn + skip | 工具不可用不应阻断整个会话 |
| QueryEngine 单例 | 进程级单实例 | 状态一致性，避免并发竞争 |
| Prompt Cache 预热 | 首次请求建立 | 同一 cwd 的后续请求显著降低成本 |
