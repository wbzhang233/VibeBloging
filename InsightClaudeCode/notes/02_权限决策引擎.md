## 一、7种权限模式（完整定义）

源码文件 `src/types/permissions.ts`：

```typescript
export const EXTERNAL_PERMISSION_MODES = [
  'acceptEdits',
  'bypassPermissions',
  'default',
  'dontAsk',
  'plan',
] as const

export type InternalPermissionMode = ExternalPermissionMode | 'auto' | 'bubble'
export type PermissionMode = InternalPermissionMode
```

### External Modes（5种，用户可配置）

| 模式 | 标题 | 颜色 | 核心行为 |
|------|------|------|---------|
| `default` | Default | text（中性） | Deny规则优先→Allow规则→其余请求用户确认 |
| `acceptEdits` | Accept edits | autoAccept（绿） | 工作目录内的文件Write/Edit自动放行，不提示 |
| `plan` | Plan Mode | planMode（蓝灰） | 阻止所有写操作，只读通过；符号: ⏸ |
| `bypassPermissions` | Bypass Permissions | error（红） | 跳过几乎所有权限检查（见例外），最宽松 |
| `dontAsk` | Don't Ask | error（红） | 所有ask→deny，静默拒绝，无头/自动化场景 |

### Internal Modes（2种，代码/Feature Gate限定）

| 模式 | 标题 | 颜色 | 核心行为 |
|------|------|------|---------|
| `auto` | Auto mode | warning（黄） | AI分类器自主决策，需要TRANSCRIPT_CLASSIFIER gate |
| `bubble` | (无显示) | — | 向父上下文冒泡权限请求；外部映射到'default' |

**注意**：`auto`模式虽是internal，对ant（内部）用户可访问；外部用户只有5种External模式。

## 三、权限决策核心流水线（`hasPermissionsToUseToolInner`）

每次工具调用的完整决策路径（共9步）：

```
工具调用到达
 │
 ▼ Step 1a: Deny rule for entire tool?
 │  deny rules 命中 → DENY
 ▼ Step 1b: Ask rule for entire tool?
 │  ask rule 命中 → ASK（除非sandbox自动允许）
 ▼ Step 1c: tool.checkPermissions(input, context)
 │  工具自身的权限检查（如Bash子命令规则、路径检查）
 ▼ Step 1d: tool returned deny? → DENY
 ▼ Step 1e: tool.requiresUserInteraction()? → ASK（强制，不可绕过）
 ▼ Step 1f: content-specific ask rule? → ASK（ask规则，bypass-immune）
 ▼ Step 1g: safetyCheck? (.git/.claude/.vscode/shell configs)
 │  → ASK（bypass-immune！bypassPermissions也无法绕过）
 │
 ▼ Step 2a: Should bypass permissions?
 │  (mode === 'bypassPermissions')
 │  OR (mode === 'plan' && isBypassPermissionsModeAvailable)
 │  → ALLOW（直接放行）
 │
 ▼ Step 2b: Always-allowed rule（整体工具allow规则）?
 │  → ALLOW
 │
 ▼ Step 3: passthrough → ask（转换兜底值）
 │
 ▼ 后处理（hasPermissionsToUseTool wrapper）:
    if mode === 'dontAsk':       ASK → DENY
    if mode === 'auto' (or plan+autoModeActive):
        → AI分类器（YOLO分类器）
            acceptEdits快速路径 → ALLOW（跳过分类器）
            安全工具白名单    → ALLOW（跳过分类器）
            YoloClassifier    → ALLOW or DENY
            拒绝次数超限      → 退回interactive提示
    if shouldAvoidPermissionPrompts（无头）:
        → PermissionRequest hooks → 或 AUTO-DENY
```

**关键设计**：bypass-immune checks（1g步的safetyCheck、1f步的ask规则、1e步的用户交互工具）即使在`bypassPermissions`模式下也不可绕过。

## 四、Handler Chain（`useCanUseTool.tsx`，在流水线之后）

当 `hasPermissionsToUseTool()` 返回 **ask** 时，进入Handler Chain：

```
Step 4（ask返回后）:
  │

  │  → coordinatorHandler（先跑自动检查）
  │
  ▼ swarmWorkerHandler
  │  Swarm Worker模式下→转发给Lead Agent via mailbox
  │
  ▼ BASH_CLASSIFIER pendingClassifierCheck?
  │  2s竞态：speculative classifier先跑
  │  高置信度match → ALLOW（跳过dialog）
  │
  ▼ interactiveHandler（显示确认对话框）
     Yes → allow（本次）
     No  → deny（记入permissionDenials）
     Always → allow + 写入settings.json
```

## 五、`auto`模式 AI分类器深度解析

**触发条件**：
- Feature Gate `TRANSCRIPT_CLASSIFIER` 开启
- `mode === 'auto'` 或 `(mode === 'plan' && autoModeStateModule.isAutoModeActive())`

**分类路径（快→慢三级）**：

```
1. acceptEdits Fast-path（跳分类器）：
   如果该工具在acceptEdits模式下会被允许 → ALLOW
   （跳过昂贵的API调用）

2. Safe-tool Allowlist（跳分类器）：
   isAutoModeAllowlistedTool(tool.name) → ALLOW

3. YoloClassifier（调用API）：
   formatActionForClassifier(tool.name, input)
   → classifyYoloAction(messages, action, ...)
   2阶段分类：fast + thinking
   → shouldBlock: false → ALLOW（记 decisionReason.classifier = 'auto-mode'）
   → shouldBlock: true  → DENY（记录auto-mode-denial）
```

**断路器**：
- `consecutiveDenials` 和 `totalDenials` 追踪拒绝次数
- 超限 → fallback到interactive（让用户审查）
- `autoModeCircuitBroken` = true 时，不进入auto（GrowthBook远程可关闭）
- `tengu_iron_gate_closed`：分类器API故障时的行为开关（true=fail-closed拒绝，false=fail-open放行）

**PowerShell例外**：
- 需要`POWERSHELL_AUTO_MODE` flag才允许auto模式处理PowerShell（防止绕过分类器）

## 六、`bypassPermissions`模式的例外清单

以下情况**即使在bypassPermissions模式下也无法绕过**：
1. `tool.requiresUserInteraction()` 返回true（step 1e）
2. 内容特定的ask规则（step 1f，`ruleBehavior === 'ask'`）
3. safetyCheck路径（step 1g）：`.git/`、`.claude/`、`.vscode/`、shell配置文件等
4. 全工具deny规则（step 1a）

## 七、`dontAsk`模式的特殊行为

```
正常流水线 → 返回 'ask'
                        │
                        ▼ mode === 'dontAsk'
                    ASK → DENY
              message: DONT_ASK_REJECT_MESSAGE(tool.name)
              decisionReason: { type: 'mode', mode: 'dontAsk' }
```

典型用途：CLI `--dangerously-skip-permissions` + 无人值守自动化场景。

## 八、`acceptEdits`模式的行为边界

- 自动允许：工作目录内文件的Write/Edit操作（`checkPathSafetyForAutoEdit` 不触发）
- 仍然需要确认：`safetyCheck`路径（.git/.claude/.vscode等）
- 也是`auto`模式的快速路径：auto模式先模拟acceptEdits检查，若通过则跳过分类器

## 九、权限规则来源（PermissionRuleSource）

```typescript
type PermissionRuleSource =
  | 'userSettings'      // ~/.claude/settings.json
  | 'projectSettings'   // .claude/settings.json
  | 'localSettings'     // .claude/settings.local.json（gitignore）
  | 'flagSettings'      // --allowed-tools CLI标志
  | 'policySettings'    // 企业策略（只读）
  | 'cliArg'            // 运行时CLI参数
  | 'command'           // 斜杠命令（只读，不可删除）
  | 'session'           // 会话内临时授权（会话结束失效）
```

**优先级**（来源重要性从高到低）：policySettings > flagSettings > userSettings > projectSettings > localSettings > cliArg > session > command

## 十一、配置文件层级（4级）

```
~/.claude/settings.json            ← userSettings（全局）
  └─ .claude/settings.json         ← projectSettings（项目级，可git提交）
       └─ .claude/settings.local.json ← localSettings（本地，gitignore）
            └─ 会话运行时规则        ← session（会话结束失效）
```

**企业策略**：`policySettings`优先级最高，由管理员通过GrowthBook/远程控制推送，`shouldAllowManagedPermissionRulesOnly()`开启时清除所有非policy规则。
