# 上下文压缩系统 — 深度研究笔记

> 来源: src/services/compact/autoCompact.ts + s06-context-compact.md + query.ts

---

## 为什么需要压缩

- 模型上下文窗口有限（200K tokens）
- 每次 API 调用包含完整对话历史 + 所有工具结果
- 读一个1000行文件 ≈ 4000 tokens
- 读30个文件 + 运行20次命令 = 100K+ tokens → 上下文满

---

## 6种压缩策略（来自源码）

### 策略0: persistedOutput（工具执行时）
- 触发：工具输出超过阈值（通用50KB，Bash 30KB）
- 动作：将完整输出写到磁盘，消息中替换为 `<persisted-output>` 预览标记
- 模型后续可用 `read_file` 读取完整内容

```
实际保存路径: .task_outputs/tool-results/abc123.txt
消息中替换为:
<persisted-output>
  Output too large (48.8KB). Full output saved to: .task_outputs/tool-results/abc123.txt
  Preview (first 2.0KB): ...first 2000 chars...
</persisted-output>
```

### 策略1: microCompact（每轮静默）
- 触发：每次 LLM 调用前自动执行
- 动作：将3轮以前的旧 tool_result 替换为 `[Previous: used {tool_name}]`
- 例外：`read_file` 的结果保留（参考材料）
- 对用户不可见

### 策略2: autoCompact（阈值触发）
- 触发：`tokens > AUTOCOMPACT_THRESHOLD`（来自 `getAutoCompactThreshold(model)`）
- 关键常量：
  - `AUTOCOMPACT_BUFFER_TOKENS = 13,000`
  - `WARNING_THRESHOLD_BUFFER_TOKENS = 20,000`
  - `MAX_CONSECUTIVE_AUTOCOMPACT_FAILURES = 3`（断路器）
  - `MAX_OUTPUT_TOKENS_FOR_SUMMARY = 20,000`
- 动作：
  1. 将完整 transcript 保存到磁盘（`.transcripts/` 目录）
  2. 调用 LLM 对对话进行摘要
  3. 将所有消息替换为 `[Compressed]\n\n{summary}`
- 环境变量覆盖：
  - `CLAUDE_CODE_AUTO_COMPACT_WINDOW`
  - `CLAUDE_AUTOCOMPACT_PCT_OVERRIDE`

### 策略3: manualCompact（模型主动触发）
- 触发：模型自主调用 `compact` 工具
- 动作：与 autoCompact 相同机制
- 区别：决策者是模型而非阈值

### 策略4: snipCompact（Feature Gate: HISTORY_SNIP）
- 触发：`HISTORY_SNIP` feature gate 激活
- 策略：裁剪（snip）历史中最大的工具结果片段
- 比 autoCompact 更外科手术式，保留更多历史结构

### 策略5: reactiveCompact（Feature Gate: REACTIVE_COMPACT）
- 触发：`REACTIVE_COMPACT` feature gate + 特定条件
- 策略：响应式压缩，在检测到上下文压力时主动压缩

### 策略6: sessionMemoryCompact（SessionMemory服务）
- 触发：会话结束或切换时
- 策略：将会话记忆压缩持久化到 `.claude/session-memory/`

### 策略7: contextCollapse（Feature Gate: CONTEXT_COLLAPSE）
- 触发：`CONTEXT_COLLAPSE` feature gate
- 策略：极端情况下的完全上下文折叠

## 压缩触发状态机

```
正常对话
  │ tokens 持续增长
  ▼
[WARNING_THRESHOLD_BUFFER = 20K 前] → 显示警告提示
  │
  ▼
[AUTOCOMPACT_THRESHOLD] ← getAutoCompactThreshold(model) 计算
  │ 触发 autoCompact
  ▼
[成功] → 继续，has_attempted_compact = True
[失败] → MAX_CONSECUTIVE_AUTOCOMPACT_FAILURES = 3 断路器
  │ 3次连续失败
  ▼
[放弃压缩，报错或以原始内容继续]
```

## 完整四杠策略（s06 概念框架）

```
每次工具调用结果:
  ↓ [Lever 0: persisted-output] 大输出 → 磁盘 + 预览标记
  ↓ [Lever 1: micro_compact] 旧tool_result → 占位符
  ↓ [Check: tokens > 50K?]
     │ 否 → 继续
     │ 是 → [Lever 2: auto_compact] 摘要并替换全部消息
  ↓ [Lever 3: compact tool] 模型主动压缩（同Lever2机制）
```

## QueryState 中的压缩相关字段

```python
state = {
    "has_attempted_compact": False,  # 是否已尝试过压缩（防重复）
    "transition": "compact_retry",   # 压缩后的继续原因
    "max_output_tokens_override": N, # 压缩期间临时缩减输出上限
}
```
