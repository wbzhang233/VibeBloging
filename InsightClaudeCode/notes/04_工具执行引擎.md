## StreamingToolExecutor 并发执行模型

### 核心设计

每个工具请求被包装为 `TrackedTool`，携带关键标志：
```typescript
interface TrackedTool {
  id: string
  name: string
  input: any
  isConcurrencySafe: boolean  // ← 关键：决定是否可并发
  status: 'pending' | 'running' | 'done' | 'error'
  result?: any
  abortController: AbortController
}
```

### 并发执行规则

```
并发安全工具（isConcurrencySafe = true）:
  ┌──────────┐  ┌──────────┐  ┌──────────┐
  │ Read(a)  │  │ Read(b)  │  │ Grep(.)  │  ← 并行执行
  └──────────┘  └──────────┘  └──────────┘

非并发工具（isConcurrencySafe = false）:
  ┌──────────┐
  │  Bash()  │  ← 独占执行，其他工具等待
  └──────────┘

  ┌──────────┐
  │  Write() │  ← 然后这个
  └──────────┘
```

### 结果排序

- 工具调用结果按原始请求顺序缓冲并排序后输出
- 即使并发执行，输出顺序与模型请求顺序一致

### siblingAbortController

- 当一个 Bash 执行失败时，`siblingAbortController` 终止所有兄弟子进程
- 防止失败后其他并发进程继续执行造成副作用

## 工具控制平面（Tool Control Plane）

来自 s02a，工具系统的4层架构：

```
1. ToolSpec（工具规格）
   └─ 模型看到的：name + description + input_schema

2. Tool Router（路由器）
   └─ 决策路由：native / MCP / agent / task / team
   ├─ native:  直接本地执行
   ├─ mcp__*:  路由到MCP Server
   ├─ agent:   派生子代理
   └─ task:    创建后台任务

3. ToolUseContext（执行上下文，关键！）
   └─ 工具执行环境：
      - tools: 处理器Map
      - permission_context: 当前权限状态
      - mcp_clients: MCP连接池
      - messages: 当前消息历史（可读）
      - app_state: 应用状态
      - notifications: 通知通道
      - cwd: 当前工作目录

4. ToolResultEnvelope（结果封包）
   └─ 标准化返回：{ok, content, is_error, attachments}
```

**核心认知**：高完成度系统的关键不在于工具数量，而在于共享执行上下文（ToolUseContext）的设计质量。

## 工具调用完整管道（单次）

```
模型输出 tool_use XML/JSON
  │
  ▼
XML/JSON 解析（工具名 + 参数）
  │
  ▼
Permission Gate（5阶段权限检查）
  │ 通过
  ▼
PreToolUse Hooks 执行
  │ exit 0 → 继续 | exit 2 → 注入消息继续 | exit 1/JSON{decision:block} → 阻止
  ▼
StreamingToolExecutor.execute()
  │ isConcurrencySafe? → 并发 or 串行
  ▼
实际工具处理函数（Read/Bash/Agent/...）
  │
  ▼
PostToolUse Hooks 执行
  │
  ▼
ToolResultEnvelope 封装
  │
  ▼
messages.append(tool_result)
  │
  ▼
平台状态更新（task/runtime/memory/worktree）
  │
  ▼
下一轮推理
```

## 后台 Bash 模式

- Bash 工具支持后台执行模式
- `run_in_background: true` 参数
- 通过 `TaskOutput` 工具异步获取结果
- `TaskStop` 可终止后台进程
