# 记忆与上下文系统详解

> 本文档是可视化 `images/04_记忆上下文系统.html` 的技术脚本。

---

## 一、记忆体系全景

Claude Code 实现了四层记忆架构，解决了 LLM 天生无状态的问题：

```
┌─────────────────────────────────────────────────────┐
│  Layer 4: Prompt Cache（提示缓存）                    │
│  目的：减少重复内容的 token 消耗                      │
│  存储：Anthropic 服务端，TTL ~5 分钟                  │
├─────────────────────────────────────────────────────┤
│  Layer 3: Memory Files（持久记忆）                    │
│  目的：跨会话保存用户偏好、项目知识                    │
│  存储：.claude/memory/ 目录（本地文件）                │
├─────────────────────────────────────────────────────┤
│  Layer 2: Dialog History（动态上下文）                │
│  目的：当前会话连贯性                                  │
│  存储：内存（会话结束丢失）                            │
├─────────────────────────────────────────────────────┤
│  Layer 1: CLAUDE.md（固定上下文）                     │
│  目的：项目规范、约定、始终生效                        │
│  存储：本地文件（可提交到 git）                        │
└─────────────────────────────────────────────────────┘
```

---

## 二、Layer 1：CLAUDE.md（固定上下文）

### 加载机制

Claude Code 启动时按**从内到外**的顺序查找并合并 CLAUDE.md：

```
优先级（高 → 低）：
1. 当前工作目录的 CLAUDE.md
2. 父目录的 CLAUDE.md（逐级向上）
3. ~/.claude/CLAUDE.md（全局用户配置）
```

### 内容规范

```markdown
# 典型 CLAUDE.md 结构

## Project Overview
项目简介、技术栈、核心目标

## Working Conventions
- 代码风格约定（如：使用 snake_case）
- 测试要求（如：所有新函数必须有单元测试）
- 禁止事项（如：不要修改 migrations 目录）

## Architecture Notes
关键文件路径、模块职责、依赖关系

## Memory Section
由 Claude 自动维护的项目决策记录
```

### 动态更新

用户可在不退出会话的情况下更新 CLAUDE.md：

```
/memory  →  打开编辑器直接修改
```

## 三、Layer 2：Dialog History（动态上下文）

### 增长与压缩

对话历史随交互线性增长，是 token 消耗的主要来源：

```
会话开始：
  System Prompt（~5K tokens）
  + 用户消息（~200 tokens）
  = ~5.2K tokens

10轮对话后（包含文件读取）：
  System Prompt + 所有消息 + 所有工具结果
  = ~50K-100K tokens

上下文 70% 阈值触发自动压缩
  压缩后：~15K-25K tokens
```

### 压缩策略

```
/compact 执行过程：

输入：完整对话历史（100K tokens）
         ↓
[LLM 生成摘要]
    "目前已完成：1. 读取了 auth.py，发现 JWT 实现；
     2. 修改了 middleware.py 添加了 token 验证；
     3. 测试全部通过。待完成：4. 更新 API 文档"
         ↓
输出：摘要 + 最近 N 轮对话（~15K tokens）
```

## 四、Layer 3：Memory Files（持久记忆）

### 目录结构

```
~/.claude/projects/<project-hash>/memory/
├── MEMORY.md          ← 主记忆文件（每次对话加载前 200 行）
├── debugging.md       ← 调试问题专项记忆
├── patterns.md        ← 代码模式约定
└── architecture.md    ← 架构决策记录
```

### 记忆更新策略

Claude Code 在识别到需要记忆的信息时主动更新 Memory Files：

```
触发条件：
✅ 用户明确要求记住（"从现在起记住..."）
✅ 跨会话需要的稳定模式/约定
✅ 已确认的架构决策
✅ 调试洞见（如"该框架的 config 在 x 路径"）

不应记录：
❌ 临时的任务上下文
❌ 未验证的假设
❌ 可从代码中重新推断的信息
❌ 与 CLAUDE.md 重复的内容
```

## 五、Layer 4：Prompt Cache（提示缓存）

### 工作原理

```
第一次 API 调用：
  System Prompt (10K) + 用户消息 (200) = 10.2K tokens 全部计费
                    ↓
  Anthropic 服务器缓存 System Prompt

第二次 API 调用（5 分钟内）：
  System Prompt (10K) [缓存命中, 0 费用] + 新消息 (500) = 500 tokens 计费
```

### 缓存命中率

```
高命中率（>90%）：
  - System Prompt（固定，每次对话开始都相同）
  - CLAUDE.md 内容（每次加载相同）

中等命中率（50-70%）：
  - 对话早期历史（在压缩前不变）

低命中率（<30%）：
  - 最近几轮的工具结果（动态变化）
```

## 六、信息放置决策树

```
"我有一条信息，应该放在哪里？"

是否需要跨会话持久保存？
├── 否 → 直接在对话中使用（无需存储）
└── 是
    │
    是否是项目规范/约定（所有人都应遵守）？
    ├── 是 → CLAUDE.md（提交到 git，团队共享）
    └── 否
        │
        是否是 Claude 的个人工作记忆（用户偏好、调试经验）？
        ├── 是 → Memory Files（本地，不提交 git）
        └── 否
            │
            是否是可以从代码重新推断的架构信息？
            ├── 是 → 既然可以重新推断，不需要存储
            └── 否 → Memory Files（注明来源和可信度）
```
