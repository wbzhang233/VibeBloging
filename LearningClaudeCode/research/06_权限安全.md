## 一、安全设计哲学

Claude Code 的安全模型基于两个核心原则：

1. **最小权限原则**：默认不信任，只授予完成任务所需的最小权限
2. **人在回路（Human-in-the-Loop）**：高风险操作强制人工确认，不可绕过

## 二、四级权限层级

│  级别 1（最外层）: Global Settings                           │
│  位置: ~/.claude/settings.json                              │

│                                                             │
│  ┌─────────────────────────────────────────────────────┐    │

│  │ 位置: .claude/settings.json（项目根目录）             │    │
│  │ 作用: 项目级权限，覆盖全局设置                         │    │
│  │                                                     │    │
│  │  ┌─────────────────────────────────────────────┐   │    │
│  │  │ 级别 3: Runtime Dynamic Authorization       │   │    │
│  │  │ 触发: 会话期间，用户批准特定操作              │   │    │
│  │  │ 作用: 会话内临时授权，会话结束后失效           │   │    │
│  │  │                                             │   │    │
│  │  │  ┌───────────────────────────────────────┐ │   │    │
│  │  │  │ 级别 4（最内层）: One-time Approval    │ │   │    │
│  │  │  │ 触发: 单次操作的临时允许               │ │   │    │

│  │  │  └───────────────────────────────────────┘ │   │    │
│  │  └─────────────────────────────────────────────┘   │    │
│  └─────────────────────────────────────────────────────┘    │

## 三、配置文件结构

### Global Settings（~/.claude/settings.json）

```json
{
  "permissions": {
    "allow": [
      "Bash(git *)",
      "Bash(npm *)",
      "Bash(python *)",
      "Read(*)",
      "Write(*)",
      "Edit(*)"
    ],
    "deny": [
      "Bash(rm -rf *)",
      "Bash(git push --force *)",
      "Bash(curl * | bash)",
      "Bash(sudo *)"
    ]
  },
  "env": {
    "DEBUG": "false"
  }
}
```

### Project Settings（.claude/settings.json）

```json
{
  "permissions": {
    "allow": [
      "Bash(docker *)",
      "Bash(./scripts/*.sh)"
    ],
    "deny": [
      "Bash(* production *)"
    ]
  }
}
```

## 四、高危操作拦截清单

Claude Code 内置的高危操作检测（即使有全局 allow，也会弹出二次确认）：

| 类型 | 示例命令 | 拦截级别 |
|------|---------|---------|
| **数据毁灭** | `rm -rf /` `rm -rf ./` `rm -rf *` | 强制拒绝 |
| **强制推送** | `git push --force` `git push -f` | 强制确认 |
| **强制覆盖** | `git reset --hard` `git checkout .` | 强制确认 |
| **密钥暴露** | `cat ~/.ssh/id_rsa` 发送到外部 | 强制确认 |
| **外部消息发送** | Slack/Email/GitHub 发消息 | 强制确认 |
| **CI/CD 修改** | 修改 `.github/workflows/` | 强制确认 |
| **数据库破坏** | `DROP TABLE` `TRUNCATE` | 强制确认 |
| **软件包篡改** | 修改已发布包 | 强制确认 |
| **管道执行** | `curl * | bash` | 强制拒绝 |
| **权限提升** | `sudo` `chmod 777` | 强制确认 |

## 五、操作风险矩阵

```
              可逆性
              高 ←────────────────────→ 低
              │                         │
高  ────────  │  Read（读文件）          │  rm -rf（删除文件）
              │  Glob（列文件）          │
影            │  Grep（搜索）            │
响            │─────────────────────────│
范            │  Edit（修改文件）        │  git push --force
围            │  Write（新建文件）       │  DROP TABLE
              │  Bash（运行脚本）        │
低  ────────  │  WebFetch（读网页）      │  发送外部消息
              │                         │  修改 CI/CD
```

**颜色编码**（可视化中使用）：
- 🟢 绿色：低风险，自动允许
- 🟡 黄色：中风险，会话内授权
- 🔴 红色：高风险，强制确认
- ⛔ 黑色：极端风险，默认拒绝

## 六、Plan Mode（计划模式）

Plan Mode 将任务执行分为两个严格隔离的阶段：

```
探索阶段（Exploration Phase）
─────────────────────────────
允许工具：Read, Glob, Grep, WebFetch, WebSearch, Agent(Explore/Plan)
禁止工具：Write, Edit, Bash（含执行）, Agent(general-purpose)

用户操作：查看计划 → 修改/批准

↓ 用户批准 ↓

执行阶段（Execution Phase）
─────────────────────────────
允许工具：全部工具
按照批准的计划执行
```

**价值**：在任务开始就对齐方向，避免偏差越跑越远；对复杂/危险任务尤为重要。

## 七、权限检查流程（Harness 实现）

```python
def check_permissions(tool_call: ToolCall) -> PermissionResult:
    tool_name = tool_call.name
    tool_input = tool_call.input

    # 1. 极端危险操作 → 直接拒绝
    if matches_hardcoded_deny(tool_name, tool_input):
        return PermissionResult.DENY

    # 2. 全局 deny 规则
    for rule in global_settings.deny:
        if matches(tool_name, tool_input, rule):
            return PermissionResult.DENY

    # 3. 全局 allow 规则
    for rule in global_settings.allow:
        if matches(tool_name, tool_input, rule):
            return PermissionResult.ALLOW

    # 4. 项目级 deny 规则
    for rule in project_settings.deny:
        if matches(tool_name, tool_input, rule):
            return PermissionResult.DENY

    # 5. 项目级 allow 规则
    for rule in project_settings.allow:
        if matches(tool_name, tool_input, rule):
            return PermissionResult.ALLOW

    # 6. 运行时临时授权
    if tool_call.id in runtime_approvals:
        return PermissionResult.ALLOW

    # 7. 默认 → 询问用户
    return PermissionResult.NEEDS_APPROVAL
```

## 八、安全最佳实践

| 场景 | 推荐配置 |
|------|---------|
| 个人开发机 | 较宽松的全局 allow，加入常用命令 |
| 团队共享环境 | 严格的 deny 规则，尤其是生产环境操作 |
| CI/CD 自动化 | `--print` 非交互模式 + 最小化 allow 列表 |
| 代码审查场景 | Plan Mode 优先，执行前人工确认 |
| 多 Agent 场景 | Explore/Plan 子代理只读，Execute 子代理 worktree 隔离 |
