## 写在前面

2025 年 2 月，OpenAI 前研究科学家 Andrej Karpathy 在 X 上发了一条帖子：

> *"There's a new kind of coding I call 'vibe coding', where you fully give in to the vibes, embrace exponentials, and forget that the code even exists."*

这条帖子迅速引爆技术社区。一年后，这个词已经不再是玩笑——它正在成为一种真实的生产力范式。

本文不讲玄学，只讲方法：**Vibe Coding 是什么，为什么它有效，以及如何用 Claude Code 把它做到又快又准。**

---

## 一、什么是 Vibe Coding

### 1.1 Karpathy 的原始定义

Karpathy 描述的 Vibe Coding，有几个关键特征：

1. **完全信任 AI**：不逐行审查代码，接受 AI 给的结果，测试能跑就行
2. **用语言，不用逻辑**：用自然语言描述你想要什么，而不是告诉 AI 怎么做
3. **接受"指数级增长"**：能力边界不再受限于你的技术栈深度
4. **忘掉代码的存在**：你关注的是结果，而不是实现路径

他的例子非常具体：遇到 bug，直接把报错丢给 AI，不自己看，不自己想，让 AI 修；AI 给了修复方案，直接接受，不读代码。

### 1.2 它和传统 AI 辅助编程的区别

| 维度 | 传统 AI 辅助编程 | Vibe Coding |
|------|----------------|-------------|
| 主导者 | 开发者主导，AI 辅助 | AI 主导，开发者引导 |
| 关注点 | 每一行代码正确性 | 结果是否达到预期 |
| 交互方式 | 我说具体怎么做 | 我说想要什么 |
| 审查方式 | 逐行 Code Review | 运行验证、结果验收 |
| 适用场景 | 生产核心代码 | 原型、工具、探索性开发 |

### 1.3 Vibe Coding 的哲学本质

Vibe Coding 的核心不是"偷懒"，而是一种**认知资源的重新分配**：

```
传统开发的认知分配：
  30% 思考架构
  40% 思考实现细节（语法、API 调用、边界处理）
  30% 调试

Vibe Coding 的认知分配：
  60% 思考架构和目标
  10% 引导 AI 方向
  30% 验证结果
```

你把"实现细节"外包给了 AI，把节省出来的认知资源用于"做什么、为什么这么做"——这才是工程师最稀缺的能力。

### 1.4 Vibe Coding 适合什么，不适合什么

**适合**：
- 原型快速验证（Product-Market Fit 探索）
- 内部工具、脚本、自动化流程
- 不熟悉的技术栈快速上手
- 需求明确的功能模块实现

**不适合**：
- 对正确性要求极高的核心业务逻辑（支付、权限、密码学）
- 需要严格审计的合规场景
- 你不理解任何代码的情况下写生产系统

> **原则**：Vibe Coding 不是不看代码，而是战略性地选择在哪些地方用认知资源审查代码。

## 二、为什么 Claude Code 是 Vibe Coding 的最佳拍档

### 2.1 Claude Code 是 Agent，不是补全工具

Vibe Coding 需要的不是"写下一行代码"的工具，而是"理解我的目标并自主完成任务"的系统。

Claude Code 恰好提供了这种能力：

```
用户："帮我给这个 Express API 加上 JWT 认证"

补全工具的反应：
  等待你把光标放到某个位置，提示下一行代码

Claude Code 的反应：
  1. 扫描项目结构，理解当前的路由和中间件架构
  2. 确认是否已有 jsonwebtoken 依赖
  3. 创建认证中间件文件
  4. 修改需要保护的路由
  5. 添加 token 生成的登录逻辑
  6. 运行测试验证
  7. 告知完成并说明使用方式
```

### 2.2 Claude Code 的 Agentic Loop 天然契合 Vibe 节奏

Vibe Coding 的本质是"给目标，等结果"。Claude Code 的 Agentic Loop 正是这个模式：

```
你的角色：
  定义目标 → 验证结果

Claude Code 的角色：
  理解目标 → 探索方案 → 调用工具执行 → 验证 → 循环直到完成
```

你不需要关心它中间做了什么，只需要关心最终结果符不符合预期。

## 三、高效 Vibe Coding 的 7 个核心技巧

### 技巧 1：用结果描述需求，而不是步骤

**反模式**（告诉 AI 怎么做）：
```
"先读取 users.json 文件，然后过滤 active 为 true 的用户，
再对 email 字段去重，最后写入 output.json"
```

**Vibe 模式**（告诉 AI 要什么）：
```
"从 users.json 里提取所有活跃用户的唯一邮箱列表，保存到 output.json"
```

区别在哪？前者你在做软件工程，后者你在表达意图。

意图描述会让 AI 有更多自由度选择最优实现路径。更重要的是，当需求发生变化时，意图描述比步骤描述更容易调整。

**进阶版**：加上"为什么"，帮助 AI 做更好的权衡：
```
"从 users.json 提取所有活跃用户的唯一邮箱。
这是用于给用户发送营销邮件的，所以去重必须准确，
宁可少发也不要重复发。"
```

---

### 技巧 2：上下文给够，但不要给多余的

这是 Vibe Coding 最容易出问题的地方。AI 的输出质量，直接取决于上下文质量。

**上下文不足的后果**：
```
用户："帮我优化这个查询"
AI：写了一个通用的 SQL 优化建议
结果：根本不适合当前的业务场景
```

**上下文过多的后果**：
```
用户：把所有代码、所有文档、所有历史对话全贴进去
AI：在信息海洋里迷失，抓不住重点
```

**正确做法**：给"刚好够用"的上下文

```
好的上下文结构：
1. 当前问题是什么
2. 相关的代码在哪里（让 Claude Code 自己去读，而不是粘贴）
3. 已经尝试了什么（避免 AI 重复已经踩过的坑）
4. 验证成功的标准是什么

示例：
"src/services/UserService.ts 里的 getUserById 方法，
在高并发下会有 N+1 查询问题，日志里能看到（我发给你看）。
已经试过加索引，没用。希望从查询层面解决，
验证标准是 1000 QPS 下响应时间 < 100ms。"
```

**利用 CLAUDE.md 沉淀上下文**

不要每次都重复说项目背景。把不变的东西写进 CLAUDE.md：

```markdown
# CLAUDE.md

## 项目背景
这是一个面向 B 端客户的 SaaS 产品后端，用户量级 10万+。

## 技术约定
- 所有异步操作使用 async/await，禁止回调嵌套
- 错误处理统一用 Result<T, Error> 模式
- 数据库查询性能要求：P99 < 200ms

## 不允许的操作
- 不允许直接修改 migrations/ 目录
- 不允许在未写测试的情况下修改核心业务逻辑
```

这样你每次 Vibe Coding 时，不需要反复交代背景——Claude Code 已经知道了。

---

### 技巧 3：先规划，再执行（Plan Mode 的正确用法）

直接让 AI 开始写代码，在复杂任务里是陷阱。

当任务涉及多个文件、不确定的边界、或者有多种可行方案时，**先让 AI 出计划，你确认后再执行**。

```
命令：/plan（或者在 VS Code 扩展里切换 Plan Mode）

提示词范式：
"我想在现有系统里加上多租户支持。
先给我一个方案，包括：
1. 需要改动哪些文件
2. 数据库层面需要什么变更
3. 潜在风险点
4. 你打算用什么顺序来做

不要开始写代码，先等我确认方案。"
```

为什么这很重要？

**没有计划直接干的后果**：
- AI 改了一半，你发现方向不对，但已经改了 20 个文件
- 撤销成本极高，比重新做还麻烦

**有计划再干的好处**：
- 你可以在一个 diff 都没发生之前，对齐方向
- 发现方案问题时，修改成本为零

> **高级技巧**：让一个 Claude 出计划，再开一个新会话让另一个 Claude 审计这个计划，最后再执行。以 AI 审 AI 的方式，显著提升方案质量。

---

### 技巧 4：小步快跑，不要一次性给大任务

Vibe Coding 的节奏是：给一个子任务 → 验证 → 给下一个子任务。

**不推荐**：
```
"帮我实现整个用户系统，包括注册、登录、找回密码、
权限管理、用户画像、消息通知..."
```

**推荐**：
```
第一轮："帮我实现用户注册功能，包括邮箱验证"
  → 测试通过

第二轮："在刚才的基础上加登录和 JWT"
  → 测试通过

第三轮："加找回密码"
  → 测试通过
```

小步快跑的好处：
1. 每一步都是可验证的
2. 出了问题，范围明确（只有这一步可能有问题）
3. 上下文保持干净（不会在一个大任务里越来越混乱）
4. 可以在任意节点调整方向

---

### 技巧 5：验证驱动，而不是信任驱动

Karpathy 说"忘掉代码的存在"，但他没说"忘掉结果验证"。

Vibe Coding 的验证原则：**不审查过程，但必须验证结果**。

**验证的三个层次**：

```
层次 1：运行不崩溃（最低要求）
  → 跑起来，主流程能走通

层次 2：功能正确（核心要求）
  → 测试用例覆盖核心路径

层次 3：边界安全（高要求）
  → 对于核心代码：注入测试、边界值测试、并发测试
```

在 Claude Code 里，验证可以被自动化：

```
提示词：
"实现完这个功能后，顺手写几个测试用例验证：
1. 正常路径能走通
2. 输入为空的边界情况
3. 并发调用不会出现竞争条件

用 pytest 跑一下，把结果告诉我。"
```

更进一步，用 Hooks 把验证变成自动行为：

```json
// .claude/settings.json
{
  "hooks": {
    "PostToolUse": [{
      "matcher": "Edit|Write",
      "hooks": [{
        "type": "command",
        "command": "npm test -- --watchAll=false"
      }]
    }]
  }
}
```

每次 Claude Code 修改文件后，自动跑测试——这样验证不再需要你主动操心。

---

### 技巧 6：管理好上下文窗口

上下文窗口是 Vibe Coding 的"汽油"，用完了 AI 就开始"犯糊涂"。

**症状识别**：
- AI 开始忽视你之前的约定
- AI 重复已经解决的错误
- AI 给出的方案和项目风格不一致

**应对策略**：

**策略一：主动 /compact**

不要等到上下文自动压缩，在上下文达到 70-80% 时主动压缩：

```
/compact
```

**策略二：长任务换新会话**

开始新会话前，让 AI 写一份"交接文档"：

```
"在我们开启新会话之前，帮我总结一下：
1. 到目前为止我们完成了什么
2. 当前状态是什么
3. 下一步打算怎么做
4. 有哪些还没解决的问题

我把这个总结带到新会话里，你读一遍就能接着干。"
```

**策略三：用子 Agent 隔离重量级操作**

如果需要扫描大量文件，用子 Agent 处理：

```
"启动一个 Explore Agent，扫描整个 src/ 目录，
找出所有直接操作数据库的代码（不经过 Repository 层的），
给我一个文件列表，不需要读每个文件的详细内容。"
```

子 Agent 在独立的上下文里运行，不消耗你的主会话 token。

---

### 技巧 7：让 AI 主动汇报，而不是你主动问

在 Vibe Coding 的节奏里，你不应该每隔五分钟问"你做到哪了"。

**好的提示词应该包含"完成后做什么"**：

```
差的提示词：
"帮我重构这个模块"

好的提示词：
"帮我重构这个模块。完成后：
1. 告诉我改了哪些文件、改了什么
2. 跑一下相关测试，告诉我通过率
3. 如果有任何改动你不确定的地方，明确标注出来"
```

这样你不需要盯着过程，只需要等最终的报告。

## 四、准确性保障：5 个让 AI 不跑偏的原则

效率解决了"快"，接下来解决"准"。

### 原则 1：约束优先于鼓励

很多人习惯告诉 AI"你很厉害，尽情发挥"——这是 Vibe Coding 里最容易出问题的做法。

准确的 AI 输出，来自**清晰的约束**：

```
❌ 鼓励式：
"你是一个资深工程师，帮我用最优雅的方式实现这个功能"

✅ 约束式：
"帮我实现这个功能，约束：
- 只能修改 src/services/ 目录下的文件
- 不能引入新的第三方依赖
- 必须兼容现有的 UserService 接口
- 函数必须有 JSDoc 注释"
```

约束减少了 AI 的"创意空间"，但大幅提升了输出的可预测性。

### 原则 2：给反例，不只给正例

只描述"我想要什么"，AI 的理解可能有偏差。加上"我不想要什么"，准确度会显著提升：

```
"帮我写一个数据格式化函数。
我想要：输入任意对象，输出标准化的 API 响应格式
我不想要：
- 不要用 any 类型
- 不要在函数里有 console.log
- 不要改变原始对象（纯函数）
- 不要处理嵌套超过 2 层的对象"
```

### 原则 3：阶段性确认，而不是全程放任

Vibe Coding 不是让 AI 自己跑完所有步骤，而是在关键决策点介入：

```
多步骤任务的节奏：

第一步：让 AI 说出计划 → 你确认
第二步：让 AI 完成第一个关键步骤 → 你验证
第三步：给出下一步的 OK，继续执行
...
```

在哪些节点需要介入：
- **不可逆操作之前**（删数据、重构接口、修改 DB Schema）
- **涉及多个团队代码的改动**
- **你对结果有明确预期但 AI 可能有多种理解的需求**

### 原则 4：错误反馈要具体，不要含糊

当 AI 给的结果不对，很多人会说"这不对，重新来"——这几乎总会给出同样错误的结果。

**具体的错误反馈**：

```
❌ 模糊反馈：
"这个不对，重新写"

✅ 具体反馈：
"你给的这个函数有一个问题：当 users 数组为空时，
第 23 行的 users[0].id 会抛出 TypeError。
另外，你用了 filter().map()，性能比直接用 reduce() 差。
针对这两点修改一下，其他部分保持不变。"
```

具体的反馈让 AI 知道：什么是对的（保持），什么是错的（改）。

### 原则 5：用 Hooks 强制执行检查

对于重要的约束，不要靠"提醒 AI"，用 Hooks 让它成为强制行为：

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write",
        "hooks": [{
          "type": "command",
          "command": "eslint {{file}} --fix"
        }]
      }
    ]
  }
}
```

每次 AI 写完文件，自动跑 linter。AI 不遵守代码规范？不存在的——规范在 Hook 里强制执行了。

## 六、常见误区

### 误区 1："Vibe Coding 就是不看代码"

Karpathy 的原话是在特定语境下的夸张表达。实际上：

- **原型阶段**：可以不精读每一行，跑通就行
- **生产代码**：关键路径必须审查，尤其是涉及安全、数据、权限的部分

Vibe 的意思是"流状态"，不是"无脑接受"。

### 误区 2："任务越大越好，一次给完"

事实恰恰相反：任务越大，AI 的准确率越低，你的验证成本越高。

最有效的 Vibe Coding 是：把大任务拆成小任务，每一步都快速迭代。

### 误区 3："AI 理解我的意图，不需要明确说"

AI 没有读心术。含糊的需求会得到含糊的结果。

越是"Vibe"，越需要精准地表达**目标**——不需要说实现路径，但目标必须清晰。

### 误区 4："Vibe Coding 不需要技术基础"

技术基础决定了你：
- 能否判断 AI 的方案是否可行
- 能否给出准确的错误反馈
- 能否在关键节点介入并纠偏

没有技术基础的 Vibe Coding，本质上是运气驱动的。技术基础让你能**战略性地放权**，而不是被迫放权。

## 七、Vibe Coding 的本质：人机分工的新范式

最后，回到哲学层面。

Vibe Coding 不是一个技巧，而是对**人与 AI 协作方式**的重新定义：

```
旧范式：
  人 = 思考者 + 实现者
  AI = 辅助实现者

Vibe Coding 范式：
  人 = 思考者 + 决策者 + 验证者
  AI = 规划者 + 实现者 + 自动验证者
```

这意味着：

- 工程师的核心价值从"会写代码"转向"会定义问题、验证结果、做架构决策"
- "能不能写出这段代码"不再是瓶颈，"知不知道要写什么、写完了对不对"才是

Claude Code 是目前最接近这个范式的工具——它有足够的自主性执行复杂任务，有足够的透明度让你知道它在做什么，有足够的可控性让你在关键节点介入。

**会用 Vibe Coding 的工程师，不是写更少代码的工程师——是写出更重要代码的工程师。**

## 快速参考卡

```
提示词结构：
  "在 [范围] 里，[结果描述]。
   约束：[不允许做什么]。
   完成后：[验证方式 + 汇报内容]。"

上下文管理：
  /init → 新项目时生成 CLAUDE.md
  /context → 查看当前使用量
  /compact → 70-80% 时压缩
  /clear → 任务切换时清空
  /plan → 复杂任务先规划

节奏控制：
  简单任务 → 直接下指令 → 验证
  复杂任务 → 出计划 → 确认 → 执行 → 验证
  超大任务 → 拆小 → 循环"简单任务"节奏

验证层次：
  L1：能运行（跑不崩）
  L2：功能正确（测试通过）
  L3：边界安全（边界用例覆盖）
```
