## 1. n8n 是什么——先从本质说清楚

### 一句话定义

> n8n 是一个**工作流自动化平台**（Workflow Automation Platform），和 Zapier、Make（原 Integromat）是同类产品——它的核心能力是：把不同的服务和系统连接起来，按照你定义的流程自动执行。

n8n 的 slogan 是 **"Workflow automation for technical people"**——比 Zapier 更灵活（可以写代码），比纯代码框架更直观（有可视化界面）。

### 核心特点

- **可视化拖拽**：节点图界面，所见即所得
- **400+ 内置集成**：Slack、飞书、GitHub、数据库、HTTP...
- **可以写代码**：JavaScript / Python 代码节点，技术人员无限制
- **Self-hosted**：可以完全部署在自己服务器上，数据不出墙
- **开源**：GitHub ⭐ 55k+，MIT 许可证（企业功能需付费）

## 2. n8n vs AI Agent 框架：根本区别在哪里

这是最重要的概念辨析。

### 控制权在谁手里

```
n8n（工作流平台）：
  你定义好每一步 → 按固定流程执行 → AI 是某个步骤里的工具

AI Agent 框架（AgentScope / AutoGen / CrewAI）：
  你给出目标 → AI 自主决定执行路径 → AI 是流程的驱动者
```

### 类比说明

用餐厅来比喻：

- **n8n** = 餐厅的 SOP 手册，每个步骤写死，某个步骤可能用到 AI 厨师来做菜
- **AI Agent 框架** = 一个有自主判断力的大厨，他自己决定先做什么、遇到问题怎么处理、什么时候叫助手帮忙

### 功能对比

| 维度 | n8n | AI Agent 框架 |
|------|-----|--------------|
| **流程控制** | 人定义固定工作流 | AI 自主决策执行路径 |
| **AI 的角色** | 工作流中的一个步骤 | 工作流本身的驱动者 |
| **多 Agent** | 不支持 | 核心能力 |
| **上手难度** | 低（拖拽即用） | 中-高（需要编程） |
| **适用人群** | 技术 + 非技术人员 | 开发者为主 |
| **适合场景** | 定时任务、事件触发、服务集成 | 复杂推理、多步骤决策、角色协作 |

### 什么时候应该用 n8n 而不是 AI Agent 框架？

```
用 n8n 的信号：
- "每天 9 点自动抓取新闻，AI 分析后发到飞书"
- "当 GitHub 有新 PR，自动通知相关人员并生成摘要"
- "定时同步数据库 A 的数据到数据库 B"
- 流程步骤固定，只是某个步骤需要 AI 辅助

用 AI Agent 框架的信号：
- "让 AI 自己决定用哪些工具来完成这个复杂任务"
- "需要多个 AI 角色互相讨论、碰撞、审核"
- "任务路径不固定，需要 AI 根据中间结果动态调整"
```

## 3. n8n 核心架构与设计理念

### 3.1 节点图模型

n8n 的所有内容都是**节点（Node）**，工作流就是节点之间的连线：

```
触发器（Trigger Node）
       |
       v
处理节点1（Transform/Filter）
       |
       v
处理节点2（HTTP Request / Database）
       |
    +--+--+
    |     |
    v     v
分支A   分支B（条件路由）
    |     |
    +--+--+
       |
       v
   AI Agent 节点
   （AI 处理某一步）
       |
       v
   输出节点（发送消息/写入数据库）
```

### 3.2 节点类型

| 类型 | 作用 | 示例 |
|------|------|------|
| **触发器** | 启动工作流的条件 | 定时、Webhook、邮件接收、文件变更 |
| **数据处理** | 变换/过滤数据 | Set、Filter、Merge、Split |
| **服务集成** | 调用外部 API | HTTP Request、飞书、GitHub、Slack |
| **AI 节点** | 调用 LLM | AI Agent、Chat Model、Vector Store |
| **代码节点** | 自定义逻辑 | Code（JS/Python） |
| **流程控制** | 分支/循环 | IF、Switch、Loop Over Items |

### 3.3 数据流模型

n8n 里每个节点处理的是**条目数组（Items Array）**：

```javascript
// 每个节点接收和输出的数据格式
[
  { json: { title: "新闻标题1", content: "..." } },
  { json: { title: "新闻标题2", content: "..." } },
  ...
]

// Loop Over Items 节点：对每条数据独立处理
// Merge 节点：合并多个来源的数据
```

## 4. AI Agent 节点：n8n 如何集成 AI 能力

2023 年起，n8n 深度集成了 AI 能力，核心是 **AI Agent 节点**。

### 4.1 AI Agent 节点结构

```
AI Agent 节点
├── Chat Model（选择 LLM）
│   ├── OpenAI（GPT-4o）
│   ├── Claude（Sonnet / Haiku）
│   ├── Ollama（本地模型）
│   └── 其他支持的模型
│
├── Memory（对话记忆）
│   ├── Window Buffer Memory（保留最近 N 轮）
│   ├── Summary Memory（摘要压缩）
│   └── 无记忆（每次独立）
│
├── Tools（工具集）
│   ├── HTTP Request Tool（调用任意 API）
│   ├── Code Tool（执行 JS/Python）
│   ├── Vector Store Tool（语义搜索）
│   └── 其他 n8n 内置工具
│
└── System Prompt（系统提示词）
```

### 4.2 一个完整的 AI Agent 节点配置示例

场景：每日财经新闻分析助手

```
System Prompt：
你是一个专业的 A 股市场分析助手。每天收到一批财经新闻后，你需要：
1. 筛选出与 A 股相关的重要新闻（忽略无关内容）
2. 分析每条新闻对市场的潜在影响（利多/利空/中性）
3. 汇总输出今日市场情绪评分（1-10分）和 3 条最重要的投资信号

Tools 配置：
- HTTP Request（用于搜索更多背景信息）
- Code Tool（用于计算数值指标）

Memory：Window Buffer（保留最近 5 轮，支持追问）
```

### 4.3 AI Agent 节点 vs 真正的 Multi-Agent 框架

这里要特别说清楚一个常见误解：

```
n8n 的 AI Agent 节点：
  一个 LLM + 几个工具 = 单个 Agent

真正的 Multi-Agent（AgentScope / AutoGen）：
  多个独立 Agent + 消息传递协议 + 角色分工 = Agent 系统

n8n 可以通过多个 AI Agent 节点串联来模拟多 Agent，
但这不是真正的多 Agent 协作——节点之间没有对话，只有数据传递。
```

## 5. 快速上手：三个典型场景

### 场景一：市场情报日报（定时触发）

这是 n8n 最典型的 AI 赋能场景：

```
工作流设计：

[Schedule Trigger：每天 8:00]
          |
          v
[HTTP Request：抓取多个财经 RSS 订阅]
          |
          v
[Code Node：解析 RSS，提取标题+摘要+链接]
          |
          v
[AI Agent Node：
  模型：Claude Haiku（成本低）
  任务：筛选 A 股相关新闻，逐条分析情绪（利多/利空），计算综合评分
  工具：HTTP Request（必要时搜索更多背景）]
          |
          v
[Code Node：格式化为飞书卡片 JSON]
          |
          v
[Feishu Node：发送到指定群组]
```

整个工作流在 n8n 中**纯拖拽 + 配置**即可实现，无需写代码（除了 RSS 解析的简单 JS）。

### 场景二：智能客服机器人（Webhook 触发）

```
工作流设计：

[Webhook Trigger：接收飞书/微信消息]
          |
          v
[IF Node：判断消息类型（文字/图片/语音）]
          |
    +-----+-----+
    |           |
    v           v
 文字消息    图片消息
    |           |
    v           v
[AI Agent   [Vision API
  Node：     分析图片]
  RAG+对话]      |
    |           |
    +-----+-----+
          |
          v
[Feishu Node：回复消息]
```

### 场景三：代码 Review 自动化（GitHub Webhook）

```
工作流设计：

[GitHub Webhook：PR opened/updated]
          |
          v
[HTTP Request：获取 PR 的 diff 内容]
          |
          v
[AI Agent Node：
  模型：GPT-4o
  任务：代码审查（安全性、性能、规范性），生成结构化评论
  工具：HTTP Request（查询相关文档/规范）]
          |
          v
[GitHub Node：自动在 PR 上发表 Review 评论]
          |
          v
[Slack/飞书：通知 PR 作者]
```

## 6. Self-hosted 部署指南

n8n 最大的优势之一是**完全私有化部署**，数据不出墙。

### 6.1 Docker 快速启动

```bash
# 最简单：单容器启动
docker run -it --rm \
  --name n8n \
  -p 5678:5678 \
  -v ~/.n8n:/home/node/.n8n \
  n8nio/n8n
```

### 6.2 Docker Compose（推荐生产环境）

```yaml
version: '3.8'

services:
  n8n:
    image: n8nio/n8n:latest
    restart: unless-stopped
    ports:
      - "5678:5678"
    environment:
      - N8N_BASIC_AUTH_ACTIVE=true
      - N8N_BASIC_AUTH_USER=admin
      - N8N_BASIC_AUTH_PASSWORD=your_password
      - N8N_HOST=your-domain.com
      - N8N_PROTOCOL=https
      - WEBHOOK_URL=https://your-domain.com/
      - GENERIC_TIMEZONE=Asia/Shanghai
    volumes:
      - n8n_data:/home/node/.n8n
    depends_on:
      - postgres

  postgres:
    image: postgres:15
    restart: unless-stopped
    environment:
      - POSTGRES_USER=n8n
      - POSTGRES_PASSWORD=n8n_password
      - POSTGRES_DB=n8n
    volumes:
      - postgres_data:/var/lib/postgresql/data

volumes:
  n8n_data:
  postgres_data:
```

### 6.3 接入本地 LLM

n8n 支持 Ollama，可以完全脱离外网运行 AI 工作流：

```
配置步骤：
1. 本地启动 Ollama：ollama run qwen2.5:7b
2. n8n → AI Agent Node → Chat Model → Ollama
3. 填写 Ollama 地址：http://host.docker.internal:11434
4. 选择模型：qwen2.5:7b
```

## 7. n8n 的边界：什么时候不该用它

### 7.1 不适合的场景

```
不适合 n8n 的信号：
- 需要多个 AI 互相对话、碰撞、审核
- AI 需要自主决定执行路径（不是人预设好的流程）
- 需要精细控制 LLM 的推理过程
- 高并发生产级 API 服务
- 复杂的 Python 数据处理（用 LangChain/原生代码更合适）
```

### 7.2 常见误区

**误区一：n8n 的 AI Agent 节点 = 多智能体框架**

n8n 的 AI Agent 节点只是一个增强版的 LLM 调用，它有工具调用和记忆，但没有：
- 多个 Agent 之间的消息传递
- Agent 角色分工机制
- Agent 之间的协商和审核

**误区二：n8n 可以替代 LangChain**

n8n 适合"固定流程 + AI 某步辅助"，LangChain 适合"AI 自主决策整个任务链"。两者互补，不是替代关系。

**误区三：n8n 只适合非技术人员**

实际上，n8n 的 Code 节点支持完整的 JavaScript 和 Python，技术人员用起来同样得心应手，且可以在某些节点写任意复杂的逻辑。

## 8. 总结

n8n 的本质定位：

```
n8n = 工作流自动化平台
    + 可视化界面（非技术友好）
    + Self-hosted（数据隐私）
    + AI 集成能力（AI 赋能某个步骤）
```

它不是 AI Agent 框架，但它是 **AI 赋能的自动化平台**。两者不是竞争关系，而是互补：

- 你的任务有固定流程，只是某个环节需要 AI 智能处理 → **n8n**
- 你的任务需要 AI 自主规划路径，多 Agent 协作完成 → **AgentScope / AutoGen / CrewAI**
- 在实际项目中，n8n 负责调度和触发，AI Agent 框架负责核心推理，两者配合才是最佳实践
