> 作者：内部技术分享
> 时间：2026-03-30
> 受众：产品 + 研发混合，重点兼顾业务人员

## 写在前面

如果你用过 Claude Code，你可能注意到它能通过 `/commit`、`/review-pr` 这样的斜杠命令快速执行复杂任务。

这背后的机制叫 **Skills**。

Skills 是 Claude Code 最被低估的功能之一。很多人以为它只是"自定义快捷方式"，但实际上，它是一套完整的**任务级可复用知识封装系统**。

理解 Skills，不只是理解一个功能——而是理解如何把一个组织的业务经验，**系统化地注入 AI 的执行上下文**。

本文从技术原理讲起，覆盖创建方法、写作范式、常见误区，最后重点讲业务人员如何把自己的日常工作流程"skill 化"。

---

## 一、什么是 Skills

### 1.1 一句话定义

**Skill 是一个可复用的、带上下文注入能力的任务执行模板。**

它的本质是：当 Claude 判断当前任务与某个 Skill 相关时，自动把该 Skill 的完整指令加载进对话上下文，并按照这些指令完成任务。

### 1.2 从用户视角看

你在 Claude Code 里输入 `/code-review`，或者直接说"帮我做一次代码审查"：

```
方式一（显式调用）：
  用户 → /code-review
  ↓
  Claude 加载 code-review Skill 的完整指令
  ↓
  按照预设标准执行代码审查

方式二（自动触发）：
  用户 → "帮我看看这段代码有没有问题"
  ↓
  Claude 扫描描述，匹配到 code-review Skill
  ↓
  自动加载并执行
```

两种方式都能触发同一套预设流程，输出结果一致、可预期。

### 1.3 Skills 与其他概念的对比

Skills 经常和 MCP Tools、Hooks、Memory 混淆，一张表说清楚：

| 维度 | Skills | MCP Tools | Hooks | Memory |
|------|--------|-----------|-------|--------|
| **本质** | 任务指令模板 | 外部能力接入 | 生命周期钩子 | 持久化记忆 |
| **触发方式** | 用户显式或 Claude 自动 | Claude 判断是否调用 | 系统事件自动触发 | Claude 随时参考 |
| **存储位置** | `.claude/skills/` 目录 | 独立 MCP 服务器 | `settings.json` | `.claude/memory/` 目录 |
| **做什么** | 注入操作流程和知识 | 调用外部 API / 执行代码 | 在特定事件前后执行逻辑 | 记录会话间的偏好和状态 |
| **典型用途** | 代码审查流程、部署规范 | 查询数据库、调用 GitHub API | 提交前自动格式化 | 记住用户偏好、项目约定 |

**记忆口诀：**
- Skills → **我该怎么做这件事**（流程和知识）
- MCP Tools → **我去哪里取数据 / 执行操作**（外部能力）
- Hooks → **每次做 X 之前/之后强制做 Y**（自动拦截）
- Memory → **我记得用户上次说过…**（跨会话记忆）

## 二、Skills 的工作原理

### 2.1 三层延迟加载机制

Skills 不会把所有内容一次性塞进 Claude 的上下文。它采用**三层延迟加载**策略，在节省 token 的同时保证相关内容按需注入：

```
第一层：元数据（始终加载）
  ├─ name: "code-review"
  ├─ description: "按团队标准执行代码审查..."
  └─ 常驻 Claude 的系统提示中（~200 tokens）

第二层：指令体（触发时加载）
  ├─ SKILL.md 的完整正文
  ├─ 详细的操作步骤和规范
  └─ 触发后注入对话上下文

第三层：引用资源（按需加载）
  ├─ references/ 目录中的文档
  ├─ scripts/ 目录中的脚本
  └─ Claude 按需读取
```

这个设计的好处：
- 你可以注册几十个 Skill，不会因为元数据太多影响基础对话性能
- 只有真正被调用的 Skill，它的完整内容才会进入上下文
- 大型知识库可以拆成多个引用文件，按需加载

### 2.2 触发机制：纯语言匹配

**Skills 的触发完全由语言模型判断，没有硬编码的关键词匹配。**

Claude 在处理每个请求时：

1. 读取所有已注册 Skill 的 `name` 和 `description`
2. 理解用户的意图
3. 判断哪个 Skill（如果有）与当前意图最匹配
4. 加载该 Skill 的完整指令体

这意味着：
- ✅ 用户说"帮我 review 一下代码"→ 触发 code-review Skill
- ✅ 用户说"看看这段逻辑有没有问题"→ 也能触发
- ✅ 用户说"这个函数写得怎么样"→ 也能触发
- ❌ 用户说"帮我画一个流程图"→ 不会误触发

**触发质量完全取决于 description 写得好不好。** 这是 Skill 最核心的设计决策。

### 2.3 上下文注入的完整流程

```
用户输入："帮我部署到生产环境"
    ↓
Claude 扫描可用 Skill 列表（元数据）
    ↓
匹配到 deploy Skill（description 相关）
    ↓
检查：disable-model-invocation 是否为 true？
  → 是：拒绝自动触发，提示用户手动 /deploy
  → 否：继续
    ↓
加载 deploy/SKILL.md 完整内容
    ↓
更新 allowed-tools 权限（如有限制）
    ↓
在隔离上下文（context: fork）或当前上下文执行
    ↓
按 SKILL.md 指令逐步完成任务
```

### 2.4 隔离执行模式（context: fork）

设置 `context: fork` 后，Skill 会在一个**子 Agent** 中执行：

```
主对话上下文
  ↓
[fork] 创建子 Agent
  ├─ 独立上下文（看不到主对话历史）
  ├─ 独立工具权限（按 allowed-tools 限制）
  ├─ 执行完整 Skill 流程
  └─ 返回最终结果给主对话

主对话继续（不受子 Agent 执行过程影响）
```

适合用于：
- 危险操作（部署、数据库变更），隔离风险
- 长流程任务，不污染主对话上下文
- 需要专注执行、不受外部干扰的场景

## 三、如何创建一个 Skill

### 3.1 目录结构

Skill 是 `.claude/skills/` 下的一个**目录**，目录名就是 Skill 的名称：

```
.claude/
└── skills/
    ├── code-review/           # 代码审查 Skill
    │   ├── SKILL.md           # 必须：核心指令文件
    │   ├── references/        # 可选：知识库文档
    │   │   ├── style-guide.md
    │   │   └── checklist.md
    │   └── scripts/           # 可选：可执行脚本
    │       └── run-checks.sh
    │
    ├── deploy/                # 部署 Skill
    │   └── SKILL.md
    │
    └── weekly-report/         # 周报 Skill
        ├── SKILL.md
        └── references/
            └── template.md
```

### 3.2 SKILL.md 文件格式

SKILL.md 由两部分组成：**YAML 前置元数据** + **Markdown 指令正文**。

```markdown
---
name: code-review
description: |
  按团队编码规范执行代码审查。当用户提交代码、请求代码检查、
  或询问代码质量时触发。检查内容包括：命名规范、函数复杂度、
  错误处理、安全漏洞（SQL注入、XSS）、性能问题。
allowed-tools: Read, Grep, Glob
---

# 代码审查标准流程

## 审查清单

按以下顺序逐项检查：

### 1. 命名规范
- 变量名是否语义明确（禁止 `tmp`、`data`、`obj` 等无意义命名）
- 函数名是否动词开头（`getUserById` 而非 `userById`）
- 常量是否全大写（`MAX_RETRY_COUNT`）

### 2. 函数复杂度
- 单个函数不超过 50 行
- 圈复杂度不超过 10（if/for/while 嵌套层数）
- 发现超标时：拆分函数并说明原因

### 3. 错误处理
- 所有 async 函数必须有 try-catch
- 不允许空 catch（`catch(e) {}`）
- 错误信息必须有上下文

### 4. 安全检查
- SQL 查询必须参数化，禁止字符串拼接
- 用户输入必须转义后才能进入 HTML
- 密钥和密码不得硬编码

## 输出格式

以 Markdown 表格输出审查结果：

| 文件 | 行号 | 类型 | 问题描述 | 建议修改 |
|------|------|------|---------|---------|
| ... | ... | 错误/警告/建议 | ... | ... |

最后给出：**总体评分（1-10）** 和 **核心改进建议（3条以内）**
```

### 3.3 前置元数据字段详解

| 字段 | 类型 | 必须 | 说明 |
|------|------|------|------|
| `name` | string | ✅ | Skill 的唯一标识符，即斜杠命令名（如 `code-review` → `/code-review`）。只能包含小写字母、数字、连字符，最长 64 字符，必须与目录名一致。 |
| `description` | string | ✅ | **最关键的字段**。Claude 根据这段描述判断何时调用。需说明：做什么、什么场景用、解决什么问题。最长 1024 字符，建议 200 字左右。不能包含 XML 标签。 |
| `disable-model-invocation` | boolean | ❌ | `true` = 禁止 Claude 自动触发，只能用户手动 `/skill-name` 调用。**危险操作（部署、数据库写入、发消息）必须设置此项。** |
| `user-invocable` | boolean | ❌ | `false` = 不在用户菜单中显示，只有 Claude 能自动调用。适合纯知识型 Skill（规范文档、架构决策）。 |
| `allowed-tools` | string | ❌ | 逗号分隔的工具名列表，限制该 Skill 能使用的工具。提供安全沙箱。示例：`Read, Grep, Glob`（只读）；`Read, Write, Bash`（读写）。 |
| `context` | string | ❌ | 设为 `fork` 则在隔离子 Agent 中执行。适合危险操作或长流程任务。 |
| `metadata` | object | ❌ | 自定义键值对，用于版本管理、作者标注等。不影响执行逻辑。 |

### 3.4 不同场景的配置模板

**场景 A：只读分析型 Skill（安全，允许自动触发）**

```yaml
---
name: analyze-performance
description: 分析代码性能瓶颈，定位慢查询、内存泄漏、无效渲染等问题
allowed-tools: Read, Grep, Glob
---
```

**场景 B：危险操作型 Skill（必须手动触发）**

```yaml
---
name: deploy-prod
description: 部署应用到生产环境，执行完整安全检查和回滚预案
disable-model-invocation: true
context: fork
---
```

**场景 C：知识注入型 Skill（仅供 Claude 自动参考）**

```yaml
---
name: team-conventions
description: 团队技术规范、架构决策和代码风格指南
user-invocable: false
---
```

**场景 D：带版本管理的业务流程 Skill**

```yaml
---
name: weekly-report
description: 生成周报，汇总本周完成事项、下周计划和风险点
metadata:
  version: "2.1.0"
  author: "ops-team"
  last-updated: "2026-03-28"
  owner: "运营部"
---
```

## 四、写好 Skill 的原则与范式

### 4.1 核心原则：Description First

**Skill 最重要的不是正文，是 description。**

Description 决定了 Claude 是否知道"该在什么时候用这个 Skill"。正文再完善，description 写得模糊，Skill 就形同虚设。

**写好 description 的三要素：**

```
1. 主动描述：以动词开头，说明 Skill 做什么
   ❌ "代码相关的检查"
   ✅ "按照团队规范执行代码审查，检查命名、复杂度、安全性"

2. 场景描述：说明在什么情况下触发
   ❌ "代码审查"
   ✅ "当用户提交 PR、请求代码检查、或问代码写得好不好时触发"

3. 范围描述：说明覆盖哪些方面，避免歧义
   ❌ "处理代码问题"
   ✅ "检查内容包括：命名规范、函数复杂度、错误处理、SQL注入等安全漏洞"
```

**一个好 description 的完整示例：**

```
按团队编码规范执行代码审查。触发条件：用户提交代码审查请求、
描述"看看这段代码"、"有没有问题"、"check一下"等。
覆盖范围：命名规范（禁无意义变量名）、函数复杂度（≤50行）、
错误处理（必须有try-catch）、安全漏洞（SQL注入/XSS）。
输出：Markdown表格 + 总体评分（1-10）。
```

### 4.2 正文写作范式

好的 Skill 正文遵循 **STEP 范式**：

```
S - Structure（结构化）：用编号列表，不用叙述性文字
T - Target（明确目标）：每一步说清楚要做什么、产出什么
E - Examples（带例子）：至少 3 个触发场景示例
P - Precise Output（精确输出）：明确规定输出格式
```

**❌ 不好的正文示例：**

```markdown
# 代码审查

请检查代码质量，看看有没有问题，然后给出一些建议。
注意代码规范，如果有错误的话提醒一下用户。
```

**✅ 好的正文示例：**

```markdown
# 代码审查标准流程

## 触发场景示例
- "帮我 review 这个 PR"
- "这段代码写得怎么样"
- "checkoutController.js 有没有问题"

## 审查步骤（按顺序执行，不可跳过）

### Step 1：读取目标文件
使用 Read 工具读取用户指定的文件，若未指定则读取最近修改的文件。

### Step 2：命名规范检查
- [ ] 变量名是否有语义（禁：tmp, data, obj, x, y）
- [ ] 函数名是否动词开头（getUserById ✓，userById ✗）
- [ ] 常量是否 UPPER_CASE

### Step 3：复杂度检查
- [ ] 单函数不超过 50 行 → 超过则标记并建议拆分
- [ ] 嵌套层数不超过 3 层

### Step 4：错误处理检查
- [ ] async 函数必须有 try-catch
- [ ] catch 块不得为空

### Step 5：安全检查
- [ ] SQL 查询是否参数化
- [ ] 用户输入是否转义

## 输出格式（严格按此格式）

| 文件 | 行号 | 严重级别 | 问题 | 建议 |
|------|------|---------|------|------|

**总体评分：X/10**
**Top 3 改进建议：**
1. ...
2. ...
3. ...
```

### 4.3 文件组织范式

Skill 正文超过 **200 行**时，考虑拆分：

```
code-review/
├── SKILL.md              ← 核心流程（<100 行）
│   内容：触发示例 + 步骤索引 + 输出格式
│   引用：详细标准见 references/
│
└── references/
    ├── naming-rules.md    ← 命名规范（详细版）
    ├── security-checks.md ← 安全检查清单
    └── output-examples.md ← 输出示例
```

SKILL.md 用简洁的引用代替大段内容：

```markdown
## 命名规范

详细规则见 `references/naming-rules.md`。核心要求：
- 变量名必须有语义
- 函数名动词开头
- 常量 UPPER_CASE
```

### 4.4 安全设计原则

按操作的"不可逆程度"选择配置：

```
只读分析（可逆 ✓）
  → allowed-tools: Read, Grep, Glob
  → 不需要 disable-model-invocation

文件修改（可逆 ⚠️）
  → allowed-tools: Read, Write, Edit
  → 视情况设置 disable-model-invocation

外部操作（不可逆 ❌）
  → 如：部署、数据库写入、发邮件/消息
  → disable-model-invocation: true
  → context: fork（隔离执行）
```

**黄金法则：凡是影响生产环境或对外发送数据的 Skill，必须 `disable-model-invocation: true`。**

## 五、常见误区

### 误区 1：Description 写成标题，不写场景

```yaml
# ❌ 错误
description: 代码审查工具

# ✅ 正确
description: |
  按团队规范执行代码审查。用户说"review代码"、"看看代码"、
  "有没有问题"时触发。检查命名、复杂度、错误处理、安全漏洞。
```

**后果：** Claude 不知道何时调用，或在不合适的时候乱触发。

### 误区 2：一个 Skill 做太多事

```
# ❌ 错误：一个 Skill 涵盖所有开发任务
name: development-helper
- 代码审查
- 单元测试生成
- 性能分析
- 文档生成
- 部署上线

# ✅ 正确：每个任务一个 Skill
name: code-review       # 专注审查
name: generate-tests    # 专注测试
name: analyze-perf      # 专注性能
```

**后果：** Description 无法精准描述场景，Claude 选错 Skill；正文混乱，执行结果不稳定。

### 误区 3：危险操作不设保护

```yaml
# ❌ 错误：部署 Skill 允许 Claude 自动触发
name: deploy
description: 部署到生产环境
# 没有 disable-model-invocation: true

# 后果：用户说"把最新代码上线"，Claude 可能直接触发部署
```

### 误区 4：正文全是叙述性文字

```markdown
# ❌ 错误
请先读取用户的代码，然后仔细查看是否有问题，
如果发现了问题的话，可以给用户一些建议和改进意见，
总体上保持友好的语气。

# ✅ 正确
## 步骤（按顺序，不可跳过）
1. 读取目标文件（使用 Read 工具）
2. 执行命名检查（见 Checklist A）
3. 执行安全检查（见 Checklist B）
4. 输出结果（严格按照输出格式）
```

**后果：** Claude 执行结果不稳定，每次输出不一致。

### 误区 5：混淆 Skill 和 MCP Tools 的边界

```
# 不应该用 Skill 做的事：
- 实时查询数据库 → 用 MCP Tools
- 调用第三方 API → 用 MCP Tools
- 动态获取外部数据 → 用 MCP Tools

# 应该用 Skill 做的事：
- 定义操作流程 → 用 Skill
- 注入领域知识 → 用 Skill
- 规范输出格式 → 用 Skill
```

### 误区 6：忘记写触发示例

触发示例是 description 的重要补充，Claude 会参考它来判断边界：

```yaml
# ❌ 没有示例
description: 生成周报

# ✅ 有示例
description: |
  生成本周工作周报。触发示例：
  "帮我写周报"、"总结一下这周"、"生成周报"、
  "我要发周报了"、"写一下本周情况"。
  禁止触发：用户问具体技术问题时不触发。
```

### 误区 7：Skill 过大导致上下文爆炸

```
# ❌ 错误
SKILL.md: 2000 行（把所有规范都写进去）

# ✅ 正确
SKILL.md: 80 行（核心流程 + 引用）
references/style-guide.md: 500 行（详细规范）
references/security-checklist.md: 300 行（安全清单）
```

**SKILL.md 建议控制在 300 行以内。**

## 六、业务流程 Skill 化指南

> 本节专为业务人员准备。即使你不写代码，也可以用这套方法把自己的工作流程封装成 Skill。

### 6.1 什么样的业务流程适合 Skill 化

不是所有流程都值得 Skill 化。先用这 4 个维度评估：

```
┌─────────────────────────────────────────┐
│           Skill 化价值矩阵               │
│                                         │
│  高频使用 × 规范明确 = 强烈建议 Skill 化  │
│  高频使用 × 规范模糊 = 先梳理流程再 Skill  │
│  低频使用 × 规范明确 = 可以但优先级低     │
│  低频使用 × 规范模糊 = 不建议 Skill 化   │
└─────────────────────────────────────────┘
```

**适合 Skill 化的信号：**
- "这个我们每次都这么做，但每次还要重新讲一遍给 Claude 听"
- "这个流程有固定的检查项，容易漏"
- "这个任务对输出格式要求很严格"
- "这个流程需要固定的专业知识支撑"

**不适合 Skill 化的信号：**
- "这个每次情况都不一样" → 流程太灵活，不适合模板化
- "我们还在摸索最佳做法" → 流程未定型，先探索再 Skill 化
- "这个只用过一次" → 不够高频，直接和 Claude 描述就好

### 6.2 业务流程梳理的五步法

在动手写 SKILL.md 之前，先做流程梳理：

**Step 1：完整走一遍（记录版本）**

先把这个任务完整做一遍，把你给 Claude 说的每一句话、提供的每一份资料，全部记录下来。

```
示例：竞品分析任务
我给 Claude 说的：
- "帮我分析竞品 X 和竞品 Y"
- "对比维度包括：功能、价格、用户口碑、技术架构"
- "最后给出一个建议，我们的产品应该如何差异化"
- "输出格式用表格"

我额外提供的：
- 我们公司的产品介绍文档
- 竞品的官网截图
```

**Step 2：提炼可复用的部分**

区分"每次固定的"和"每次变化的"：

```
竞品分析任务：
  ✅ 固定的（写入 Skill）：
    - 对比维度（功能/价格/口碑/技术）
    - 分析框架（SWOT 或自定义框架）
    - 输出格式（Markdown 表格）
    - 差异化建议维度

  🔄 变化的（调用时提供）：
    - 具体的竞品名称
    - 分析的时间范围
    - 重点关注的维度
```

**Step 3：确定触发时机**

写下所有你可能对 Claude 说的话，这些就是 description 的触发示例：

```
我会这样要求：
- "分析一下竞品"
- "做个竞品对比"
- "看看 X 和 Y 比我们怎么样"
- "市场上有哪些竞品，我们的优势在哪里"
```

**Step 4：确定安全级别**

问自己：这个 Skill 的输出会不会直接影响外部？

```
只是生成文档 → 普通 Skill，允许自动触发
会发邮件/消息 → disable-model-invocation: true
会修改系统数据 → disable-model-invocation: true + context: fork
```

**Step 5：写 SKILL.md**

按照格式写，重点在 description 和步骤的清晰度。

### 6.3 粒度判断：Skill 应该多大

Skill 的粒度是最难判断的问题。**以下是粒度判断框架：**

**粒度过大的信号（需要拆分）：**

```
❌ 一个 Skill 涵盖整个项目生命周期：
   name: product-launch
   内容：市场调研 + 竞品分析 + 产品定义 + 上线策略 + ...

→ 拆成：market-research、competitive-analysis、product-spec、launch-strategy
```

- 触发条件模糊，描述超过 400 字还不够精准
- 完成一次任务需要用户多次补充信息
- 输出结果经常偏离预期

**粒度过小的信号（需要合并）：**

```
❌ 每个小动作都是一个 Skill：
   name: write-email-subject    # 只写邮件主题
   name: write-email-body       # 只写邮件正文
   name: write-email-signature  # 只写邮件签名

→ 合并成：write-business-email
```

- 每次使用都需要连续调用 2-3 个 Skill
- 单个 Skill 不到 15 行内容
- Skill 之间强依赖（A 必须在 B 之前）

**恰好的粒度标准：**

```
✅ 一个 Skill 对应一个独立完整的工作产出
   - "完成这个 Skill，我会得到一份可以直接使用的东西"
   - 触发条件可以用 100 字以内清晰描述
   - 独立执行，不依赖其他 Skill
   - 执行时间：1-15 分钟（太短说明太碎，太长说明太大）
```

**业务场景粒度示例：**

| 业务场景 | 过细（❌） | 适中（✅） | 过粗（❌） |
|---------|----------|----------|----------|
| 运营 | 写标题/写正文/写CTA（各一个） | 写推广文案（一个） | 运营工作（一个） |
| 销售 | 写开场白/写需求探索/写结尾（各一个） | 写销售邮件（一个） | 全套销售流程（一个） |
| 产品 | 写背景/写方案/写指标（各一个） | 写 PRD（一个） | 产品设计全流程（一个） |
| 数据 | 数据清洗/数据分析/写报告（各一个） | 数据分析报告（一个） | 所有数据工作（一个） |

### 6.4 业务 Skill 模板（可直接复用）

以下是几个常见业务场景的 SKILL.md 模板：

**模板 A：文档撰写类**

## 生成步骤
1. 收集信息：询问用户 [缺少的必要信息]
2. 撰写内容：按以下结构输出
3. 格式检查：确认符合输出格式要求

## 内容结构
### 章节一：[标题]
[说明这部分写什么，字数范围，语气要求]

### 章节二：[标题]
[说明这部分写什么，字数范围，语气要求]

## 输出格式
[明确规定：字数、格式（Markdown/纯文本）、语气（正式/轻松）]

**模板 B：分析决策类**

```markdown
---
name: [分析任务名，如 competitive-analysis]

---

# [分析任务] 执行框架

## 分析维度（固定，不可省略）
1. [维度一]：[说明分析角度和关注点]
2. [维度二]：[说明分析角度和关注点]

## 数据来源优先级
1. 用户提供的文件/链接
2. 用户描述的信息
3. 如数据不足，明确告知用户缺少什么，不得猜测

## 结论框架
- 关键发现（3-5 条）
- 风险提示（如有）
- 行动建议（1-3 条，可操作的）

## 输出格式
[具体格式规定]
```

```markdown
---
name: [审核任务名，如 contract-review]

- [ ] [检查项3]：[说明检查标准]

### 建议项（有问题时建议修改）

**问题清单：**

```

### 6.5 业务 Skill 的命名规范

好的命名让团队一眼看懂这个 Skill 做什么：

```
格式：[动词]-[对象](-[范围])

✅ 好的命名：
  write-weekly-report      写周报
  review-contract          审合同
  analyze-competitor       分析竞品
  generate-meeting-notes   生成会议纪要
  draft-proposal           起草方案
  check-compliance         合规检查

❌ 不好的命名：
  helper          太模糊
  tool1           无意义
  my-skill        不描述功能
  do-everything   太宽泛
```

## 七、完整示例：从业务流程到 Skill

以"竞品分析报告"为例，走完完整的 Skill 化流程。

### 7.1 业务流程梳理（运营视角）

**原始流程：**
> 每周市场同学需要对 2-3 个竞品做分析，要看功能更新、价格变化、用户口碑，最后产出一份 Markdown 格式的报告，发给产品和管理层。

**梳理结果：**

| 维度 | 内容 |
|------|------|
| 触发频率 | 每周 1-2 次 |
| 固定输入 | 竞品名称，可选：时间范围 |
| 固定分析维度 | 功能/价格/口碑/技术架构/市场定位 |
| 固定输出格式 | Markdown 对比表 + 战略建议 |
| 安全级别 | 只读分析，无外部影响 |
| 粒度判断 | 独立完整的工作产出，适中 ✅ |

### 7.2 最终 SKILL.md

```markdown
---
name: competitive-analysis
description: |
  执行竞品分析，生成结构化对比报告。
  触发条件：用户说"分析竞品"、"做竞品对比"、"看看竞品情况"、
  "X和我们比怎么样"、"竞品有什么新动作"等。
  分析维度：功能/价格/用户口碑/技术架构/市场定位。
  输出：Markdown 对比表 + 差异化战略建议。
  不触发：用户询问内部产品功能或要求写代码时。
allowed-tools: Read
metadata:
  version: "1.0.0"
  owner: "市场部"
  last-updated: "2026-03-30"
---

# 竞品分析标准流程

## 触发前提

分析前确认以下信息（如未提供，先询问）：
- 分析对象：哪些竞品？（必须，1-5个）
- 时间范围：最近多久的动态？（默认：最近 3 个月）
- 重点关注：有没有特别关注的维度？（可选）

## 分析维度（全部必须覆盖）

### 1. 核心功能对比
- 列出各竞品的核心功能模块
- 标注我们有/没有的功能
- 标注竞品近期新增/删除的功能

### 2. 价格体系
- 各竞品定价策略（免费/订阅/按量）
- 价格区间和目标客群
- 近期是否有价格调整

### 3. 用户口碑
- 主要平台评价（App Store / 知乎 / 行业论坛）
- 用户主要痛点和好评点
- 净推荐值（NPS）如有公开数据

### 4. 技术架构（如可获取）
- 技术栈关键信息
- 性能/稳定性相关用户反馈

### 5. 市场定位
- 目标客群定位
- 核心差异化卖点
- 近期营销动作

## 数据来源说明

分析时：
1. 优先使用用户提供的资料文件
2. 基于已知公开信息进行分析
3. **数据不足时：明确说明缺少什么，不得编造或猜测**

## 输出格式

### 第一部分：对比总表

| 维度 | 我们的产品 | 竞品 A | 竞品 B |
|------|-----------|--------|--------|
| 核心功能 | ... | ... | ... |
| 价格区间 | ... | ... | ... |
| 目标客群 | ... | ... | ... |
| 核心卖点 | ... | ... | ... |
| 主要弱点 | ... | ... | ... |

### 第二部分：关键发现

列出 3-5 条最重要的发现，格式：
> **发现 N**：[一句话描述] → **影响**：[对我们的影响]

### 第三部分：差异化建议

提供 2-3 条可操作的战略建议：
1. **[建议方向]**：[具体说明] — 优先级：高/中/低
```

## 八、用 Skill-Creator Agent 创建技能：业务人员的人机协作框架

> 本节针对我们内部 AI 工作流平台的使用场景。你不需要会写代码，也不需要理解 YAML 语法——你只需要知道自己的业务流程，然后和 Skill-Creator Agent 对话，它会帮你把流程转化成可复用的 Skill。

### 8.1 整体工作流程

在我们的内部平台上，技能创建的完整流程如下：

```
业务人员
    ↓ 发起对话："我想创建一个 XXX 的技能"

Skill-Creator Agent（技能创建助手）
    ↓
  阶段一：探索访谈
  ├─ 提问：这个流程是什么？你通常怎么做？
  ├─ 提问：什么时候会用到？触发词是什么？
  ├─ 提问：需要什么输入？输出是什么格式？
  └─ 提问：有没有注意事项、特殊规则、例外情况？
    ↓
  阶段二：结构化草稿
  └─ 生成 SKILL.md 草稿（description + 完整正文）
    ↓
  阶段三：人工审核（Human in the Loop ← 关键环节）
  ├─ 业务人员对照真实业务检查草稿
  ├─ 指出不准确、遗漏、表达有误的地方
  └─ Agent 修改 → 再次确认 → 直到通过
    ↓
  阶段四：试跑验证
  └─ 用一个真实案例跑一遍，确认输出符合预期
    ↓
  阶段五：发布上线
  └─ 技术人员或管理员部署到平台 Skill 库

所有同事可以使用这个 Skill
```

**为什么这个流程有效：**
- AI 负责**结构化**（把你说的话转成规范格式）
- 你负责**知识输入**（只有你知道业务真正怎么运作）
- Human in the Loop 负责**质量把关**（防止 AI 理解偏差造成的错误沉淀）

### 8.2 Human in the Loop：为什么人不能离开这个循环

很多人以为"让 AI 自动生成技能"就完事了。这个想法有一个致命缺陷：

**AI 没有你的业务知识。**

Skill-Creator Agent 很擅长把信息整理成结构化格式，但它不知道：
- 你们公司的"竞品分析"和行业通用的有什么不同
- 你们发给客户的邮件有什么特殊的禁忌词
- 某个审批流程里有个"口头规定"，但文档里没写
- 某类异常情况在你们业务里特别常见，需要特殊处理

这些隐性的、经验性的业务知识，只存在于业务人员的脑子里。如果没有 Human in the Loop，这些知识要么丢失，要么变成错误的规则被固化到 Skill 里——而且每次执行都会错。

**Human in the Loop 的三个关键节点：**

```
节点 1：知识输入阶段（提供原材料）
  你的职责：完整、准确地描述流程和规则
  AI 的职责：提问、追问、整理
  风险点：说得太简单 → AI 只能猜 → Skill 不准确

节点 2：草稿审核阶段（质量把关）
  你的职责：像看合同一样认真检查每一条规则
  AI 的职责：解释每条规则的来源，接受修改意见
  风险点：走马观花 → 错误规则上线 → 全团队都在按错误流程执行

节点 3：试跑验证阶段（实际检验）
  你的职责：提供真实案例，判断输出是否达到预期
  AI 的职责：按 Skill 执行，展示输出
  风险点：用过于简单的案例测试 → 上线后遇到复杂场景就出错
```

**一个实际案例说明这个问题：**

> 某运营同学想创建"客户投诉处理邮件"技能。她简单描述说："收到投诉后，道歉，说明处理方案，给补偿。"AI 生成的草稿看起来没问题。但她没有说的是：对 VIP 客户的补偿标准是普通客户的 3 倍，而且绝对不能在邮件里提"赔偿"这个词（有法务风险）。如果没有人工审核，这个 Skill 上线后对 VIP 客户就会给出错误的补偿金额，而且可能触发法律风险。

### 8.3 角色分工：每个人该做什么

| 角色 | 职责 | 关键能力要求 |
|------|------|------------|
| **业务人员**（Skill 发起者） | 提供业务知识、审核草稿、验证输出 | 熟悉自己的业务流程，能识别 AI 理解是否准确 |
| **Skill-Creator Agent** | 访谈、结构化、生成草稿、修改迭代 | — |
| **技术管理员**（可选） | 最终部署上线、权限设置、版本管理 | 了解技术配置（allowed-tools、context 等） |
| **团队负责人**（可选） | 审批高影响力的 Skill（如涉及对外发送） | 业务和风险判断 |

**业务人员不需要具备的能力：**
- 不需要会写代码
- 不需要理解 YAML 格式
- 不需要懂 Claude Code 的技术细节
- 不需要了解 Skill 的底层机制

**业务人员必须具备的能力：**
- 能完整描述自己的工作流程（包括例外情况）
- 能判断 AI 生成的规则是否符合真实业务
- 能提供真实案例用于验证

### 8.4 如何高质量提问：信息提供指南

Skill-Creator Agent 会主动问你问题，但你回答的质量直接决定 Skill 的质量。

**提问前的准备工作（5 分钟）：**

在开始对话之前，建议先快速梳理以下信息：

```
我要创建的技能：[用一句话说清楚]
这个流程我通常做的步骤：[1、2、3...]
最常见的触发场景（我会怎么说）：[示例1、示例2...]
必须要有的输入信息：[...]
输出格式要求：[...]
有没有特殊规则或例外：[...]
```

**回答质量对比：**

Agent 问："这个竞品分析流程通常是怎么做的？"

```
❌ 低质量回答（太简单，AI 只能猜）：
"就是分析一下竞品的情况。"

⚠️ 中等质量回答（有框架，但缺细节）：
"看功能、价格、用户口碑，然后写总结。"

✅ 高质量回答（AI 能直接转化成规则）：
"我通常先看他们最新的功能更新（去官网产品页），然后对比我们
有没有类似功能。价格这块主要看定价页面和促销活动。口碑主要看
知乎和行业论坛。最后写一个对比表，加上3条对我们的启示。
特别注意：我们绝对不能在报告里说'竞品比我们好'，要说'竞品
在某方面有不同的策略'。"
```

**被 Agent 追问时，如何回答：**

Agent 经常会问"这种情况下你会怎么处理"、"有没有例外"。这类问题很关键，不要跳过：

```
Agent："如果竞品没有公开定价，你怎么处理？"

❌ 不好的回答："跳过就好了。"

✅ 好的回答："如果没有公开定价，我会在报告里注明'定价未公开，
据用户反馈约在X-Y元区间'，不能空着不写，因为读报告的人会觉得
这块我们漏查了。"
```

**5 类必须主动告知的信息（即使 Agent 没有问）：**

1. **禁忌项**：有没有不能说的词、不能做的操作？
2. **特殊人群**：对不同类型的对象（VIP 客户/普通客户、新用户/老用户）处理方式是否有差异？
3. **例外场景**：什么情况下流程会不一样？
4. **审批节点**：哪些步骤需要人工确认才能继续？
5. **输出接收方**：这个输出给谁看？他们有什么特殊要求？

### 8.5 交互对话框架：标准协作流程

以下是与 Skill-Creator Agent 协作的**完整对话框架**，分为五个阶段：

---

**阶段一：意图确认（~2 分钟）**

> 目标：让 Agent 明白你要做什么、这个 Skill 的价值是什么

```
你：我想创建一个"客户续费提醒邮件"的技能

Agent 会问：
1. "这个技能要解决什么问题？现在是怎么做的？"
2. "使用频率大概是多少？一天几次还是一周几次？"
3. "谁会用这个技能？只有你，还是整个销售团队？"

你需要回答清楚：
- 现状痛点（每次都要重新想措辞 / 不同人发的格式不一致 / 容易漏关键信息）
- 使用频率（高频 = 值得仔细打磨）
- 使用范围（范围越广 = 越需要严格的 Human in the Loop）
```

---

**阶段二：流程访谈（~10-20 分钟，最核心的阶段）**

> 目标：把你脑子里的业务知识完整转移给 Agent

Agent 会按以下顺序提问，你需要尽量详细地回答：

```
1. 流程框架
   "请描述完整的处理流程，从开始到结束，分步骤说明。"
   → 你：说出每一步你实际会做什么

2. 输入信息
   "执行这个流程需要哪些前置信息？哪些是必须的，哪些是可选的？"
   → 你：列出所有需要用到的数据、文件、背景信息

3. 触发场景
   "你通常在什么情况下做这件事？举3-5个你会怎么说的例子。"
   → 你：说出真实的触发词，越口语化越好

4. 输出规格
   "最终产出是什么？格式是什么？给谁看？"
   → 你：说清楚格式、语气、长度要求

5. 特殊规则
   "有没有特别要注意的、容易出错的、规定死的东西？"
   → 你：这是最重要的地方，把所有潜规则说出来

6. 例外处理
   "遇到XXX情况怎么办？遇到YYY情况怎么办？"
   → 你：把所有"如果...那么..."的规则说清楚
```

**注意：Agent 可能追问 3-5 轮才够用，不要嫌烦。**

---

**阶段三：草稿审核（~15 分钟，最关键的 Human in the Loop 节点）**

> 目标：把 AI 的结构化结果和你的业务知识对齐

Agent 生成 SKILL.md 草稿后，你需要**逐节检查**：

```
检查清单（必须逐项过）：

□ Description 是否准确？
  → 触发词是否覆盖了我常说的话？
  → 有没有误触发的可能？

□ 步骤是否完整？
  → 有没有漏掉某个步骤？
  → 步骤顺序对不对？

□ 规则是否准确？
  → 每条规则是否符合实际业务？
  → 有没有规则写反了或写错了？

□ 禁忌是否已经写入？
  → 我告知的禁忌词是否体现在 Skill 里？
  → 例外处理逻辑是否正确？

□ 输出格式是否对？
  → 格式、语气、长度是否符合要求？
  → 接收方能接受吗？
```

**发现问题时如何反馈：**

```
❌ 模糊反馈（Agent 不知道怎么改）：
"感觉不太对，再改改。"

✅ 精准反馈（Agent 可以直接修改）：
"第3步'根据情况给补偿'太模糊了。
 改成：VIP客户（年消费>10万）补偿当月费用的20%，
       普通客户补偿当月费用的5%，
       补偿上限500元，
       补偿方式只能是账户余额，不能是现金。"
```

---

**阶段四：试跑验证（~10 分钟）**

> 目标：用真实案例验证 Skill 的实际执行质量

```
你准备一个真实案例：
  "这是我上周处理的一个真实客户情况：[具体信息]"

Agent 执行 Skill，给出输出

你检查：
  □ 输出内容是否准确？
  □ 格式是否符合要求？
  □ 有没有我实际上不会这么说的表达？
  □ 如果这封邮件真发出去，会有什么问题吗？

如果不满意：告诉 Agent 哪里不对 → 回到阶段三修改
如果满意：进入阶段五
```

**选择验证案例的原则：**
- 用**真实**的案例，不要用简化版
- 选一个**有点复杂**的案例，覆盖 1-2 个例外场景
- 不要只用最简单的、最标准的情况来测试

---

**阶段五：发布准备**

> 目标：确认 Skill 可以给更多人用了

```
你需要确认：
□ Skill 名称：清晰、规范（动词-对象格式）
□ 所有者：填写部门/姓名，便于后续维护
□ 版本：1.0.0（初始版本）
□ 安全级别确认：
  - 这个 Skill 会不会直接发送邮件/消息？
    → 会：需要设置 disable-model-invocation: true
    → 不会：普通配置即可

提交给技术管理员部署上线
```

### 8.6 一次完整的对话示例

以创建"销售跟进邮件"技能为例：

---

**你**：我想创建一个销售跟进邮件的技能，每次拜访客户后要发跟进邮件，我们整个销售团队都会用。

**Agent**：好的，我来帮你创建这个技能。先了解一下背景：
1. 目前这封邮件通常在拜访后多久内发送？有没有时间规定？
2. 邮件主要目的是什么——确认拜访内容、推进下一步，还是两者都有？
3. 不同阶段的客户（初次接触、意向客户、即将签单）发的内容有区别吗？

**你**：一般要求24小时内发出。目的是确认拜访内容和推进下一步行动。不同阶段的客户内容差很多：
- 初次接触：主要是建立关系，不要太推销
- 意向客户：重点强调我们的差异化优势，推进到下一次会面
- 即将签单：跟进合同进展，解决剩余顾虑

**Agent**：明白了。再问几个细节：
1. 邮件里需要附上什么材料吗（产品手册、方案文档等）？
2. 有没有禁忌——某些词不能用，或者某些话不能说？
3. 你们的邮件有固定签名格式吗？

**你**：一般意向客户和即将签单阶段会附方案文档，但不是每次都有，要根据情况判断。
禁忌：不能说"我们比竞品便宜"，要说"我们在价值方面更有优势"；不能催单，不能说"你们什么时候决定"。
签名有固定格式，我稍后把模板发给你。

**Agent**：收到。关于例外情况：如果拜访不顺利（客户反馈负面）或者客户明确说暂时不考虑，跟进邮件的策略会不一样吗？

**你**：对，如果客户说暂时不考虑，邮件要换成"保持联系"模式，不能提产品，重点是表达理解和留下好印象。这种情况我们叫"冷却期跟进"。

**Agent**：非常有帮助。我现在生成草稿，请你检查…

*[Agent 生成 SKILL.md 草稿]*

**Agent**：草稿已生成，请重点检查以下几点是否符合你们的实际情况：
1. 三种客户阶段的判断标准，你来确认分类是否准确
2. "冷却期跟进"模式的触发条件
3. 关于附件判断的逻辑（我写了一个简单的判断规则，不知道是否符合实际）

**你**：第2条不对，"冷却期跟进"不只是客户说暂时不考虑——如果拜访过程中客户对价格有很大异议也算，这种情况也要用冷却期模式，给他一些消化时间…

*[几轮修改后]*

**你**：好，这个版本对了。用上周跟XX公司的拜访记录跑一下试试。

*[Agent 执行，生成邮件样本]*

**你**：语气还可以，但"期待进一步合作"这句话出现了两次，只留一次。另外我们公司名字后面不用加"团队"，直接写公司名。

*[Agent 修改]*

**你**：好了，这个可以了。技能名字叫 `write-sales-followup`，归属销售部。

**Agent**：已整理好完整的 SKILL.md，这个技能发送的是邮件草稿（还是最终发送？），建议是否需要设置为手动触发…

### 8.7 常见的协作卡点与解法

| 卡点 | 原因 | 解法 |
|------|------|------|
| Agent 生成的草稿"看起来不错但感觉哪里不对" | 缺乏真实案例对照 | 拿一个真实的历史案例，让 Agent 执行草稿，对比实际处理结果 |
| 描述了半天，Agent 还是没理解 | 描述太抽象 | 说"举个例子"，给出一个具体情境，Agent 从具体案例反推规则 |
| 流程太复杂，一个 Skill 装不下 | 粒度太大 | 和 Agent 一起拆分，先识别哪些步骤是独立的工作产出 |
| 审核草稿时不知道从哪里看起 | 缺乏结构 | 先只看 description，确认触发场景对不对；再看步骤数量和顺序；最后看细则 |
| 试跑结果和预期不一致，但说不清哪里不对 | 缺乏对比基准 | 先手动写一份"理想输出"，再让 Agent 执行，逐句对比差异 |
| 技能上线后有人反映结果不对 | 边缘场景未覆盖 | 收集反馈案例，和 Agent 一起分析原因，更新 Skill 的例外处理规则 |

### 8.8 Skill 的生命周期管理

Skill 不是一次性产出，它需要随着业务变化持续维护：

```
创建 → 上线 → 使用 → 反馈 → 更新 → 再上线
  ↑                                        ↓
  └────────── 每季度审查一次是否仍然准确 ────┘
```

**什么时候需要更新 Skill：**
- 业务流程发生了变化（政策调整、产品改版、组织架构变化）
- 团队成员反馈 Skill 执行结果不符合预期
- 发现了新的高频例外场景没有被覆盖
- 禁忌词或规则有更新（如法务要求变化）

**更新时的 Human in the Loop：**
- 小改动（补充例外规则、调整表述）：原 Skill 负责人审核即可
- 大改动（流程框架变化）：重新走完整的创建流程

## 九、一图总结

```
Skills 全局视图：

用户 / Claude
    ↓ 触发（显式 /skill 或自动匹配 description）

SKILL.md
┌─────────────────────────────────────────────┐
│  前置元数据                                   │
│  ├─ name          → 斜杠命令名                │
│  ├─ description   → 触发的判断依据 ⭐最重要    │
│  ├─ allowed-tools → 安全沙箱                  │
│  ├─ disable-model-invocation → 危险操作保护   │
│  └─ context: fork → 隔离执行                 │
├─────────────────────────────────────────────┤
│  正文                                         │
│  ├─ 触发场景示例  → 帮 Claude 判断边界         │
│  ├─ 执行步骤      → 结构化，编号，可检查       │
│  └─ 输出格式      → 明确规定，不留歧义         │
└─────────────────────────────────────────────┘
    ↓
参考资源（按需加载）
├─ references/*.md  → 详细知识库
└─ scripts/*        → 可执行脚本
```

**写好 Skill 的核心原则，只有一句话：**

> **把一个有经验的人处理这件事的完整思路，用结构化的方式写下来，让 Claude 每次都能按这个标准执行。**
