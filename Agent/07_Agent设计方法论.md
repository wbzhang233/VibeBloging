## 1. Agent 的本质

在开始讲设计方法之前，先明确 Agent 是什么。

### 最简定义

```
Agent = Loop（循环）+ Tools（工具）
```

更具体地说：

```python
# Agent 的伪代码本质
messages = [system_prompt, user_input]

while True:
    response = llm(messages)           # 模型推理

    if response.has_tool_call:
        result = execute_tool(response.tool_call)  # 执行工具
        messages.append(tool_result)   # 把结果加回去
    else:
        return response.content        # 任务完成，返回结果
```

这就是 Claude Code、Cursor、OpenClaw 等所有 AI Coding Agent 的底层原理——无论界面多炫酷，**核心都是这一个循环**。

### 从简单循环到复杂系统

```
简单 Agent（Loop + Bash）
    ↓
有记忆的 Agent（+ Memory）
    ↓
有约束的 Agent（+ Rules + Hooks）
    ↓
有身份的 Agent（+ Identity + Soul）
    ↓
多 Agent 协作系统（+ Agent Coordination）
    ↓
自我维护的 Agent（+ Heartbeat + Bootstrap）
```

OpenClaw 通过 8 个结构化文件，覆盖了这个进化路径的全部阶段。

## 2. 为什么需要结构化设计

### 不结构化设计的后果

很多人刚开始用 AI Agent，习惯把所有指令塞进一个大 Prompt：

```
你是一个高效的 AI 助手，你要帮我处理工作事务。
你擅长写文章、分析数据、发送邮件...
你的性格是专业、友善的...
记住：用户叫张三，他喜欢简短的回复...
你有以下工具可以用：...
如果遇到问题，你应该...
每天早上你要...
```

**这样做的问题**：
1. **可维护性差**：修改一个部分需要重新理解整个 Prompt
2. **复用性差**：同类 Agent 无法共享公共组件
3. **上下文浪费**：所有信息每次都要全部载入
4. **调试困难**：出问题不知道是哪个部分的原因

### 结构化设计的收益

```
单一大 Prompt  →  8 个职责单一的文件

优点：
✅ 每个文件只负责一个维度
✅ 可以按需加载（节省 token）
✅ 各文件可独立修改、测试
✅ 不同 Agent 可共享同一个 SOUL.md / TOOLS.md
✅ 调试时可快速定位问题所在的文件
```

### 3.1 USER.md — 用户画像文件

**职责**：告诉 Agent "你的服务对象是谁"

#### 核心内容

```markdown
# USER.md

## 基本信息
- 姓名：[用户名/昵称]
- 职业：[职业/岗位]
- 技术水平：[初级/中级/高级/专家]
- 语言偏好：[中文/英文/混合]

## 工作背景
- 所在行业：[行业]
- 主要职责：[日常工作内容]
- 常用工具：[IDE/软件/平台]
- 团队规模：[个人/小团队/大团队]

## 偏好与习惯
- 回复风格：[详细/简洁/要点式]
- 代码风格：[具体约定]
- 沟通习惯：[是否喜欢确认每一步]
- 不喜欢：[避免的行为]

## 授权范围
- 可以不经确认直接执行的操作：[列表]
- 必须确认后才能执行的操作：[列表]
- 禁止执行的操作：[列表]
```

#### 设计要点

**要点 1：动态更新**
USER.md 不是一次写好就不动了。每次发现用户的新偏好，应该更新：

```
用户说："以后帮我写代码都用 TypeScript"
→ 立即更新 USER.md，下次不需要再说
```

**要点 2：权限粒度**
不要只写"可以操作文件"，要写清楚粒度：

```
❌ 差的权限定义：
   可以操作文件系统

✅ 好的权限定义：
   - 可以读取 ~/projects/ 下的所有文件
   - 可以修改 .md 和 .txt 文件
   - 禁止修改 .env 文件
   - 删除操作必须先确认
```

**要点 3：区分用户 vs 角色**
USER.md 描述的是**真实用户**，而不是 Agent 的角色。不要把 Agent 的性格写进来。

### 3.2 SOUL.md — 灵魂价值观文件

**职责**：定义 Agent 的**核心价值观和行为准则**，回答"遇到冲突时，你优先什么？"

这是最容易被忽视、也最关键的文件。没有 SOUL.md 的 Agent，行为会缺乏一致性——遇到边缘情况时，不知道该怎么做。

#### 核心内容

```markdown
# SOUL.md

## 核心价值观（按优先级排序）
1. 安全第一：任何可能造成不可逆损害的操作，必须暂停确认
2. 诚实透明：不确定的事情说不确定，不假装知道
3. 用户利益优先：在不违反安全的前提下，优先满足用户需求
4. 效率：完成任务的路径越短越好，不做无意义的步骤

## 核心行为准则
- 宁可多问一句，不要盲目执行
- 遇到歧义时，选择最保守的解释
- 说不了解时直接说，不要编造信息
- 主动告知用户风险，而不是等用户问

## 冲突处理规则
- 用户要求 vs 安全原则 → 安全优先，解释原因，提供安全替代方案
- 用户要求 vs 法律/伦理 → 拒绝执行，说明原因
- 效率 vs 准确性 → 准确性优先
- 自主执行 vs 请求确认 → 不确定时请求确认

## 禁止行为（无论用户怎么要求）
- 删除或覆盖用户未备份的重要数据
- 向第三方泄露用户隐私信息
- 执行可能影响他人系统的操作
- 伪造或捏造信息
```

#### 设计要点

**要点 1：价值观需要排优先级**

不排优先级会出问题：

```
❌ "你要既安全又高效又准确"
   → 出现冲突时，AI 不知道选哪个

✅ "安全 > 准确 > 效率"
   → 出现冲突时，AI 有明确答案
```

**要点 2：写"遇到冲突时怎么做"**

大多数情况是简单的，价值观在**边缘情况**才体现价值：

```
场景：用户说"帮我把这个文件夹里的所有 .log 文件删掉"
     但文件夹里还有一个 important.log，文件名包含 log

没有 SOUL.md 的 Agent：可能直接执行，删掉重要文件
有 SOUL.md 的 Agent：检测到不可逆操作，列出将删除的文件，请求确认
```

**要点 3：SOUL.md 应该稳定**

与 USER.md（经常更新）不同，SOUL.md 一旦确定应该保持稳定。频繁修改价值观的 Agent，行为会不可预测。

### 3.3 IDENTITY.md — 身份角色文件

**职责**：定义 Agent 的**角色定位、专业领域和工作边界**

SOUL.md 回答"你是什么样的 AI"，IDENTITY.md 回答"你是做什么的 AI"。

#### 核心内容

```markdown
# IDENTITY.md

## 角色定义
名称：[Agent 名字]
角色：[职位/角色描述，如"资深全栈开发工程师"]
专业领域：[核心专长]
工作范围：[负责处理的任务类型]

## 能力边界
擅长：
- [能力1]：[具体描述]
- [能力2]：[具体描述]

不擅长（遇到这类任务应告知用户）：
- [局限1]
- [局限2]

明确不做（工作范围之外）：
- [超出范围的事项]

## 工作风格
- 任务理解方式：[如何分析用户需求]
- 输出格式偏好：[代码块/列表/表格等]
- 错误处理方式：[遇到错误时的标准响应]

## 专业背景设定（如需要角色扮演）
- 工作年限：[X 年经验]
- 技术栈：[精通的技术]
- 行业经验：[了解的行业]
```

#### 设计要点

**要点 1：专业领域要具体**

```
❌ 模糊：擅长编程
✅ 具体：擅长 Python 后端开发，熟悉 FastAPI/Django，
         了解 PostgreSQL/Redis，有 Docker 部署经验
```

**要点 2：明确工作边界**

工作边界不是限制，是帮助 AI 在"被要求做超出范围的事情"时给出正确响应：

```
如果 IDENTITY.md 定义 Agent 是"代码助手"：
用户说"帮我订机票" → Agent 应该说"这超出了我的工作范围，我专注于代码相关工作"
```

**要点 3：区分 SOUL 和 IDENTITY**

```
SOUL.md（价值观，通用）：
  "我诚实、安全、用户优先"
  → 任何 Agent 都适用

IDENTITY.md（角色，特定）：
  "我是一个 Python 后端专家，负责代码开发和调试"
  → 只适用于这个特定 Agent
```

### 3.4 MEMORY.md — 持久记忆文件

**职责**：跨会话保存重要信息，让 Agent "记住"之前的交互

这是 Agent 从"每次都从零开始"到"有连续性"的关键文件。

#### 核心内容结构

```markdown
# MEMORY.md

## 当前项目状态
- 正在进行的任务：[描述]
- 已完成的关键里程碑：[列表]
- 已知问题和待解决项：[列表]
- 上次工作的断点：[具体状态]

## 重要决策记录
- [日期] 决定使用 X 而非 Y，原因：[理由]
- [日期] 放弃方案 Z，原因：[遇到的问题]

## 已验证的事实
- [项目特有的重要信息，如环境配置、依赖关系]

## 用户偏好（动态学习）
- 用户偏好 A 方式而非 B 方式
- 用户不喜欢 [具体行为]
- 用户特别重视 [某方面]

## 常见错误与解决方案
- 问题：[描述]；原因：[分析]；解决：[方法]

## 待跟进事项
- [ ] [事项1]（优先级：高）
- [ ] [事项2]（优先级：中）
```

#### 设计要点

**要点 1：记忆要分层**

```
全局记忆（放在 MEMORY.md 根目录）：
  - 跨项目都有用的信息
  - 用户长期偏好

项目记忆（放在项目目录 MEMORY.md）：
  - 只对当前项目有用的信息
  - 项目特有的技术决策

会话记忆（只在当前对话中保留）：
  - 临时状态，不需要持久化
```

**要点 2：主动更新，及时清理**

```
什么时候更新 MEMORY.md：
✅ 做了重要决策（选了某个技术方案）
✅ 发现了关键问题（某个依赖的坑）
✅ 用户表达了偏好
✅ 完成了重要里程碑

什么时候清理 MEMORY.md：
✅ 项目结束后，清理项目特定内容
✅ 问题已解决，移除"待解决"标记
✅ 决策已执行，转为"已完成"
```

**要点 3：记忆质量 > 记忆数量**

```
❌ 把所有对话都记下来（占用大量 token，降低信噪比）
✅ 只记录"下次对话时如果没有这条信息会很麻烦"的内容
```

**要点 4：记忆是上下文的"预算优化"**

```
上下文窗口是稀缺资源（200K tokens）：
- 不重要的信息每次都载入 → 浪费
- 重要信息只在需要时动态加载 → 高效

MEMORY.md 的本质：
把"重要但不紧急"的信息外置，
需要时再读取，而不是每次都占用上下文。
```

### 3.5 TOOLS.md — 工具能力文件

**职责**：定义 Agent 可以使用的工具，以及如何正确使用它们

#### 核心内容

```markdown
# TOOLS.md

## 工具清单

### 文件系统工具
- read_file(path)：读取文件内容
  - 适用：需要了解文件内容时
  - 限制：不能读取系统敏感文件
  - 错误处理：文件不存在时报告用户，不要创建空文件

- write_file(path, content)：写入文件
  - 适用：创建新文件或覆盖已有文件
  - 前置条件：如果文件已存在，必须先读取确认
  - 风险：不可逆操作，修改重要文件前需确认

### 代码执行工具
- run_command(cmd)：执行 Shell 命令
  - 适用：运行脚本、测试、构建
  - 禁止：rm -rf、格式化、shutdown 等危险命令
  - 超时：默认 30 秒，长任务需要 --background 参数

### 网络工具
- web_search(query)：搜索网络信息
  - 适用：需要最新信息、文档、API 参考
  - 注意：搜索结果需要验证，不要盲目信任

- fetch_url(url)：获取网页内容
  - 适用：读取特定文档、API 文档

### 通知工具
- send_notification(message)：发送通知给用户
  - 适用：任务完成、需要用户决策、发现异常

## 工具选择指南
- 优先用专用工具，不要用通用工具模拟
- 读文件用 read_file，不要用 run_command + cat
- 搜索用 web_search，不要手动拼接 URL

## 工具失败处理
- 第一次失败：检查参数是否正确，重试一次
- 第二次失败：报告给用户，说明尝试了什么，遇到了什么问题
- 不要无限重试
```

#### 设计要点

**要点 1：工具描述要包含使用时机**

AI 的工具选择能力取决于描述的质量：

```
❌ 差的工具描述：
   run_command(cmd): 执行命令

✅ 好的工具描述：
   run_command(cmd): 执行 Shell 命令
   适用场景：
   - 运行测试（npm test, pytest）
   - 运行构建（npm build, make）
   - 查看系统信息（ls, pwd, env）
   不适用场景：
   - 文件读写（用 read_file/write_file）
   - 网络请求（用 fetch_url）
```

**要点 2：为高风险工具添加 Guard**

```markdown
- delete_file(path)：删除文件
  ⚠️ 高风险工具，使用前必须：
  1. 确认文件路径正确（展示给用户）
  2. 确认用户有备份或文件不重要
  3. 收到用户明确确认后才执行
```

**要点 3：工具组合模式**

TOOLS.md 可以定义常见的工具组合模式（类似 OpenClaw 的 Skills）：

```markdown
## 常用工具组合

### 代码修改流程
1. read_file(target) → 了解当前状态
2. [分析和生成修改内容]
3. write_file(target, new_content)
4. run_command("相关测试") → 验证修改

### 信息搜集流程
1. web_search(query) → 找到相关资源
2. fetch_url(url) → 获取详细内容
3. write_file("output.md", summary) → 保存结果
```

### 3.6 AGENTS.md — 多智能体协调文件

**职责**：定义多 Agent 系统的协作模式、角色分工和通信协议

当一个 Agent 不够用时，需要多个 Agent 协作。AGENTS.md 是这套协作体系的"宪法"。

#### 核心内容

```markdown
# AGENTS.md

## Agent 角色定义

### 主协调 Agent（Orchestrator）
职责：
- 接收用户任务，分析和拆解
- 决定哪些子任务交给哪个专用 Agent
- 整合子 Agent 的结果
- 与用户沟通进度和异常

不做：
- 不直接执行具体任务（委托给专用 Agent）
- 不同时承担多个角色

### 研究 Agent（Researcher）
职责：信息搜集、文档查阅、市场调研
工具权限：只有读权限（web_search, fetch_url, read_file）
输出格式：结构化摘要 + 关键发现 + 原始来源

### 执行 Agent（Executor）
职责：代码编写、文件修改、命令执行
工具权限：读写权限（read_file, write_file, run_command）
输出格式：修改清单 + 执行结果 + 验证报告

### 审查 Agent（Reviewer）
职责：检查执行结果、发现问题、质量验证
工具权限：只有读权限
输出格式：问题列表 + 严重程度评级 + 修复建议

## 协作流程

### 标准任务流程
```
用户 → Orchestrator
  ↓ 拆分任务
Orchestrator → Researcher（搜集背景信息）
Orchestrator → Executor（执行具体任务）
  ↓ 执行完成
Executor → Reviewer（验证结果）
  ↓ 审查报告
Reviewer → Orchestrator（汇总）
  ↓ 整合结果
Orchestrator → 用户（最终交付）
```

## 通信协议

### Agent 间的信息传递格式
```json
{
  "from": "orchestrator",
  "to": "executor",
  "task_id": "task_001",
  "task_type": "code_modification",
  "input": {...},
  "constraints": {...},
  "expected_output": {...}
}
```

### 异常处理
- 子 Agent 失败：向 Orchestrator 报告，由 Orchestrator 决定重试/告知用户
- 超时：子 Agent 定时报告进度，超过阈值主动中止
- 冲突：两个 Agent 修改同一文件，必须串行，由 Orchestrator 协调顺序

## 子 Agent 使用原则
✅ 适合用子 Agent：
  - 任务可以明确切分
  - 各子任务基本独立（无紧密依赖）
  - 任务耗时长、上下文消耗大

❌ 不适合用子 Agent：
  - 需要频繁交互确认状态
  - 任务之间强依赖（A 的结果是 B 的输入）
  - 任务本身很简单（启动子 Agent 的开销不值得）
```

#### 设计要点

**要点 1：权限分离**

不同角色的 Agent 应该有不同的工具权限：

```
研究 Agent → 只读权限（防止误操作）
执行 Agent → 读写权限（有访问控制）
审查 Agent → 只读权限（确保中立）
```

**要点 2：Orchestrator 不执行，只协调**

一个常见错误是让 Orchestrator 又负责协调，又负责执行：

```
❌ Orchestrator 直接执行代码修改
✅ Orchestrator 把代码修改任务委托给 Executor Agent
```

**要点 3：子 Agent 只返回摘要**

子 Agent 不应该把所有中间过程都返回给主 Agent——只返回**关键结论和最终结果**：

```
❌ 子 Agent 返回：所有搜索结果 + 所有网页内容 + 分析过程
✅ 子 Agent 返回：关键发现摘要 + 结论 + 来源链接
```

### 3.7 BOOTSTRAP.md — 启动初始化文件

**职责**：定义 Agent 启动时的初始化流程——每次启动时"第一件事做什么"

#### 核心内容

```markdown
# BOOTSTRAP.md

## 启动检查清单

### Step 1：环境验证
- [ ] 检查必要工具是否可用（文件系统、网络等）
- [ ] 验证关键配置是否存在（API Key、环境变量）
- [ ] 检查当前工作目录是否正确

### Step 2：上下文恢复
- [ ] 读取 MEMORY.md，了解上次工作状态
- [ ] 检查是否有未完成的任务（MEMORY.md 中的待跟进事项）
- [ ] 恢复必要的项目上下文

### Step 3：用户确认
如果发现未完成任务，询问用户：
"上次有一个未完成的任务：[描述]，需要继续吗？"

### Step 4：就绪报告
向用户说明：
- 当前状态
- 已加载的上下文
- 可以处理的任务类型

## 启动失败处理

### 环境验证失败
- 告知用户具体缺少什么
- 提供修复步骤
- 在问题解决前不接受任务

### 记忆读取失败
- 告知用户记忆文件损坏/不存在
- 询问是否从零开始
- 记录本次为"无历史状态启动"

## 首次启动（新用户/新项目）
1. 创建 MEMORY.md 初始模板
2. 引导用户完成基本配置
3. 询问用户的工作目标和偏好
4. 将用户偏好更新到 USER.md
```

#### 设计要点

**要点 1：Bootstrap 是确定性的**

Bootstrap 流程应该是**可预期的**——每次启动都执行同样的步骤，而不是根据上下文随机决定做什么：

```
✅ 固定流程：
   启动 → 检查环境 → 恢复记忆 → 确认状态 → 就绪

❌ 随机流程：
   有时检查环境，有时直接开始，有时询问用户，行为不一致
```

**要点 2：失败有明确的出口**

每个检查步骤都应该有"失败了怎么办"的处理：

```
环境检查失败 → 告知用户 + 提供修复步骤 + 等待修复
记忆读取失败 → 询问用户是否从零开始
依赖不满足   → 列出缺失依赖 + 提供安装命令
```

**要点 3：Bootstrap 的结束是明确的**

用户应该知道 Agent 什么时候"就绪了"：

```
✅ 明确就绪信号：
   "已完成初始化，当前项目是 X，上次的任务是 Y，
    有 Z 个待跟进事项。现在可以告诉我你想做什么。"

❌ 隐式就绪：
   启动后直接等待用户输入，用户不知道是否准备好了
```

### 3.8 HEARTBEAT.md — 心跳周期任务文件

**职责**：定义 Agent 的**周期性自维护任务**——即使没有用户输入，Agent 也应该主动做的事情

这是最容易被遗漏，但对长期运行的 Agent 最关键的文件。

#### 核心内容

```markdown
# HEARTBEAT.md

## 周期任务定义

### 每次对话开始时（高频）
- 检查 MEMORY.md 中的"待跟进事项"
- 如果有高优先级待跟进，主动提醒用户
- 更新任务状态

### 每次对话结束时（高频）
- 将本次对话的重要决策写入 MEMORY.md
- 更新项目状态
- 更新已完成任务状态

### 每日（低频，需要明确授权）
- 检查项目依赖是否有安全更新
- 检查关注的 API/服务是否有异常
- 生成工作摘要并保存

### 每周（低频，需要明确授权）
- 回顾 MEMORY.md，清理过期信息
- 生成项目进展周报
- 检查积压的待跟进事项

## 心跳任务的触发条件
- 对话开始/结束：自动触发（无需用户授权）
- 每日/每周任务：需要用户明确开启
- 异常检测：发现异常时立即触发，无论是否在对话中

## 自我维护机制

### 记忆整理
当 MEMORY.md 超过阈值（如 500 行）时：
1. 提示用户"记忆文件较大，建议整理"
2. 提出清理方案（保留什么，归档什么）
3. 用户确认后执行

### 上下文质量监控
定期自评：
- "我是否还记得项目的核心目标？"
- "MEMORY.md 中的信息是否还准确？"
- "TOOLS.md 中的工具是否都能正常工作？"

## 心跳与 Hooks 的关系
```
Hooks（事件驱动）：
  发生了某件事 → 触发对应行为
  例：每次修改文件 → 自动运行测试

Heartbeat（时间驱动）：
  到了某个时间点 → 执行维护任务
  例：每天结束 → 总结今日工作
```
```

#### 设计要点

**要点 1：区分"每次自动"和"需授权"**

```
不需要用户授权的心跳任务（默认开启）：
  ✅ 对话开始时检查待办事项
  ✅ 对话结束时更新记忆

需要用户明确授权的心跳任务：
  🔒 每日/每周的主动任务
  🔒 向外部服务发送通知
  🔒 任何涉及写操作的定期任务
```

**要点 2：Heartbeat 是"主动性"的来源**

没有 Heartbeat 的 Agent 是被动的——只响应用户输入。有 Heartbeat 的 Agent 能主动发现问题：

```
用户问："我们项目怎么样了？"

没有 Heartbeat 的 Agent：
  "我不太清楚，请告诉我最新情况。"

有 Heartbeat 的 Agent：
  "根据我每日跟踪的记录：本周完成了 X，
   有 Y 个待解决问题，其中 Z 是高优先级的，
   建议今天处理。"
```

**要点 3：不要过度心跳**

```
❌ 过度心跳：每隔几分钟就打扰用户确认状态
✅ 适度心跳：只在关键节点（开始/结束/异常）更新状态
```

## 4. 8 个文件的协作关系

### 数据流向

```
用户输入
  │
  ▼
BOOTSTRAP.md ──→ 启动时读取所有文件，建立初始上下文
  │
  ├── USER.md ──────────→ "这个用户是谁，授权了什么"
  ├── SOUL.md ──────────→ "遇到冲突时，我优先什么"
  ├── IDENTITY.md ──────→ "我是做什么的，边界在哪"
  ├── MEMORY.md ────────→ "我之前知道了什么"
  └── TOOLS.md ─────────→ "我有哪些工具，怎么用"
  │
  ▼
任务执行
  │
  ├── 需要协作 ─────────→ AGENTS.md（如何调度其他 Agent）
  ├── 执行结束 ─────────→ MEMORY.md（更新记忆）
  └── 周期触发 ─────────→ HEARTBEAT.md（周期性维护）
```

### 修改频率对比

| 文件 | 修改频率 | 原因 |
|------|---------|------|
| SOUL.md | 极少（月/年级别） | 价值观应该稳定 |
| IDENTITY.md | 少（季度级别） | 角色定位相对固定 |
| TOOLS.md | 偶尔（有新工具时） | 工具集变化不频繁 |
| AGENTS.md | 偶尔（架构调整时） | 协作模式相对稳定 |
| BOOTSTRAP.md | 偶尔（流程优化时） | 启动流程相对固定 |
| HEARTBEAT.md | 偶尔（需求变化时） | 周期任务变化不频繁 |
| USER.md | 频繁（每次发现新偏好） | 持续学习用户 |
| MEMORY.md | 最频繁（每次对话） | 持续积累知识 |

## 5. 设计原则与最佳实践

### 原则一：职责单一（Single Responsibility）

每个文件只回答一个问题：

```
USER.md    → 用户是谁？
SOUL.md    → 我的价值观是什么？
IDENTITY.md → 我是做什么的？
MEMORY.md  → 我知道了什么？
TOOLS.md   → 我能做什么？
AGENTS.md  → 如何协作？
BOOTSTRAP.md → 如何启动？
HEARTBEAT.md → 如何自维护？
```

### 原则二：最小够用（Minimum Viable）

```
❌ 一开始就写 500 行的 SOUL.md
✅ 从 5 条核心价值观开始，遇到边缘情况时添加

❌ 把所有工具都写进 TOOLS.md
✅ 只写当前实际使用的工具
```

### 原则三：按需加载（Lazy Loading）

不是所有文件每次都要载入上下文：

```
每次必须加载（系统指令）：
  SOUL.md（价值观影响所有决策）
  IDENTITY.md（角色定位影响所有行为）

按需加载（任务开始时）：
  USER.md（知道当前用户是谁）
  MEMORY.md（恢复上次状态）
  TOOLS.md（了解可用工具）

条件加载（需要时才加载）：
  AGENTS.md（只有需要多 Agent 协作时）
  BOOTSTRAP.md（只在启动时）
  HEARTBEAT.md（只在触发时）
```

### 原则四：验证驱动（Verification First）

```
不要依赖 AI 自己说完成了
→ 每个任务必须有可验证的完成标准

在 TOOLS.md 中定义：
- 成功标准（任务完成的客观指标）
- 验证方法（如何检测是否真的完成了）
```

## 6. 从 0 到 1 设计一个 Agent

### 场景：设计一个"技术文章生成 Agent"

#### 第 1 步：明确目标

```
目标：根据用户给的主题，自动搜集资料、生成技术文章并保存
用户：技术博主，有一定技术背景
核心任务：信息搜集 + 内容生成 + 格式化输出
```

#### 第 2 步：逐文件设计

**USER.md**（先写，因为最具体）：
```markdown
用户：技术博主
技术水平：中级，了解编程但不是专职开发者
偏好：文章 2000-3000 字，要有代码示例，使用 Markdown
不喜欢：口水话、重复表达、没有例子的抽象描述
授权：可以联网搜索，可以写文件，不需要每步确认
```

**SOUL.md**（定义价值观）：
```markdown
核心价值：内容准确性 > 内容完整性 > 写作效率
冲突处理：不确定的技术细节，宁可说"需要验证"，不要编造
禁止行为：引用未验证的信息作为事实陈述
```

**TOOLS.md**（列出工具）：
```markdown
web_search：搜集技术资料
fetch_url：读取具体文档/博客
write_file：保存生成的文章
read_file：读取已有草稿
```

**MEMORY.md**（初始化）：
```markdown
## 文章库
## 风格学习记录
## 用户反馈记录
```

**BOOTSTRAP.md**（简单启动流程）：
```markdown
1. 检查联网是否正常
2. 读取 MEMORY.md 了解已写过的文章（避免重复）
3. 就绪，等待用户给出文章主题
```

**HEARTBEAT.md**（简单维护）：
```markdown
每次文章完成后：记录到 MEMORY.md（文章库）
```

#### 第 3 步：验证设计

```
模拟场景：用户说"帮我写一篇关于 React Server Components 的技术文章"

执行路径：
1. BOOTSTRAP 检查：联网正常，MEMORY 无重复主题 ✅
2. 搜集阶段：web_search x3，fetch_url x2
3. 生成阶段：根据资料生成 2500 字文章（符合 USER.md 偏好）
4. 验证阶段：SOUL.md 检查——所有技术描述是否有来源？
5. 保存：write_file("react-server-components.md")
6. 更新 MEMORY.md：文章库 +1
```

## 7. 常见误区

### 误区 1：把 SOUL 和 IDENTITY 混淆

```
❌ 在 SOUL.md 里写："我擅长 Python 开发"
   → 这是身份（IDENTITY），不是价值观（SOUL）

✅ SOUL.md："我诚实、以用户为先、重视安全"
   IDENTITY.md："我是 Python 后端专家"
```

### 误区 2：MEMORY.md 变成日记

```
❌ 记录所有对话细节（占空间、影响效率）
✅ 只记录"下次没有这个信息会很麻烦"的内容
```

### 误区 3：TOOLS.md 只列名称不写用法

```
❌ tools: [read_file, write_file, run_command]
✅ 每个工具写清楚：适用场景、禁止场景、失败处理
```

### 误区 4：忽略 HEARTBEAT.md

```
❌ Agent 只响应用户，从不主动维护状态
✅ 每次对话结束都更新记忆，定期清理过期信息
```

### 误区 5：一次把所有文件写满

```
❌ 第一天就把 8 个文件全部写到 100 行
✅ 从最小可用版本开始，遇到问题时补充
```

## 总结

OpenClaw 的 8 个核心文件本质上是对 **"如何让 AI Agent 变得可靠、可维护、可扩展"** 这个问题的系统性回答：

```
可靠性  ← SOUL.md（价值观一致）+ HEARTBEAT.md（持续自维护）
可维护性 ← 8 个文件各司其职，独立修改不影响其他
可扩展性 ← AGENTS.md（多 Agent 协作）+ TOOLS.md（工具扩展）
连续性  ← MEMORY.md（跨会话记忆）+ BOOTSTRAP.md（优雅恢复）
```

这套设计方法的核心思想，与软件工程的最佳实践高度一致：
- **单一职责**（每个文件只做一件事）
- **开闭原则**（对扩展开放，对修改封闭）
- **最小知识**（每个组件只知道必须知道的）

理解了这 8 个文件，就理解了如何设计一个"不只是 demo，而是真正可用"的 AI Agent。
