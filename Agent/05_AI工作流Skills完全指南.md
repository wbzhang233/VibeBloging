# AI 工作流 Skills 完全指南：从原理到业务实践

> 作者：内部技术分享
> 时间：2026-03-30
> 受众：产品 + 研发混合，重点兼顾业务人员

---

## 写在前面

你有没有遇到过这样的情况：

> 同样的任务，每次都要重新给 AI 描述一遍背景、要求和格式。写到第十次，你开始怀疑 AI 能不能记住"我们公司的竞品分析就是这么做的"。

这个问题的根本原因不是 AI 不够聪明，而是**你的业务知识没有被妥善封装**。

这就是 **Skills** 要解决的问题。

Skill（技能）是现代 AI 工作流平台中的核心概念之一。它不是一个特定平台的专有功能，而是一种**把可复用业务流程注入 AI 执行上下文**的通用设计模式。理解它，就是理解如何让 AI 真正"懂"你的业务，而不是每次都从零开始。

本文从原理讲起，覆盖创建方法、写作范式、常见误区，重点面向业务人员，讲清楚如何把日常工作流程系统化地"Skill 化"。

---

## 一、什么是 Skill

### 1.1 一句话定义

**Skill 是一个可复用的、带上下文注入能力的任务执行模板。**

它的本质是：当 AI Agent 判断当前任务与某个 Skill 相关时，自动把该 Skill 的完整指令加载进对话上下文，并按照这些指令完成任务。

### 1.2 从用户视角看

你在 AI 工作流平台里输入 `/竞品分析`，或者直接说"帮我做一次竞品分析"：

```
方式一（显式调用）：
  用户 → /竞品分析
  ↓
  AI 加载竞品分析 Skill 的完整指令
  ↓
  按照预设的分析框架执行任务

方式二（自动触发）：
  用户 → "帮我看看这几个竞品最近有什么新动作"
  ↓
  AI 扫描描述，匹配到竞品分析 Skill
  ↓
  自动加载并执行
```

两种方式都能触发同一套预设流程，输出结果一致、可预期。

### 1.3 Skill 与相关概念的对比

在 AI 工作流体系中，Skill 经常和其他几个概念混淆，一张表说清楚：

| 维度 | Skill（技能） | 工具调用（Tool Use） | 自动化规则（Hook） | 上下文记忆（Memory） |
|------|-------------|-------------------|-----------------|-------------------|
| **本质** | 任务指令模板 | 外部能力接入 | 生命周期钩子 | 持久化记忆 |
| **触发方式** | 用户显式或 AI 自动匹配 | AI 判断是否调用 | 系统事件自动触发 | AI 随时参考 |
| **存储形式** | 结构化文本文件 / 平台配置 | 独立服务 / API 接口 | 事件监听配置 | 向量数据库 / 文件 |
| **做什么** | 注入操作流程和知识 | 调用外部 API / 执行代码 | 在特定事件前后执行逻辑 | 记录跨会话的偏好和状态 |
| **典型用途** | 竞品分析流程、周报模板 | 查询数据库、发送邮件 | 提交前自动校验 | 记住用户偏好、项目约定 |

**记忆口诀：**
- Skill → **我该怎么做这件事**（流程和知识）
- 工具调用 → **我去哪里取数据 / 执行操作**（外部能力）
- 自动化规则 → **每次做 X 之前/之后强制做 Y**（自动拦截）
- 上下文记忆 → **我记得用户上次说过…**（跨会话记忆）

## 二、Skill 的工作原理

### 2.1 三层延迟加载机制

Skill 不会把所有内容一次性塞进 AI 的上下文窗口。主流平台的设计都采用**三层延迟加载**策略，在节省计算资源的同时保证相关内容按需注入：

```
第一层：元数据（始终加载）
  ├─ 技能名称："竞品分析"
  ├─ 触发描述："当用户要求分析竞品时执行..."
  └─ 常驻 AI 的系统提示中（占用极少的上下文空间）

第二层：指令体（触发时加载）
  ├─ 技能定义文件的完整正文
  ├─ 详细的操作步骤和规范
  └─ 触发后注入对话上下文

第三层：引用资源（按需加载）
  ├─ 附属知识库文档
  ├─ 辅助脚本
  └─ AI 执行过程中按需读取
```

这个设计的好处：
- 可以注册几十个 Skill，不会因为元数据太多影响基础对话性能
- 只有真正被调用的 Skill，它的完整内容才会进入上下文
- 大型知识库可以拆成多个引用文件，按需加载

### 2.2 触发机制：语义匹配，而非关键词匹配

**Skill 的触发完全由语言模型的语义理解来判断，没有硬编码的关键词匹配。**

AI 在处理每个请求时：

1. 读取所有已注册 Skill 的名称和触发描述
2. 理解用户的意图
3. 判断哪个 Skill（如果有）与当前意图最匹配
4. 加载该 Skill 的完整指令体

这意味着：
- ✅ 用户说"帮我分析一下竞品"→ 触发竞品分析 Skill
- ✅ 用户说"看看市场上有哪些对手，他们最近有什么动作"→ 也能触发
- ✅ 用户说"X 和我们比怎么样"→ 也能触发
- ❌ 用户说"帮我写一封销售邮件"→ 不会误触发竞品分析

**触发质量完全取决于触发描述写得好不好。** 这是 Skill 设计中最核心的决策点。

### 2.3 上下文注入的完整流程

```
用户输入："帮我做一次竞品分析"
    ↓
AI 扫描可用 Skill 列表（元数据）
    ↓
匹配到竞品分析 Skill（触发描述相关）
    ↓
检查：是否设置了"仅手动触发"？
  → 是：提示用户手动调用，不自动执行
  → 否：继续
    ↓
加载技能定义文件完整内容
    ↓
更新工具权限（如有限制）
    ↓
在隔离上下文或当前上下文中执行
    ↓
按技能指令逐步完成任务
```

### 2.4 隔离执行模式

部分平台支持将 Skill 在**独立的子 Agent 中**执行，与主对话上下文完全隔离：

```
主对话上下文
  ↓
[隔离] 创建子 Agent
  ├─ 独立上下文（看不到主对话历史）
  ├─ 受限的工具权限
  ├─ 执行完整 Skill 流程
  └─ 返回最终结果给主对话

主对话继续（不受子 Agent 执行过程影响）
```

适合用于：
- 危险操作（部署、数据库变更），隔离风险
- 长流程任务，不污染主对话上下文
- 需要专注执行、不受外部干扰的场景

## 三、Skill 的结构：一份技能定义文件包含什么

无论具体平台的实现方式如何，一个完整的 Skill 在概念上都由相同的几个要素构成。

### 3.1 Skill 的核心构成

```
一个 Skill 的完整定义
┌─────────────────────────────────────────────┐
│  元数据配置                                   │
│  ├─ 技能名称     → 唯一标识符，也是调用命令    │
│  ├─ 触发描述     → AI 判断何时调用的依据       │
│  ├─ 触发模式     → 自动/手动/仅自动            │
│  ├─ 工具权限     → 允许使用哪些工具            │
│  └─ 执行模式     → 内联执行 / 隔离执行         │
├─────────────────────────────────────────────┤
│  指令正文                                     │
│  ├─ 触发场景示例  → 帮 AI 判断调用边界         │
│  ├─ 执行步骤     → 结构化，编号，可检查        │
│  └─ 输出格式     → 明确规定，不留歧义          │
└─────────────────────────────────────────────┘
    ↓
附属资源（按需引用）
├─ 知识库文档   → 详细规范、背景知识
└─ 辅助脚本    → 自动化执行的脚本
```

### 3.2 标准技能定义文件格式

以目前最通用的 YAML 头部 + Markdown 正文格式为例，一个标准的技能定义文件如下：

```markdown
---
name: competitive-analysis          # 技能名称（唯一标识符）
description: |                      # 触发描述（最关键）
  执行竞品分析，生成结构化对比报告。
  触发条件：用户说"分析竞品"、"做竞品对比"、"看看竞品情况"、
  "X和我们比怎么样"等。
  分析维度：功能/价格/用户口碑/市场定位。
  输出：Markdown 对比表 + 差异化战略建议。
trigger-mode: auto                  # 触发模式：auto（自动）/ manual（手动）
allowed-tools: read-file, search    # 工具权限（可选）
metadata:                           # 附加信息（可选）
  version: "1.0.0"
  owner: "市场部"
---

# 竞品分析标准流程

## 触发场景示例
- "分析一下竞品 X 和 Y"
- "竞品最近有什么动作"
- "我们和竞品比有什么优势"

## 执行步骤（按顺序，不可跳过）

### Step 1：确认分析范围
分析前确认（如未提供，先询问）：
- 分析对象：哪些竞品？（必须）
- 时间范围：最近多久？（默认：3个月）
- 重点维度：有没有特别关注的方面？（可选）

### Step 2：逐维度分析
（详细步骤...）

## 输出格式
（具体格式规定...）
```

### 3.3 元数据字段的通用说明

无论平台语法如何，这几个配置项在概念上是通用的：

| 配置项 | 必须 | 作用 |
|--------|------|------|
| **技能名称** | ✅ | 唯一标识符，同时也是手动调用时的命令名。命名规范：动词-对象，如 `analyze-competitor`、`write-weekly-report`。 |
| **触发描述** | ✅ | **最关键的配置**。AI 根据此描述判断何时调用该 Skill。需说明：做什么、什么场景触发、覆盖哪些范围。建议 200 字以内，过长反而影响匹配精度。 |
| **触发模式** | ❌ | 控制 AI 能否自动触发。`auto`（默认）= AI 判断后自动调用；`manual` = 仅用户显式调用。**涉及对外发送或系统修改的操作，必须设为 manual。** |
| **工具权限** | ❌ | 限制该 Skill 可使用的工具列表。安全沙箱设计：只读分析只给读取权限，防止 Skill 执行超出预期的操作。 |
| **执行模式** | ❌ | 是否在隔离子 Agent 中执行。危险操作或长流程任务建议开启隔离，防止影响主对话。 |
| **附加信息** | ❌ | 版本号、负责人、更新时间等管理信息，不影响执行逻辑，但有助于团队维护。 |

### 3.4 不同场景的配置策略

**场景 A：只读分析型（低风险，允许自动触发）**

```
名称: analyze-performance
触发描述: 分析业务性能指标，定位瓶颈和异常
触发模式: auto
工具权限: 仅限读取文件和搜索
```

**场景 B：对外操作型（高风险，必须手动触发）**

```
名称: send-customer-email
触发描述: 向客户发送跟进邮件，需确认内容后发送
触发模式: manual          ← 必须手动，防止误发
执行模式: isolated        ← 隔离执行，防止影响主流程
```

**场景 C：知识注入型（仅供 AI 自动参考，不在用户菜单显示）**

```
名称: team-conventions
触发描述: 团队业务规范、流程约定和常见处理方式
触发模式: auto-only       ← 仅 AI 自动触发，用户不可手动调用
```

**场景 D：带版本管理的业务流程型**

```
名称: weekly-report
触发描述: 生成本周工作周报，汇总完成事项和下周计划
附加信息:
  version: "2.1.0"
  owner: "运营部"
  last-updated: "2026-03-28"
```

## 四、写好 Skill 的原则与范式

### 4.1 核心原则：触发描述优先

**Skill 最重要的不是正文，是触发描述。**

触发描述决定了 AI 是否知道"该在什么时候用这个 Skill"。正文再完善，触发描述写得模糊，Skill 就形同虚设。

**写好触发描述的三要素：**

```
1. 主动描述：以动词开头，说明 Skill 做什么
   ❌ "竞品相关的事情"
   ✅ "执行竞品分析，对比功能、价格、口碑，输出结构化报告"

2. 场景描述：说明在什么情况下触发
   ❌ "竞品分析"
   ✅ "当用户要求分析竞品、了解竞争格局、或询问某竞品动态时触发"

3. 范围描述：说明覆盖哪些方面，避免歧义
   ❌ "处理竞品问题"
   ✅ "分析维度包括：功能对比、价格体系、用户口碑、市场定位；
      不覆盖：内部产品功能咨询、代码开发任务"
```

**一个好触发描述的完整示例：**

```
执行竞品分析，生成结构化对比报告。
触发条件：用户说"分析竞品"、"做竞品对比"、"看看竞品情况"、
"X和我们比怎么样"、"竞品有什么新动作"等。
分析维度：功能/价格/用户口碑/市场定位。
输出：Markdown 对比表 + 差异化战略建议。
不触发：用户询问内部产品功能或要求写代码时。
```

### 4.2 正文写作范式：STEP

好的 Skill 正文遵循 **STEP 范式**：

```
S - Structure（结构化）：用编号列表，不用叙述性文字
T - Target（明确目标）：每一步说清楚要做什么、产出什么
E - Examples（带例子）：至少 3 个触发场景示例
P - Precise Output（精确输出）：明确规定输出格式
```

**❌ 不好的正文（叙述式，AI 执行结果不稳定）：**

```markdown
请帮用户做竞品分析，了解竞争情况，然后给出一些建议。
注意要全面，如果有问题的话提醒一下用户。
```

**✅ 好的正文（结构化，每次执行结果一致）：**

```markdown
# 竞品分析标准流程

## 触发场景示例
- "帮我分析一下竞品 A 和 B"
- "竞品最近有什么新功能"
- "我们和市场上其他产品比有什么差距"

## 执行步骤（按顺序执行，不可跳过）

### Step 1：确认分析范围
分析前确认以下信息（如未提供，先询问）：
- 分析对象：哪些竞品？（必须，1-5 个）
- 时间范围：默认最近 3 个月
- 重点关注维度：可选

### Step 2：逐维度分析
- [ ] 核心功能对比（列出各竞品的功能模块，标注我们有/没有）
- [ ] 价格体系（定价策略、区间、近期调整）
- [ ] 用户口碑（主要平台评价，痛点和好评点）
- [ ] 市场定位（目标客群、核心卖点、营销动作）

### Step 3：数据来源说明
- 优先使用用户提供的资料
- 基于已知公开信息分析
- **数据不足时：明确说明缺少什么，不得猜测或编造**

## 输出格式（严格按此格式）

**第一部分：对比总表**
| 维度 | 我们的产品 | 竞品 A | 竞品 B |
|------|-----------|--------|--------|
（按维度填写）

**第二部分：关键发现**（3-5 条）
> **发现 N**：[一句话描述] → **影响**：[对我们的影响]

**第三部分：差异化建议**（2-3 条）
1. **[建议方向]**：[具体说明] — 优先级：高/中/低
```

### 4.3 文件组织范式

Skill 正文超过 **200 行**时，考虑拆分：

```
competitive-analysis/
├── SKILL.md              ← 核心流程（<100 行）
│   内容：触发示例 + 步骤索引 + 输出格式
│   引用：详细标准见 references/
│
└── references/
    ├── analysis-framework.md   ← 分析框架（详细版）
    ├── output-examples.md      ← 输出示例
    └── common-competitors.md   ← 常见竞品背景信息
```

主文件用简洁的引用代替大段内容：

```markdown
## 分析框架

详细框架见 `references/analysis-framework.md`。核心维度：
- 功能对比
- 价格体系
- 用户口碑
- 市场定位
```

### 4.4 安全设计原则

按操作的"不可逆程度"选择触发保护级别：

```
只读分析（可逆 ✓）
  → 工具权限：仅读取
  → 触发模式：允许自动触发

内容生成（草稿，可修改后再用 ⚠️）
  → 触发模式：自动触发，但输出需用户确认后才能使用

对外发送 / 系统修改（不可逆 ❌）
  → 如：发邮件、发消息、修改数据库
  → 触发模式：必须手动触发
  → 执行模式：隔离执行
```

**黄金法则：凡是会直接影响外部系统或对外输出的 Skill，必须设置为手动触发。**

## 五、常见误区

### 误区 1：触发描述写成标题，不写场景

```
# ❌ 错误
触发描述: 竞品分析工具

# ✅ 正确
触发描述: 执行竞品分析，生成对比报告。
          用户说"分析竞品"、"看看竞品"、"竞品有什么动作"时触发。
          分析维度：功能/价格/口碑/定位。不覆盖内部产品咨询。
```

**后果：** AI 不知道何时调用，或在不合适的时候乱触发。

### 误区 2：一个 Skill 做太多事

```
# ❌ 错误：一个 Skill 涵盖所有市场工作
名称: marketing-helper
内容：竞品分析 + 推广文案 + 用户调研 + 月报生成 + 活动策划

# ✅ 正确：每个任务一个 Skill
名称: analyze-competitor    # 专注竞品分析
名称: write-promo-copy      # 专注推广文案
名称: generate-user-survey  # 专注调研问卷
```

**后果：** 触发描述无法精准描述场景，AI 选错 Skill；正文混乱，执行结果不稳定。

### 误区 3：危险操作不设保护

```
# ❌ 错误：发客户邮件的 Skill 允许 AI 自动触发
名称: send-customer-email
触发模式: auto（未设置保护）

# 后果：用户说"给客户发一下这个信息"，AI 可能直接触发发送邮件
```

### 误区 4：正文全是叙述性文字

```
# ❌ 错误
请帮用户分析竞品，仔细了解各方面情况，
然后给出一些建议，总体上要全面、专业。

# ✅ 正确
## 步骤（按顺序，不可跳过）
1. 确认分析对象和范围
2. 逐维度收集信息（见清单）
3. 整理对比表
4. 输出关键发现和建议（严格按输出格式）
```

**后果：** AI 执行结果不稳定，每次输出不一致。

### 误区 5：混淆 Skill 和工具调用的边界

```
# 不应该用 Skill 做的事：
- 实时查询数据库 → 用工具调用
- 调用第三方 API → 用工具调用
- 动态获取外部实时数据 → 用工具调用

# 应该用 Skill 做的事：
- 定义操作流程 → 用 Skill
- 注入领域知识和规范 → 用 Skill
- 规范输出格式 → 用 Skill
```

### 误区 6：忘记写触发示例

触发示例是触发描述的重要补充，AI 会参考它来判断调用边界：

```
# ❌ 没有示例
触发描述: 生成周报

# ✅ 有示例
触发描述: |
  生成本周工作周报。触发示例：
  "帮我写周报"、"总结一下这周"、"生成周报"、
  "我要发周报了"、"写一下本周情况"。
  禁止触发：用户问具体业务问题或要求分析数据时不触发。
```

### 误区 7：Skill 过大导致上下文膨胀

```
# ❌ 错误
技能主文件: 2000 行（把所有规范都写进去）

# ✅ 正确
技能主文件: 80 行（核心流程 + 引用）
知识库文档/分析框架.md: 500 行（详细框架）
知识库文档/安全清单.md: 300 行（检查规则）
```

**技能主文件建议控制在 300 行以内。**

## 六、业务流程 Skill 化指南

> 本节专为业务人员准备。即使你不写代码，也可以用这套方法把自己的工作流程封装成 Skill。

### 6.1 什么样的业务流程适合 Skill 化

不是所有流程都值得 Skill 化。先用这 4 个维度评估：

```
┌─────────────────────────────────────────┐
│           Skill 化价值矩阵               │
│                                         │
│  高频使用 × 规范明确 = 强烈建议 Skill 化  │
│  高频使用 × 规范模糊 = 先梳理流程再 Skill  │
│  低频使用 × 规范明确 = 可以但优先级低     │
│  低频使用 × 规范模糊 = 不建议 Skill 化   │
└─────────────────────────────────────────┘
```

**适合 Skill 化的信号：**
- "这个我们每次都这么做，但每次还要重新讲一遍给 AI 听"
- "这个流程有固定的检查项，容易漏"
- "这个任务对输出格式要求很严格"
- "这个流程需要固定的专业知识支撑"

**不适合 Skill 化的信号：**
- "这个每次情况都不一样" → 流程太灵活，不适合模板化
- "我们还在摸索最佳做法" → 流程未定型，先探索再 Skill 化
- "这个只用过一次" → 不够高频，直接和 AI 描述就好

### 6.2 业务流程梳理的五步法

在动手写技能定义文件之前，先做流程梳理：

**Step 1：完整走一遍（记录版本）**

先把这个任务完整做一遍，把你给 AI 说的每一句话、提供的每一份资料，全部记录下来。

```
示例：竞品分析任务
我给 AI 说的：
- "帮我分析竞品 X 和竞品 Y"
- "对比维度包括：功能、价格、用户口碑、技术架构"
- "最后给出一个建议，我们的产品应该如何差异化"
- "输出格式用表格"

我额外提供的：
- 我们公司的产品介绍文档
- 竞品的官网截图
```

**Step 2：提炼可复用的部分**

区分"每次固定的"和"每次变化的"：

```
竞品分析任务：
  ✅ 固定的（写入 Skill）：
    - 对比维度（功能/价格/口碑/技术）
    - 分析框架
    - 输出格式（Markdown 表格）
    - 差异化建议维度

  🔄 变化的（调用时提供）：
    - 具体的竞品名称
    - 分析的时间范围
    - 重点关注的维度
```

**Step 3：确定触发时机**

写下所有你可能对 AI 说的话，这些就是触发描述的示例：

```
我会这样要求：
- "分析一下竞品"
- "做个竞品对比"
- "看看 X 和 Y 比我们怎么样"
- "市场上有哪些竞品，我们的优势在哪里"
```

**Step 4：确定安全级别**

问自己：这个 Skill 的输出会不会直接影响外部？

```
只是生成文档 → 普通 Skill，允许自动触发
会发邮件/消息 → 必须手动触发
会修改系统数据 → 手动触发 + 隔离执行
```

**Step 5：写技能定义文件**

按照格式写，重点在触发描述和步骤的清晰度。

### 6.3 粒度判断：Skill 应该多大

Skill 的粒度是最难判断的问题。**以下是粒度判断框架：**

**粒度过大的信号（需要拆分）：**

```
❌ 一个 Skill 涵盖整个项目生命周期：
   名称: product-launch
   内容：市场调研 + 竞品分析 + 产品定义 + 上线策略 + ...

→ 拆成：market-research、competitive-analysis、product-spec、launch-strategy
```

- 触发条件模糊，描述超过 400 字还不够精准
- 完成一次任务需要用户多次补充信息
- 输出结果经常偏离预期

**粒度过小的信号（需要合并）：**

```
❌ 每个小动作都是一个 Skill：
   名称: write-email-subject    # 只写邮件主题
   名称: write-email-body       # 只写邮件正文
   名称: write-email-signature  # 只写邮件签名

→ 合并成：write-business-email
```

- 每次使用都需要连续调用 2-3 个 Skill
- 单个 Skill 不到 15 行内容
- Skill 之间强依赖（A 必须在 B 之前）

**恰好的粒度标准：**

```
✅ 一个 Skill 对应一个独立完整的工作产出
   - "完成这个 Skill，我会得到一份可以直接使用的东西"
   - 触发条件可以用 100 字以内清晰描述
   - 独立执行，不依赖其他 Skill
   - 执行时间：1-15 分钟（太短说明太碎，太长说明太大）
```

**业务场景粒度示例：**

| 业务场景 | 过细（❌） | 适中（✅） | 过粗（❌） |
|---------|----------|----------|----------|
| 运营 | 写标题/写正文/写CTA（各一个） | 写推广文案（一个） | 运营工作（一个） |
| 销售 | 写开场白/写需求探索/写结尾（各一个） | 写销售邮件（一个） | 全套销售流程（一个） |
| 产品 | 写背景/写方案/写指标（各一个） | 写 PRD（一个） | 产品设计全流程（一个） |
| 数据 | 数据清洗/数据分析/写报告（各一个） | 数据分析报告（一个） | 所有数据工作（一个） |

**原始流程：**

# 竞品分析标准流程

## 触发前提

分析前确认以下信息（如未提供，先询问）：
- 分析对象：哪些竞品？（必须，1-5个）
- 时间范围：最近多久的动态？（默认：最近 3 个月）
- 重点关注：有没有特别关注的维度？（可选）

## 分析维度（全部必须覆盖）

### 1. 核心功能对比
- 列出各竞品的核心功能模块
- 标注我们有/没有的功能
- 标注竞品近期新增/删除的功能

### 2. 价格体系
- 各竞品定价策略（免费/订阅/按量）

- 近期是否有价格调整

### 3. 用户口碑
- 主要平台评价（App Store / 知乎 / 行业论坛）
- 用户主要痛点和好评点
- 净推荐值（NPS）如有公开数据

### 4. 技术架构（如可获取）
- 技术栈关键信息
- 性能/稳定性相关用户反馈

### 5. 市场定位
- 目标客群定位
- 核心差异化卖点
- 近期营销动作

## 数据来源说明

分析时：
1. 优先使用用户提供的资料文件
2. 基于已知公开信息进行分析
3. **数据不足时：明确说明缺少什么，不得编造或猜测**

### 第一部分：对比总表

| 维度 | 我们的产品 | 竞品 A | 竞品 B |
|------|-----------|--------|--------|
| 核心功能 | ... | ... | ... |

| 核心卖点 | ... | ... | ... |

### 第二部分：关键发现

### 第三部分：差异化建议

## 八、用 Skill-Creator Agent 创建技能：业务人员的人机协作框架

> 本节针对内部 AI 工作流平台的使用场景。你不需要会写代码，也不需要理解 YAML 语法——你只需要知道自己的业务流程，然后和 Skill-Creator Agent 对话，它会帮你把流程转化成可复用的 Skill。

### 8.1 整体工作流程

在内部平台上，技能创建的完整流程如下：

```
业务人员
    ↓ 发起对话："我想创建一个 XXX 的技能"

Skill-Creator Agent（技能创建助手）
    ↓
  阶段一：探索访谈
  ├─ 提问：这个流程是什么？你通常怎么做？
  ├─ 提问：什么时候会用到？触发词是什么？
  ├─ 提问：需要什么输入？输出是什么格式？
  └─ 提问：有没有注意事项、特殊规则、例外情况？
    ↓
  阶段二：结构化草稿
  └─ 生成技能定义文件草稿（触发描述 + 完整正文）
    ↓
  阶段三：人工审核（Human in the Loop ← 关键环节）
  ├─ 业务人员对照真实业务检查草稿
  ├─ 指出不准确、遗漏、表达有误的地方
  └─ Agent 修改 → 再次确认 → 直到通过
    ↓
  阶段四：试跑验证
  └─ 用一个真实案例跑一遍，确认输出符合预期
    ↓
  阶段五：发布上线
  └─ 技术人员或管理员部署到平台技能库

所有同事可以使用这个 Skill
```

**为什么这个流程有效：**
- AI 负责**结构化**（把你说的话转成规范格式）
- 你负责**知识输入**（只有你知道业务真正怎么运作）
- Human in the Loop 负责**质量把关**（防止 AI 理解偏差造成的错误沉淀）

### 8.2 Human in the Loop：为什么人不能离开这个循环

很多人以为"让 AI 自动生成技能"就完事了。这个想法有一个致命缺陷：

**AI 没有你的业务知识。**

Skill-Creator Agent 很擅长把信息整理成结构化格式，但它不知道：
- 你们公司的"竞品分析"和行业通用的有什么不同
- 你们发给客户的邮件有什么特殊的禁忌词
- 某个审批流程里有个"口头规定"，但文档里没写
- 某类异常情况在你们业务里特别常见，需要特殊处理

这些隐性的、经验性的业务知识，只存在于业务人员的脑子里。如果没有 Human in the Loop，这些知识要么丢失，要么变成错误的规则被固化到 Skill 里——而且每次执行都会错。

**Human in the Loop 的三个关键节点：**

```
节点 1：知识输入阶段（提供原材料）
  你的职责：完整、准确地描述流程和规则
  AI 的职责：提问、追问、整理
  风险点：说得太简单 → AI 只能猜 → Skill 不准确

节点 2：草稿审核阶段（质量把关）
  你的职责：像看合同一样认真检查每一条规则
  AI 的职责：解释每条规则的来源，接受修改意见
  风险点：走马观花 → 错误规则上线 → 全团队都在按错误流程执行

节点 3：试跑验证阶段（实际检验）
  你的职责：提供真实案例，判断输出是否达到预期
  AI 的职责：按 Skill 执行，展示输出
  风险点：用过于简单的案例测试 → 上线后遇到复杂场景就出错
```

**一个实际案例说明这个问题：**

> 某运营同学想创建"客户投诉处理邮件"技能。她简单描述说："收到投诉后，道歉，说明处理方案，给补偿。"AI 生成的草稿看起来没问题。但她没有说的是：对 VIP 客户的补偿标准是普通客户的 3 倍，而且绝对不能在邮件里提"赔偿"这个词（有法务风险）。如果没有人工审核，这个 Skill 上线后对 VIP 客户就会给出错误的补偿金额，而且可能触发法律风险。

### 8.3 角色分工：每个人该做什么

| 角色 | 职责 | 关键能力要求 |
|------|------|------------|
| **业务人员**（Skill 发起者） | 提供业务知识、审核草稿、验证输出 | 熟悉自己的业务流程，能识别 AI 理解是否准确 |
| **Skill-Creator Agent** | 访谈、结构化、生成草稿、修改迭代 | — |
| **技术管理员**（可选） | 最终部署上线、权限设置、版本管理 | 了解平台技术配置 |
| **团队负责人**（可选） | 审批高影响力的 Skill（如涉及对外发送） | 业务和风险判断 |

**业务人员不需要具备的能力：**
- 不需要会写代码
- 不需要理解 YAML / 配置文件格式
- 不需要懂 AI 工作流的技术细节
- 不需要了解 Skill 的底层机制

**业务人员必须具备的能力：**
- 能完整描述自己的工作流程（包括例外情况）
- 能判断 AI 生成的规则是否符合真实业务
- 能提供真实案例用于验证

### 8.4 如何高质量提问：信息提供指南

Skill-Creator Agent 会主动问你问题，但你回答的质量直接决定 Skill 的质量。

**提问前的准备工作（5 分钟）：**

在开始对话之前，建议先快速梳理以下信息：

```
我要创建的技能：[用一句话说清楚]
这个流程我通常做的步骤：[1、2、3...]
最常见的触发场景（我会怎么说）：[示例1、示例2...]
必须要有的输入信息：[...]
输出格式要求：[...]
有没有特殊规则或例外：[...]
```

**回答质量对比：**

Agent 问："这个竞品分析流程通常是怎么做的？"

```
❌ 低质量回答（太简单，AI 只能猜）：
"就是分析一下竞品的情况。"

⚠️ 中等质量回答（有框架，但缺细节）：
"看功能、价格、用户口碑，然后写总结。"

✅ 高质量回答（AI 能直接转化成规则）：
"我通常先看他们最新的功能更新（去官网产品页），然后对比我们
有没有类似功能。价格这块主要看定价页面和促销活动。口碑主要看
知乎和行业论坛。最后写一个对比表，加上3条对我们的启示。
特别注意：我们绝对不能在报告里说'竞品比我们好'，要说'竞品
在某方面有不同的策略'。"
```

**被 Agent 追问时，如何回答：**

Agent 经常会问"这种情况下你会怎么处理"、"有没有例外"。这类问题很关键，不要跳过：

```
Agent："如果竞品没有公开定价，你怎么处理？"

❌ 不好的回答："跳过就好了。"

✅ 好的回答："如果没有公开定价，我会在报告里注明'定价未公开，
据用户反馈约在X-Y元区间'，不能空着不写，因为读报告的人会觉得
这块我们漏查了。"
```

**5 类必须主动告知的信息（即使 Agent 没有问）：**

1. **禁忌项**：有没有不能说的词、不能做的操作？
2. **特殊人群**：对不同类型的对象（VIP 客户/普通客户、新用户/老用户）处理方式是否有差异？
3. **例外场景**：什么情况下流程会不一样？
4. **审批节点**：哪些步骤需要人工确认才能继续？
5. **输出接收方**：这个输出给谁看？他们有什么特殊要求？

### 8.5 交互对话框架：标准协作流程

以下是与 Skill-Creator Agent 协作的**完整对话框架**，分为五个阶段：

---

**阶段一：意图确认（~2 分钟）**

> 目标：让 Agent 明白你要做什么、这个 Skill 的价值是什么

```
你：我想创建一个"客户续费提醒邮件"的技能

Agent 会问：
1. "这个技能要解决什么问题？现在是怎么做的？"
2. "使用频率大概是多少？一天几次还是一周几次？"
3. "谁会用这个技能？只有你，还是整个销售团队？"

你需要回答清楚：
- 现状痛点（每次都要重新想措辞 / 不同人发的格式不一致 / 容易漏关键信息）
- 使用频率（高频 = 值得仔细打磨）
- 使用范围（范围越广 = 越需要严格的 Human in the Loop）
```

---

**阶段二：流程访谈（~10-20 分钟，最核心的阶段）**

> 目标：把你脑子里的业务知识完整转移给 Agent

Agent 会按以下顺序提问，你需要尽量详细地回答：

```
1. 流程框架
   "请描述完整的处理流程，从开始到结束，分步骤说明。"
   → 你：说出每一步你实际会做什么

2. 输入信息
   "执行这个流程需要哪些前置信息？哪些是必须的，哪些是可选的？"
   → 你：列出所有需要用到的数据、文件、背景信息

3. 触发场景
   "你通常在什么情况下做这件事？举3-5个你会怎么说的例子。"
   → 你：说出真实的触发词，越口语化越好

4. 输出规格
   "最终产出是什么？格式是什么？给谁看？"
   → 你：说清楚格式、语气、长度要求

5. 特殊规则
   "有没有特别要注意的、容易出错的、规定死的东西？"
   → 你：这是最重要的地方，把所有潜规则说出来

6. 例外处理
   "遇到XXX情况怎么办？遇到YYY情况怎么办？"
   → 你：把所有"如果...那么..."的规则说清楚
```

**注意：Agent 可能追问 3-5 轮才够用，不要嫌烦。**

---

**阶段三：草稿审核（~15 分钟，最关键的 Human in the Loop 节点）**

> 目标：把 AI 的结构化结果和你的业务知识对齐

Agent 生成草稿后，你需要**逐节检查**：

```
检查清单（必须逐项过）：

□ 触发描述是否准确？
  → 触发词是否覆盖了我常说的话？
  → 有没有误触发的可能？

□ 步骤是否完整？
  → 有没有漏掉某个步骤？
  → 步骤顺序对不对？

□ 规则是否准确？
  → 每条规则是否符合实际业务？
  → 有没有规则写反了或写错了？

□ 禁忌是否已经写入？
  → 我告知的禁忌词是否体现在 Skill 里？
  → 例外处理逻辑是否正确？

□ 输出格式是否对？
  → 格式、语气、长度是否符合要求？
  → 接收方能接受吗？
```

**发现问题时如何反馈：**

```
❌ 模糊反馈（Agent 不知道怎么改）：
"感觉不太对，再改改。"

✅ 精准反馈（Agent 可以直接修改）：
"第3步'根据情况给补偿'太模糊了。
 改成：VIP客户（年消费>10万）补偿当月费用的20%，
       普通客户补偿当月费用的5%，
       补偿上限500元，
       补偿方式只能是账户余额，不能是现金。"
```

---

**阶段四：试跑验证（~10 分钟）**

> 目标：用真实案例验证 Skill 的实际执行质量

```
你准备一个真实案例：
  "这是我上周处理的一个真实客户情况：[具体信息]"

Agent 执行 Skill，给出输出

你检查：
  □ 输出内容是否准确？
  □ 格式是否符合要求？
  □ 有没有我实际上不会这么说的表达？
  □ 如果这个输出真的用出去，会有什么问题吗？

如果不满意：告诉 Agent 哪里不对 → 回到阶段三修改
如果满意：进入阶段五
```

**选择验证案例的原则：**
- 用**真实**的案例，不要用简化版
- 选一个**有点复杂**的案例，覆盖 1-2 个例外场景
- 不要只用最简单的、最标准的情况来测试

---

**阶段五：发布准备**

> 目标：确认 Skill 可以给更多人用了

```
你需要确认：
□ Skill 名称：清晰、规范（动词-对象格式）
□ 所有者：填写部门/姓名，便于后续维护
□ 版本：1.0.0（初始版本）
□ 安全级别确认：
  - 这个 Skill 会不会直接发送邮件/消息/修改数据？
    → 会：需要设置为手动触发
    → 不会：普通配置即可

提交给技术管理员部署上线
```

### 8.6 一次完整的对话示例

以创建"销售跟进邮件"技能为例：

---

**你**：我想创建一个销售跟进邮件的技能，每次拜访客户后要发跟进邮件，我们整个销售团队都会用。

**Agent**：好的，我来帮你创建这个技能。先了解一下背景：
1. 目前这封邮件通常在拜访后多久内发送？有没有时间规定？
2. 邮件主要目的是什么——确认拜访内容、推进下一步，还是两者都有？
3. 不同阶段的客户（初次接触、意向客户、即将签单）发的内容有区别吗？

**你**：一般要求24小时内发出。目的是确认拜访内容和推进下一步行动。不同阶段的客户内容差很多：
- 初次接触：主要是建立关系，不要太推销
- 意向客户：重点强调我们的差异化优势，推进到下一次会面
- 即将签单：跟进合同进展，解决剩余顾虑

**Agent**：明白了。再问几个细节：
1. 邮件里需要附上什么材料吗（产品手册、方案文档等）？
2. 有没有禁忌——某些词不能用，或者某些话不能说？
3. 你们的邮件有固定签名格式吗？

**你**：一般意向客户和即将签单阶段会附方案文档，但不是每次都有，要根据情况判断。
禁忌：不能说"我们比竞品便宜"，要说"我们在价值方面更有优势"；不能催单，不能说"你们什么时候决定"。
签名有固定格式，我稍后把模板发给你。

**Agent**：收到。关于例外情况：如果拜访不顺利（客户反馈负面）或者客户明确说暂时不考虑，跟进邮件的策略会不一样吗？

**你**：对，如果客户说暂时不考虑，邮件要换成"保持联系"模式，不能提产品，重点是表达理解和留下好印象。这种情况我们叫"冷却期跟进"。

**Agent**：非常有帮助。我现在生成草稿，请你检查…

*[Agent 生成技能定义文件草稿]*

**Agent**：草稿已生成，请重点检查以下几点是否符合你们的实际情况：
1. 三种客户阶段的判断标准，你来确认分类是否准确
2. "冷却期跟进"模式的触发条件
3. 关于附件判断的逻辑（我写了一个简单的判断规则，不知道是否符合实际）

**你**：第2条不对，"冷却期跟进"不只是客户说暂时不考虑——如果拜访过程中客户对价格有很大异议也算，这种情况也要用冷却期模式，给他一些消化时间…

*[几轮修改后]*

**你**：好，这个版本对了。用上周跟XX公司的拜访记录跑一下试试。

*[Agent 执行，生成邮件样本]*

**你**：语气还可以，但"期待进一步合作"这句话出现了两次，只留一次。另外我们公司名字后面不用加"团队"，直接写公司名。

*[Agent 修改]*

**你**：好了，这个可以了。技能名字叫 `write-sales-followup`，归属销售部。这个技能生成的是草稿，不会直接发送，应该不需要手动触发保护。

**Agent**：确认。这个技能输出的是邮件草稿，由销售人员自行确认后发送，不涉及直接对外操作，触发模式设为 auto 即可。已整理好完整的技能定义文件，提交给管理员部署即可。

### 8.7 常见的协作卡点与解法

| 卡点 | 原因 | 解法 |
|------|------|------|
| Agent 生成的草稿"看起来不错但感觉哪里不对" | 缺乏真实案例对照 | 拿一个真实的历史案例，让 Agent 执行草稿，对比实际处理结果 |
| 描述了半天，Agent 还是没理解 | 描述太抽象 | 说"举个例子"，给出一个具体情境，Agent 从具体案例反推规则 |
| 流程太复杂，一个 Skill 装不下 | 粒度太大 | 和 Agent 一起拆分，先识别哪些步骤是独立的工作产出 |
| 审核草稿时不知道从哪里看起 | 缺乏结构 | 先只看触发描述，确认触发场景对不对；再看步骤数量和顺序；最后看细则 |
| 试跑结果和预期不一致，但说不清哪里不对 | 缺乏对比基准 | 先手动写一份"理想输出"，再让 Agent 执行，逐句对比差异 |
| 技能上线后有人反映结果不对 | 边缘场景未覆盖 | 收集反馈案例，和 Agent 一起分析原因，更新 Skill 的例外处理规则 |

### 8.8 Skill 的生命周期管理

Skill 不是一次性产出，它需要随着业务变化持续维护：

```
创建 → 上线 → 使用 → 反馈 → 更新 → 再上线
  ↑                                        ↓
  └────────── 每季度审查一次是否仍然准确 ────┘
```

**什么时候需要更新 Skill：**
- 业务流程发生了变化（政策调整、产品改版、组织架构变化）
- 团队成员反馈 Skill 执行结果不符合预期
- 发现了新的高频例外场景没有被覆盖
- 禁忌词或规则有更新（如法务要求变化）

**更新时的 Human in the Loop：**
- 小改动（补充例外规则、调整表述）：原 Skill 负责人审核即可
- 大改动（流程框架变化）：重新走完整的创建流程

## 九、一图总结

```
Skill 全局视图：

用户 / AI 工作流
    ↓ 触发（显式 /skill-name 或自动语义匹配）

技能定义文件
┌─────────────────────────────────────────────┐
│  元数据配置                                   │
│  ├─ 技能名称     → 唯一标识符 / 调用命令       │
│  ├─ 触发描述     → 触发判断依据 ⭐ 最重要      │
│  ├─ 工具权限     → 安全沙箱                   │
│  ├─ 触发模式     → 危险操作必须设为手动        │
│  └─ 执行模式     → 隔离执行 / 内联执行         │
├─────────────────────────────────────────────┤
│  指令正文                                     │
│  ├─ 触发场景示例  → 帮 AI 判断调用边界         │
│  ├─ 执行步骤     → 结构化，编号，可检查        │
│  └─ 输出格式     → 明确规定，不留歧义          │
└─────────────────────────────────────────────┘
    ↓
附属资源（按需加载）
├─ 知识库文档   → 详细规范、背景知识
└─ 辅助脚本    → 自动化执行的脚本
```

**写好 Skill 的核心原则，只有一句话：**

> **把一个有经验的人处理这件事的完整思路，用结构化的方式写下来，让 AI 每次都能按这个标准执行。**
