# AI 工作流 Skill 实战指南（业务人员版）
> 受众：银行等金融机构业务人员（以资产负债管理部为例）
> 核心目标：理解 Skill 核心概念，掌握业务流程 Skill 化方法
> 典型场景：债券因子挖掘全流程封装

---

## 写在前面
作为银行资产负债管理部的业务人员，你是否经常遇到这样的情况：
- 每次让 AI 帮忙做债券因子挖掘，都要重新说明“我们只分析利率债的利差/久期/流动性因子”“输出格式要包含因子有效性检验结果”；
- 不同同事用 AI 做同款任务，输出的因子分析报告格式、维度、校验标准完全不一致；
- 新人接手时，需要反复讲解“债券因子挖掘的核心步骤和风控要求”。

这不是 AI 不够专业，而是**债券因子挖掘这类核心业务流程没有被标准化封装**——而 Skill 就是解决这个问题的核心方法。

本文从业务人员视角出发，用“银行资产负债管理部债券因子挖掘”为核心案例，讲清楚 Skill 是什么、和其他 AI 能力的区别、以及如何把你的业务流程写成可复用的 Skill。

## 一、什么是 Skill
### 1.1 一句话定义
**Skill 是为特定业务场景定制的、可复用的任务执行模板**——当你或同事需要做债券因子挖掘时，AI 会自动加载这套预设的流程、规则和输出标准，无需重复描述背景和要求。

### 1.2 从业务人员视角看
你在 AI 工作流平台输入 `/债券因子挖掘`，或直接说“帮我分析利率债的利差因子有效性”：
```
方式1（显式调用）：
用户 → /债券因子挖掘
↓
AI 加载「债券因子挖掘」Skill 的完整指令
↓
按预设的因子维度、校验标准、输出格式完成分析

方式2（自动触发）：
用户 → "分析下本周利率债久期因子对组合收益的影响"
↓
AI 理解意图，匹配到「债券因子挖掘」Skill
↓
自动加载并执行，输出标准化报告
```

无论哪种方式，输出的报告都符合资负部的统一标准，无需反复调整格式和维度。

## 二、Skill 与同类能力的核心区别
业务人员常混淆 Skill、工具调用、MCP（模型上下文传递）的边界，以下表格用金融业务场景讲清楚差异：

| 维度                     | Skill（技能）                                              | 工具调用（Tool Use）                                           | MCP（Model Context Passing）                                     |
| ------------------------ | ---------------------------------------------------------- | -------------------------------------------------------------- | ---------------------------------------------------------------- |
| **本质**                 | 业务流程+规则的标准化模板                                  | 调用外部数据/计算能力的接口                                    | 跨模型/会话传递上下文的技术                                      |
| **核心解决的问题**       | 「我该按什么流程做这件事」（如因子挖掘的步骤/标准）        | 「我该去哪里取数据/算结果」（如查债券行情/算因子有效性）       | 「如何让不同 AI 模型共享上下文」（如把因子基础数据传给分析模型） |
| **触发方式**             | 显式调用（/指令）或 AI 语义匹配触发                        | AI 判断任务需要时自动调用                                      | 系统底层自动传递，无感知                                         |
| **存储形式**             | 结构化文本（流程/规则/格式）                               | 独立 API/脚本（如行情查询接口）                                | 系统级上下文通道                                                 |
| **债券因子挖掘场景示例** | 定义“因子挖掘需覆盖利差/久期/流动性维度、输出有效性校验表” | 调用中债登行情接口获取债券价格、调用 Python 脚本计算因子显著性 | 把行情数据上下文传给因子分析模型，无需重复输入                   |

**记忆口诀**：
- Skill → 业务流程的“操作手册”（怎么做）
- 工具调用 → 业务执行的“计算器/数据库”（去哪取数/计算）
- MCP → 不同 AI 能力间的“数据线”（无感传数据）

## 三、Skill 的核心工作原理（业务人员易懂版）
你无需理解技术细节，只需记住 2 个核心逻辑：

### 3.1 按需加载：不浪费资源，只加载需要的内容
Skill 不会把所有规则一次性传给 AI，而是分三层加载：
```
第一层：基础信息（始终加载）
  ├─ 技能名称：债券因子挖掘
  ├─ 触发场景：分析债券因子有效性、测算因子对组合收益影响
  └─ 占用极少空间，不影响日常对话

第二层：核心流程（触发时加载）
  ├─ 因子挖掘的步骤、校验标准、输出格式
  └─ 只有调用 Skill 时才会加载

第三层：参考资料（按需加载）
  ├─ 债券因子库文档、风控合规要求
  └─ AI 执行中需要时才读取
```

### 3.2 语义触发：懂你的业务语言，而非死记关键词
Skill 触发靠 AI 理解你的意图，而非固定关键词：
- ✅ 说“分析利率债利差因子”→ 触发 Skill
- ✅ 说“测算久期因子对本周组合收益的影响”→ 触发 Skill
- ❌ 说“查一下今天的国债收益率”→ 仅调用行情工具，不触发 Skill

触发是否精准，核心看“触发描述”是否写清楚业务场景（后文会讲）。

## 四、Skill 的核心结构（业务人员可直接套用）
一个能落地的 Skill 只需包含 2 部分：**元数据配置** + **指令正文**，以下是适配金融业务的简化版结构：

```
「债券因子挖掘」Skill 完整结构
┌─────────────────────────────────────┐
│ 元数据配置（简单填，定规则）         │
│ ├─ 技能名称：bond-factor-mining     │
│ ├─ 触发描述：分析债券因子有效性/测算因子对组合收益影响 │
│ ├─ 触发模式：auto（自动）/ manual（手动）│
│ └─ 允许使用的工具：债券行情查询、因子计算脚本 │
├─────────────────────────────────────┤
│ 指令正文（核心，写流程）             │
│ ├─ 触发场景示例（帮 AI 认场景）     │
│ ├─ 执行步骤（结构化，按业务流程写） │
│ └─ 输出格式（明确要求，不留歧义）   │
└─────────────────────────────────────┘
```

### 4.1 元数据配置（金融业务适配版）
| 配置项   | 是否必选 | 填写示例（债券因子挖掘）                                                                                                 | 核心注意点                                                   |
| -------- | -------- | ------------------------------------------------------------------------------------------------------------------------ | ------------------------------------------------------------ |
| 技能名称 | ✅        | bond-factor-mining                                                                                                       | 动词+对象，易懂（如：挖掘债券因子→bond-factor-mining）       |
| 触发描述 | ✅        | 分析债券因子有效性、测算因子（利差/久期/流动性）对组合收益的影响；仅覆盖利率债，不包含信用债；输出需含因子有效性校验结果 | 写清楚“做什么、覆盖范围、排除范围”，100字以内                |
| 触发模式 | ❌        | auto                                                                                                                     | 只读分析（如因子挖掘）选 auto；涉及交易操作选 manual（手动） |
| 允许工具 | ❌        | bond-market-query, factor-calc-script                                                                                    | 只给必要工具权限（如因子挖掘只需行情查询+计算工具）          |

### 4.2 指令正文：STEP 范式（业务人员易落地）
正文按 **S（结构化）-T（目标）-E（示例）-P（输出）** 写，避免模糊的叙述性文字：
- S：用编号步骤，不用大段文字；
- T：每步说清“要做什么、产出什么”；
- E：至少 3 个触发场景示例；
- P：明确输出格式（表格/字段/校验标准）。

## 五、业务流程 Skill 化：五步法（以债券因子挖掘为例）
无需技术背景，只需按以下步骤把你的业务流程变成 Skill：

### Step 1：完整走一遍流程，记录所有细节
先手动完成一次“债券因子挖掘”，记录：
- 你会告诉 AI 的所有要求：“只分析利率债”“因子维度包含利差/久期/流动性”“输出要做 t 检验验证因子有效性”；
- 你用到的数据源：中债登行情、内部组合持仓数据；
- 输出格式：因子对比表 + 有效性结论 + 对组合的影响分析。

### Step 2：提炼“固定部分”和“变化部分”
- ✅ 固定部分（写入 Skill）：
  因子维度（利差/久期/流动性）、校验标准（t 检验 p<0.05 为有效）、输出格式（Markdown 表格）；
- 🔄 变化部分（调用时提供）：
  分析的债券品类（如 10 年期利率债）、时间范围（本周/本月）、组合持仓范围。

### Step 3：写触发描述（核心）
按“做什么+场景+范围”写，示例：
```
执行利率债因子挖掘，分析利差/久期/流动性因子的有效性，测算因子对组合收益的影响。
触发场景：用户说“分析债券因子有效性”“测算久期因子对组合收益的影响”“评估利率债利差因子”。
覆盖范围：仅利率债；排除范围：信用债、衍生品因子。
输出：因子有效性校验表 + 收益影响分析。
```

### Step 4：确定安全级别
按业务风险选择：
- 只读分析（如因子挖掘）：触发模式选 auto，工具权限仅开放“查询/计算”；
- 涉及交易操作（如因子驱动的调仓建议执行）：触发模式选 manual（手动），避免误操作。

### Step 5：按 STEP 范式写正文
把固定流程拆成结构化步骤，明确输出格式。

## 六、完整示例：债券因子挖掘 Skill
以下是可直接复用的 Skill 模板，适配银行资产负债管理部场景：

```markdown
---
name: bond-factor-mining
description: |
  执行利率债因子挖掘，分析利差/久期/流动性因子的有效性，测算因子对组合收益的影响。
  触发场景：用户说"分析债券因子有效性"、"测算久期因子对组合收益的影响"、"评估利率债利差因子"。
  覆盖范围：仅利率债；排除范围：信用债、衍生品因子。
  输出：因子有效性校验表 + 收益影响分析 + 风控提示。
trigger-mode: auto
allowed-tools: bond-market-query, factor-calc-script
metadata:
  version: "1.0.0"
  owner: "资产负债管理部-债券组"
---

# 利率债因子挖掘标准流程

## 触发场景示例
- "分析本周10年期利率债的利差因子有效性"
- "测算久期因子对本月组合收益的影响"
- "评估流动性因子对利率债组合的风控价值"

## 执行步骤（按顺序，不可跳过）
### Step 1：确认分析范围
分析前必须确认（未提供则先询问）：
- 债券类型：具体利率债品类（如 10 年期国债/政策性金融债）；
- 时间范围：分析的时间区间（默认：最近 1 个月）；
- 组合范围：是否限定特定持仓组合（可选）。

### Step 2：因子数据采集与计算
1. 调用债券行情工具，获取指定区间的利率债成交数据、收益率曲线；
2. 调用因子计算脚本，计算利差因子（期限利差/信用利差）、久期因子（修正久期）、流动性因子（换手率/买卖价差）；
3. 数据校验：确保无缺失值、异常值（如收益率偏离均值 3σ 需标注）。

### Step 3：因子有效性检验
对每个因子执行 t 检验（p<0.05 判定为有效因子），记录：
- 因子均值/标准差；
- t 统计量、p 值；
- 有效性结论（有效/无效）。

### Step 4：收益影响分析
测算有效因子对指定组合收益的影响：
- 单因子贡献度（因子变动 × 组合久期/持仓规模）；
- 多因子叠加影响；
- 风控提示（如因子波动超过阈值的风险）。

## 输出格式（严格按此执行）
### 第一部分：因子基础信息表
| 因子类型   | 均值 | 标准差 | t 统计量 | p 值 | 有效性结论 |
| ---------- | ---- | ------ | -------- | ---- | ---------- |
| 利差因子   |      |        |          |      |            |
| 久期因子   |      |        |          |      |            |
| 流动性因子 |      |        |          |      |            |

### 第二部分：组合收益影响分析
1. 单因子贡献度：
   - 利差因子：[数值]（占总收益比例：%）
   - 久期因子：[数值]（占总收益比例：%）
   - 流动性因子：[数值]（占总收益比例：%）
2. 多因子叠加影响：[一句话总结]
3. 风控提示：[1-2 条，如“久期因子波动超 0.5，组合需关注利率风险”]

### 第三部分：数据来源说明
- 行情数据：中债登公开数据/内部持仓系统；
- 计算工具：因子计算脚本 V2.1；
- 校验规则：资负部《债券因子分析风控规范》2026版。
```

## 七、业务人员写 Skill 的常见误区
### 误区 1：触发描述太模糊，只写标题
```
# ❌ 错误
触发描述：债券因子挖掘

# ✅ 正确
触发描述：分析利率债利差/久期/流动性因子有效性，测算因子对组合收益的影响；
          触发场景：用户说'分析债券因子'/'测算因子收益影响'；
          仅覆盖利率债，不包含信用债。
```
**后果**：AI 不知道何时触发，或误触发（如用户查信用债数据时）。

### 误区 2：一个 Skill 涵盖太多业务
```
# ❌ 错误
名称：bond-all-in-one
内容：因子挖掘 + 组合调仓 + 风险报告 + 交易执行

# ✅ 正确
拆分为：bond-factor-mining（因子挖掘）、bond-portfolio-adjust（组合调仓）、bond-risk-report（风险报告）
```
**后果**：触发描述无法精准匹配，输出结果混乱。

### 误区 3：正文用叙述性文字，无结构化步骤
```
# ❌ 错误
帮我分析债券因子，仔细算一下有效性，然后说说对组合的影响。

# ✅ 正确
Step 1：确认分析范围（债券类型/时间/组合）
Step 2：计算因子数据（利差/久期/流动性）
Step 3：t 检验验证有效性（p<0.05 为有效）
Step 4：测算因子对组合收益的贡献度
```
**后果**：AI 执行结果不稳定，每次输出的维度/格式都不同。

### 误区 4：混淆 Skill 和工具调用
```
# 不该用 Skill 做的事（该用工具调用）
- 实时查中债登行情 → 工具调用
- 计算因子 t 检验 → 工具调用

# 该用 Skill 做的事
- 定义因子挖掘的维度（利差/久期/流动性）→ Skill
- 规定输出格式（校验表+收益分析）→ Skill
- 明确风控校验规则（p<0.05 为有效）→ Skill
```

## 总结
对银行资产负债管理部等金融业务人员来说，Skill 不是“技术功能”，而是**把你的业务经验和流程标准化、可复用化的工具**：
- 核心价值：一次梳理，全员复用，避免重复沟通和不标准的输出；
- 核心原则：触发描述写清楚场景，正文按 STEP 范式结构化，区分 Skill 和工具调用的边界；
- 落地关键：从高频、规则明确的业务（如债券因子挖掘）入手，先梳理流程，再按模板写 Skill。

无需懂技术，只需把你“怎么做业务”的经验，按本文的模板写清楚，就能让 AI 始终按你的专业标准完成工作。
