### 1.3 Skill 管理机制（源码事实）

**事实 8：6种操作**

`tools/skill_manager_tool.py` 第 588-646 行定义了完整的 dispatch：
- `create` — 新建 SKILL.md + 安全扫描（不通过则删除目录）
- `edit` — 全文替换 SKILL.md（保留原始内容用于回滚）
- `patch` — 模糊匹配 `fuzzy_find_and_replace` + 原子写入
- `delete` — 删除目录 + 清理空分类目录
- `write_file` — 写入辅助文件（限 references/templates/scripts/assets/）
- `remove_file` — 删除辅助文件

**事实 9：原子写入**

所有文件修改使用 `_atomic_write_text()`（第 256-285 行）：`tempfile.mkstemp()` → 写入 → `os.replace()` 原子替换。

**事实 10：模糊匹配引擎**

```python
# tools/skill_manager_tool.py: 426-430
from tools.fuzzy_match import fuzzy_find_and_replace
new_content, match_count, _strategy, match_error = fuzzy_find_and_replace(
    content, old_string, new_string, replace_all
)
```

处理空白归一化、缩进差异、转义序列、块锚点匹配。

**事实 11：大小限制**

- SKILL.md：100,000 字符（`MAX_SKILL_CONTENT_CHARS`）
- 辅助文件：1,048,576 字节 / 1 MiB（`MAX_SKILL_FILE_BYTES`）
- 名称长度：64 字符（`MAX_NAME_LENGTH`）

**事实 12：Frontmatter 校验**

必须包含 `name` 和 `description` 字段，YAML 格式正确，body 非空。

### 1.4 安全扫描（源码事实）

**事实 13：80+ 威胁模式，10+ 类别**

`tools/skills_guard.py` 第 82-400 行定义了 `THREAT_PATTERNS` 列表，按类别：

| 类别 | 代表性模式 | 模式数 |
|------|-----------|--------|
| exfiltration | curl/wget 拼接 $KEY、.ssh/.aws 访问、printenv、DNS 外泄、Markdown 图片外泄 | ~18 |
| injection | ignore instructions、role hijack、system prompt override、HTML 隐藏 | ~12 |
| destructive | rm -rf /、chmod 777、mkfs、DROP TABLE、TRUNCATE | ~8 |
| persistence | .bashrc 修改、crontab 注入、systemd 服务 | ~5 |
| network | 反向 shell（nc -e、bash -i）、端口转发、SSH 隧道 | ~6 |
| obfuscation | base64 编码执行、\x 十六进制、不可见 Unicode | ~5 |
| execution | eval()、exec()、os.system()、subprocess | ~6 |
| traversal | ../../ 路径穿越、符号链接 | ~4 |
| supply_chain | 可疑 pip/npm install、curl\|bash | ~4 |
| credential_exposure | 硬编码 API keys、密码字符串 | ~8 |

**事实 14：4 严重级别**

`critical` / `high` / `medium` / `low`（非 dangerous/caution/safe 三级）。

`safe` / `caution` / `dangerous` 是**扫描结果的综合判定**（verdict），不是单个 finding 的严重级别。

**事实 15：信任等级与安装策略矩阵**

```python

    "trusted":       ("allow",  "allow",   "block"),
    "community":     ("allow",  "block",   "block"),
    "agent-created": ("allow",  "allow",   "ask"),
```

**事实 16：结构性检查**

除正则匹配外，还检测：文件数量、文件大小、二进制文件、符号链接、不可见 Unicode 字符。

## 3. 系统架构设计

### 3.1 技术栈

| 层级 | 技术 | 选型依据 |

| 语言 | Python 3.11+ | HermesAgent 原生语言 |

## 4. 核心模块设计

### 4.1 双引擎自学习

#### 4.1.1 迭代计数器状态机

```
                    ┌──────────────┐
                    │  count = 0   │
                    └──────┬───────┘
                           │
                    每轮循环迭代          ← 注意：非每次工具调用
                    count += 1
                           │
                           ▼
                    ┌──────────────┐
                    │  count < T   │──── 继续累积（跨任务不重置）
                    └──────┬───────┘
                           │
                    ┌──────┴──────┐
                    │             │
            Engine 1 触发    count ≥ T
            (skill_manage)   (后台巡检)
                    │             │
                    ▼             ▼
            ┌────────────┐ ┌────────────────┐
            │ count = 0  │ │ ReviewAgent     │
            │ (立即归零) │ │ daemon thread   │
            └────────────┘ │ max_iters=8     │
                           │ quiet_mode=True  │
                           │ _nudge_interval=0│
                           └────────┬───────┘
                                    │
                                    ▼

                             │ count = 0  │
                             │ (完成归零) │
                             └────────────┘

T = creation_nudge_interval（默认 10，可配置）
```

#### 4.1.2 ReviewAgent 策略

审查提示聚焦四类高价值经验：
1. **非显而易见的方法** — 需要创造性解决问题
2. **试错学习曲线** — "trial and error"
3. **路径修正** — "changing course due to experiential findings"
4. **用户期望不符** — "user expected or desired a different method"

**核心设计原则**："挫折才是最好的老师"——纯粹顺利的任务不生成 Skill。

### 4.2 安全扫描

#### 4.2.1 分层判定

```
单个 Finding → severity: critical / high / medium / low
                    │
所有 Findings 聚合 → verdict: safe / caution / dangerous
                    │
verdict × trust_level → policy: allow / block / ask
```

#### 4.2.2 策略矩阵语义

| 信任等级 | 含义 | safe | caution | dangerous |
|---------|------|------|---------|-----------|
| builtin | 内置随系统发布 | allow | allow | allow |
| trusted | 官方认证仓库 | allow | allow | block |
| community | 社区贡献 | allow | block | block |
| agent-created | Agent 自主生成 | allow | allow | **ask** |

`agent-created` 的 `dangerous → ask` 是核心设计：对自进化能力保持信任，但不盲目放行危险内容。

### 4.3 条件激活

#### 4.3.1 四维过滤完整逻辑

```python
def should_show(skill, available_tools, available_toolsets):
    # 无过滤信息时：显示所有（向后兼容）
    if available_tools is None and available_toolsets is None:
        return True

    at = available_tools or set()
    ats = available_toolsets or set()

    # Rule 1: fallback_for_toolsets — 主工具集在场则隐藏
    for ts in skill.fallback_for_toolsets:
        if ts in ats:
            return False

    # Rule 2: fallback_for_tools — 主工具在场则隐藏
    for t in skill.fallback_for_tools:
        if t in at:
            return False

    # Rule 3: requires_toolsets — 依赖工具集缺失则隐藏
    for ts in skill.requires_toolsets:
        if ts not in ats:
            return False

    # Rule 4: requires_tools — 依赖工具缺失则隐藏
    for t in skill.requires_tools:
        if t not in at:
            return False

    return True
```

## 9. 技术风险与对策

| 编号 | 风险 | 影响 | 对策 |
|------|------|------|------|
| R-01 | Agent 自生成低质量 Skill | 技能污染 | max_iterations=8 限制 + 安全扫描兜底 + agent-created 信任等级 |
| R-02 | 模糊匹配引擎误匹配 | 热补丁破坏 Skill | 原子回滚 + Frontmatter 完整性校验 + 安全回扫 |
| R-03 | 后台巡检线程阻塞 | 主 Agent 无感知 | daemon=True 保证线程不阻塞进程退出 |
| R-04 | Redis 计数器丢失 | 巡检延迟触发 | AOF 持久化 + 7天 TTL 自清理 |
| R-05 | TDSQL JSON 查询性能 | 过滤慢查询 | 索引关键字段（name, status, safety_level）+ JSON 列仅存储不常查询的数组 |
| R-06 | 安全正则绕过 | 危险 Skill 通过 | 多层防御（正则 + 结构检查 + 不可见字符检测 + 信任矩阵） |
| R-07 | 递归巡检风暴 | 系统过载 | 审查 Agent 的 _skill_nudge_interval=0 切断递归 |
| R-08 | 跨集群网络分区 | Worker 断连 | Submariner 自动重连 + Worker 本地降级 |
